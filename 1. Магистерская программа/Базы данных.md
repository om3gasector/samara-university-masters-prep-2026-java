[[Магистерский хаб]]

---

#### 1.1 Понятия базы данных, банка данных, базы знаний, системы управления базой данных (СУБД). Виды СУБД
**База данных (БД)** — именованная совокупность взаимосвязанных данных, хранящихся совместно с минимальной избыточностью и используемых многими приложениями. **Банк данных** — совокупность БД, СУБД, технических средств, персонала и прикладных программ. 

**База знаний** — структурированные знания о предметной области с правилами логического вывода (в отличие от БД, где данные без вывода). 

**СУБД** — комплекс программных средств для создания, ведения и совместного использования БД. **Виды СУБД:** по модели данных (реляционные, иерархические, сетевые, объектно-ориентированные, NoSQL), по архитектуре (локальные, файл-серверные, клиент-серверные, распределённые).
##### 1.1-1 ==Понятия базы данных, банка данных, базы знаний==
**База данных (БД)** — это совместно используемый набор логически связанных данных и их описания, предназначенный для удовлетворения информационных потребностей пользователей. Ключевые признаки: интегрированность (единое хранилище), минимальная избыточность, независимость данных от программ, совместное использование.

**Банк данных** — более широкое понятие. Это организационно-техническая система, включающая:

- Базу данных (сами данные).
- СУБД (программное обеспечение).
- Аппаратные средства (серверы, дисковые массивы).
- Персонал (администраторы БД).
- Прикладные программы, работающие с БД.

Банк данных — это не просто данные, а вся инфраструктура вокруг них.

**База знаний** — это структурированный набор знаний о предметной области, представленный в формальной форме, позволяющий осуществлять логический вывод новых фактов. Отличие от БД: в БД хранятся факты, в базе знаний — факты и правила вывода. База знаний — центральный компонент экспертных систем. Пример: медицинская диагностическая система (симптомы = факты, медицинские правила = правила вывода).
##### 1.1-2 ==Система управления базой данных (СУБД)==
**СУБД (DBMS — Database Management System)** — комплекс программных и языковых средств, предназначенных для создания, ведения и совместного использования баз данных.

**Основные функции СУБД:**
- **Управление данными во внешней памяти:** хранение, чтение, запись.
- **Управление данными в оперативной памяти:** буферизация, кэширование.
- **Управление транзакциями:** обеспечение ACID-свойств (атомарность, согласованность, изоляция, долговечность).
- **Журнализация и восстановление:** возможность отката и восстановления после сбоев.
- **Поддержка языков БД:** SQL (DDL — определение данных, DML — манипулирование данными, DCL — управление доступом).

**Компоненты СУБД:**

- Ядро СУБД (обработка запросов, оптимизация).
- Компилятор/интерпретатор SQL.
- Менеджер транзакций.
- Менеджер блокировок.
- Утилиты (резервное копирование, восстановление, мониторинг).
##### 1.1-3 ==Виды СУБД==
**Классификация по модели данных:**
- **Реляционные (RDBMS):** данные в таблицах, связи через внешние ключи, язык SQL. Примеры: PostgreSQL, MySQL, Oracle, Microsoft SQL Server.
- **Иерархические:** данные в виде деревьев (родитель-потомок). Пример: IBM IMS.
- **Сетевые:** данные в виде графов (один потомок — много родителей). Пример: IDMS.
- **Объектно-ориентированные (OODBMS):** данные как объекты с методами. Пример: ObjectStore, db4o.
- **NoSQL:** нереляционные БД для больших неструктурированных данных. Типы: документные (MongoDB), ключ-значение (Redis), графовые (Neo4j), колоночные (Cassandra).

**Классификация по архитектуре:**
- **Локальные:** БД и СУБД на одном компьютере (SQLite, Microsoft Access).
- **Файл-серверные:** БД на сервере, СУБД на клиентах. Высокий сетевой трафик (dBase, FoxPro).
- **Клиент-серверные:** СУБД на сервере обрабатывает запросы, клиенты отправляют SQL. Основной тип для бизнес-приложений (PostgreSQL, Oracle).
- **Распределённые:** данные распределены по узлам сети, единая СУБД управляет ими как целым.

**Классификация по способу доступа:**
- **OLTP (Online Transaction Processing):** много коротких транзакций (банковские системы).
- **OLAP (Online Analytical Processing):** сложные аналитические запросы (хранилища данных).
#### 1.2 Уровни представления данных. Модели данных концептуального и логического уровня представления данных
Архитектура ANSI/SPARC выделяет три уровня представления данных: 
внешний (пользовательские представления), 
концептуальный (обобщённое логическое описание всей БД), 
внутренний (физическое хранение). 

Модели данных делятся на концептуальные (ER-модель, семантические сети) — описывают предметную область без привязки к СУБД, и логические (реляционная, иерархическая, сетевая, объектно-ориентированная) — описывают структуру данных в терминах конкретного типа СУБД.
##### 1.2-1 ==Уровни представления данных==
Трёхуровневая архитектура ANSI/SPARC (1975) — стандартная модель организации СУБД. Каждый уровень описывает данные с разной степенью абстракции.

**Внешний уровень (пользовательский):**
- Набор внешних схем (представлений) для разных групп пользователей.
- Каждый пользователь видит только ту часть данных, которая ему нужна.
- Одна и та же БД может иметь несколько внешних представлений.
- Обеспечивает безопасность и удобство (скрывает сложность).

**Концептуальный уровень:**
- Единое обобщённое описание всех данных БД.
- Описывает, **какие** данные хранятся и **как** они связаны, но не **как** они физически хранятся.
- Включает: сущности, атрибуты, связи, ограничения целостности.
- Независим от конкретной СУБД и физического хранения.

**Внутренний уровень (физический):**
- Описывает физическое хранение данных: файлы, индексы, страницы, расположение на диске.
- Оптимизирован под производительность.
- Зависит от конкретной СУБД и операционной системы.

**Независимость уровней:**
- **Логическая независимость:** изменение концептуальной схемы не затрагивает внешние схемы (добавление нового атрибута в таблицу не ломает существующие представления).
- **Физическая независимость:** изменение физической организации (индексы, размещение файлов) не затрагивает концептуальную схему.
##### 1.2-2 ==Модели данных концептуального уровня==
Концептуальные модели описывают предметную область в терминах, понятных человеку, без привязки к технической реализации. Используются на этапе анализа и проектирования.

**ER-модель (Entity-Relationship, «сущность-связь»):**
- Предложена Питером Ченом в 1976 году.
- Основные понятия: **сущность** (объект предметной области), **атрибут** (свойство сущности), **связь** (отношение между сущностями).
- Графическое представление: ER-диаграммы (прямоугольники — сущности, ромбы — связи, овалы — атрибуты).
- Наиболее распространённая концептуальная модель.

**Семантические сети:**
- Данные представлены в виде ориентированного графа: вершины — понятия, дуги — отношения.
- Более выразительны, чем ER-модель, но сложнее в реализации.

**UML-диаграммы классов:**
- Используются в объектно-ориентированном проектировании.
- Описывают классы, атрибуты, методы, ассоциации, наследование.

Назначение концептуальных моделей: согласовать с заказчиком структуру данных до начала программирования.
##### 1.2-3 ==Модели данных логического уровня==
Логические модели описывают структуру данных в терминах, понятных СУБД. Это промежуточный уровень между концептуальной моделью и физическим хранением.

**Реляционная модель:**
- Данные в виде отношений (таблиц).
- Каждая таблица — множество кортежей (строк).
- Связи через внешние ключи.
- Стандартный язык — SQL.
- Доминирует с 1980-х годов.

**Иерархическая модель:**
- Данные в виде дерева (леса деревьев).
- Каждый потомок имеет только одного родителя.
- Быстрый доступ по заранее известным путям.
- Пример: файловая система, LDAP-каталоги.

**Сетевая модель:**
- Данные в виде графа общего вида.
- Потомок может иметь несколько родителей (множественные связи).
- Стандарт CODASYL.
- Гибче иерархической, но сложнее в использовании.

**Объектно-ориентированная модель:**
- Данные как объекты с атрибутами и методами.
- Поддерживает наследование, инкапсуляцию, полиморфизм.
- Используется в специализированных областях (САПР, мультимедиа).

**Выбор модели** определяет тип СУБД и способ работы с данными.
#### 1.3 Реляционная модель данных. Языки манипулирования данными для реляционной модели. Операции реляционной алгебры
Реляционная модель данных (Эдгар Кодд, 1970) представляет данные в виде отношений (таблиц). 
Отношение состоит из кортежей (строк) и атрибутов (столбцов). Ключ — атрибут(ы), однозначно идентифицирующий кортеж. 

Языки манипулирования: реляционная алгебра (процедурный — что и как делать) и реляционное исчисление (декларативный — что получить без указания алгоритма). SQL объединяет оба подхода. 

Операции реляционной алгебры: объединение, пересечение, разность, проекция, селекция (выборка), декартово произведение, соединение (join), деление. Операции замкнуты относительно отношений — результат операции тоже отношение.
##### 1.3-1 ==Реляционная модель данных==
**Реляционная модель данных** предложена Эдгаром Коддом в 1970 году. Данные организованы в виде **отношений** (relations), которые пользователю представляются как **таблицы**.

**Основные понятия:**
- **Отношение** — двумерная таблица, состоящая из строк и столбцов. Каждое отношение имеет уникальное имя.
- **Кортеж** — строка таблицы, представляет экземпляр сущности или связи.
- **Атрибут** — столбец таблицы, характеризует свойство сущности. Имеет имя и домен (тип данных).
- **Домен** — множество допустимых значений атрибута (например, целые числа от 1 до 100).
- **Степень отношения (arity)** — количество атрибутов.
- **Кардинальность** — количество кортежей.

**Свойства отношения:**
- Каждый кортеж уникален (нет повторяющихся строк).
- Порядок кортежей не определён (теоретически).
- Порядок атрибутов фиксирован, но значения доступны по имени.
- Все значения атрибутов атомарны (неделимы) — требование первой нормальной формы.

**Ключи отношения:**
- **Суперключ** — любой набор атрибутов, однозначно идентифицирующий кортеж.
- **Потенциальный ключ** — минимальный суперключ (нельзя убрать ни один атрибут без потери уникальности).
- **Первичный ключ (PRIMARY KEY)** — один из потенциальных ключей, выбранный как основной. Не может содержать NULL.
- **Внешний ключ (FOREIGN KEY)** — атрибут(ы), ссылающийся на первичный ключ другого отношения. Обеспечивает связь между таблицами.
##### 1.3-2 ==Языки манипулирования данными для реляционной модели==
Языки манипулирования данными (DML — Data Manipulation Language) делятся на два класса.

**Реляционная алгебра (процедурный подход):**
- Указывает **порядок действий** для получения результата.
- Состоит из набора операций над отношениями.
- Результат каждой операции — отношение (свойство замкнутости).
- Является теоретической основой SQL.

**Реляционное исчисление (декларативный подход):**
- Указывает, **что** нужно получить, без описания алгоритма.
- Основано на логике предикатов первого порядка.
- Две разновидности: исчисление кортежей (переменные — кортежи) и исчисление доменов (переменные — значения доменов).
- Пример: `{t | ∃ s ∈ Students (s.name = t.name AND s.age > 20)}` — получить имена студентов старше 20 лет.

**SQL** как язык манипулирования объединяет оба подхода:
- Основная конструкция `SELECT... FROM... WHERE` соответствует операциям реляционной алгебры (FROM — декартово произведение и соединения, WHERE — селекция, SELECT — проекция).
- Вложенные запросы (подзапросы) базируются на реляционном исчислении.
##### 1.3-3 ==Операции реляционной алгебры==
Реляционная алгебра состоит из набора операций, каждая из которых принимает одно или два отношения и возвращает новое отношение.

**Теоретико-множественные операции (оба операнда должны быть совместимы по схеме — одинаковый набор атрибутов):**
- **Объединение (R ∪ S):** кортежи, принадлежащие R или S (или обоим). Дубликаты удаляются.
- **Пересечение (R ∩ S):** кортежи, принадлежащие и R, и S одновременно.
- **Разность (R − S):** кортежи R, не входящие в S.
- **Декартово произведение (R × S):** все возможные комбинации каждого кортежа R с каждым кортежем S. Степень = сумма степеней, кардинальность = произведение кардинальностей.

**Специальные реляционные операции:**
- **Селекция (выборка, σ — сигма):** `σ_условие(R)` — выбирает кортежи, удовлетворяющие условию. Аналог WHERE в SQL. Условие: сравнение атрибутов с константами или между собой (>, <, =, AND, OR, NOT).
- **Проекция (π — пи):** `π_атрибуты(R)` — выбирает указанные атрибуты, удаляя дубликаты кортежей. Аналог SELECT DISTINCT в SQL.
- **Соединение (⨝ — join):** `R ⨝_условие S` — декартово произведение с последующей селекцией по условию. Основной тип — **естественное соединение (natural join)**, где условие — равенство атрибутов с одинаковыми именами.
- **Деление (R ÷ S):** для каждого кортежа R, связанного со всеми кортежами S. Наиболее сложная операция. Аналог: «найти студентов, посещающих все курсы».

**Примеры операций в терминах SQL:**
- Селекция: `SELECT * FROM R WHERE условие`
- Проекция: `SELECT DISTINCT a1, a2 FROM R`
- Соединение: `SELECT * FROM R JOIN S ON R.key = S.key`
#### 1.4 Понятие функциональной зависимости между атрибутами отношений. Виды функциональных зависимостей
Функциональная зависимость (ФЗ) — связь между атрибутами отношения: зная значение X, можно однозначно определить значение Y. Обозначается X → Y (X детерминирует Y). 

Виды ФЗ: тривиальная (Y ⊆ X), нетривиальная, полная (Y зависит от всего X, а не от его части), частичная (Y зависит от части составного ключа), транзитивная (X → Z и Z → Y, при этом Z не ключ). ФЗ лежат в основе нормализации: частичные зависимости устраняются во 2НФ, транзитивные — в 3НФ.
##### 1.4-1 ==Понятие функциональной зависимости между атрибутами отношений==
**Функциональная зависимость (ФЗ)** — фундаментальное понятие реляционной модели, описывающее семантическую связь между атрибутами. Если в отношении R атрибут Y функционально зависит от атрибута X, это означает: в любой момент времени каждому значению X соответствует ровно одно значение Y.

**Обозначение:** X → Y (X детерминирует Y, X определяет Y).

**Формальное определение:** для любых двух кортежей t1 и t2 отношения R, если t1[X] = t2[X], то t1[Y] = t2[Y]. Одинаковые значения X гарантируют одинаковые значения Y.

**Пример:** в отношении Студенты(НомерЗачётки, ФИО, Группа):
- НомерЗачётки → ФИО (по номеру зачётки однозначно определяется студент).
- НомерЗачётки → Группа (студент числится в одной группе).
- Группа → Факультет (группа принадлежит одному факультету).

**Откуда берутся ФЗ:** ФЗ не вычисляются из данных, а определяются семантикой предметной области. Проектировщик на основе знания бизнес-правил фиксирует: какие атрибуты определяют другие. ФЗ являются ограничениями целостности и должны поддерживаться СУБД.
##### 1.4-2 ==Виды функциональных зависимостей==
**Тривиальная ФЗ:**  
X → Y, если Y является подмножеством X (Y ⊆ X). Например: {НомерЗачётки, ФИО} → ФИО. Всегда выполняется автоматически, не несёт полезной информации. Обратное — нетривиальная ФЗ.

**Нетривиальная ФЗ:**  
X → Y, где Y не входит в X. Содержит семантическую информацию о данных. Например: НомерЗачётки → Группа.

**Полная функциональная зависимость:**  
Y функционально зависит от X, но не зависит ни от какого подмножества X (при составном X). Если X — составной атрибут {A, B}, и удаление любого атрибута разрушает зависимость, то ФЗ полная. Например: {Студент, Курс} → Оценка. Без студента курс не даёт оценки, без курса студент — тоже.

**Частичная функциональная зависимость:**  
Y зависит от части составного ключа. Например: {НомерЗачётки, КодКурса} → ФИО_Студента. ФИО зависит только от НомераЗачётки, а не от всей пары. Частичные зависимости — аномалия, устраняются приведением ко 2НФ.

**Транзитивная функциональная зависимость:**  
X → Z и Z → Y, при этом Z не является потенциальным ключом (и Z не содержится в X). Непрямая зависимость через промежуточный атрибут. Например: НомерЗачётки → Группа → Факультет. Факультет транзитивно зависит от номера зачётки через группу. Транзитивные зависимости устраняются приведением к 3НФ.

**Многозначная зависимость (упоминание для полноты):**  
X →→ Y (X мультидетерминирует Y). Одному значению X соответствует множество значений Y, независимое от других атрибутов. Используется в 4НФ.
#### 1.5 Нормализация отношений. Виды нормальных форм отношений. Приведение отношений к третьей нормальной форме
Нормализация — процесс преобразования отношений к виду, исключающему избыточность и аномалии (вставки, обновления, удаления).

Выполняется последовательным приведением к нормальным формам: 1НФ (атомарность значений), 2НФ (устранение частичных зависимостей — каждый неключевой атрибут полностью зависит от всего первичного ключа), 3НФ (устранение транзитивных зависимостей — неключевые атрибуты не зависят друг от друга). Нормальная форма Бойса-Кодда (BCNF) — усиленная 3НФ. 4НФ устраняет многозначные зависимости, 5НФ — зависимости соединения. На практике обычно достаточно 3НФ или BCNF.
##### 1.5-1 ==Нормализация отношений==
**Нормализация** — формальный метод анализа и проектирования реляционной схемы БД, основанный на функциональных зависимостях и понятии нормальных форм. Разработан Эдгаром Коддом.

**Цели нормализации:**
- Минимизация избыточности данных.
- Исключение аномалий модификации (вставки, обновления, удаления).
- Обеспечение целостности данных.

**Аномалии (на примере ненормализованного отношения):**
- **Аномалия вставки:** нельзя добавить данные об одной сущности без данных о другой (нельзя записать нового студента, не записав его на курс).
- **Аномалия обновления:** изменение одного факта требует изменения множества строк (смена фамилии преподавателя — во всех записях его курсов).
- **Аномалия удаления:** удаление последнего факта об одной сущности удаляет данные о другой (удаление последнего курса преподавателя удаляет информацию о самом преподавателе).

**Процесс нормализации:**  
Последовательная декомпозиция (разбиение) исходного отношения на несколько связанных отношений с сохранением всех зависимостей и возможности восстановления исходных данных (свойство соединения без потерь).
##### 1.5-2 ==Виды нормальных форм отношений==
**Первая нормальная форма (1НФ):**  
Отношение находится в 1НФ, если все его атрибуты атомарны (неделимы). Каждое значение — скаляр, а не список, множество или вложенная таблица. 1НФ является базовым требованием реляционной модели.

Пример нарушения: атрибут «Телефоны» содержит несколько номеров через запятую. Решение: вынести телефоны в отдельное отношение.

**Вторая нормальная форма (2НФ):**  
Отношение находится во 2НФ, если оно в 1НФ и **каждый неключевой атрибут полностью функционально зависит от всего первичного ключа** (а не от его части). Проблема возникает только при составном первичном ключе.

Пример нарушения: отношение Успеваемость(Студент, Курс, ФИО_Студента, Оценка). Первичный ключ — {Студент, Курс}. ФИО_Студента зависит только от Студента (часть ключа). Решение: разбить на Студенты(Студент, ФИО) и Успеваемость(Студент, Курс, Оценка).

**Третья нормальная форма (3НФ):**  
Отношение находится в 3НФ, если оно во 2НФ и **ни один неключевой атрибут не зависит транзитивно от первичного ключа**. Неключевые атрибуты должны зависеть только от ключа, напрямую.

Пример нарушения: отношение Студенты(НомерЗачётки, ФИО, Группа, Факультет). Первичный ключ — НомерЗачётки. НомерЗачётки → Группа → Факультет (транзитивная зависимость). Решение: разбить на Студенты(НомерЗачётки, ФИО, Группа) и Группы(Группа, Факультет).

**4НФ:** устраняет многозначные зависимости (одному ключу соответствует множество независимых значений двух разных атрибутов).

**5НФ (нормальная форма проекции-соединения):** отношение не может быть восстановлено соединением его проекций с меньшим числом атрибутов.
##### 1.5-3 ==Приведение отношений к третьей нормальной форме==
**Пошаговый алгоритм приведения к 3НФ:**

**Шаг 1. Приведение к 1НФ:**  
Убедиться, что все атрибуты атомарны. Если есть повторяющиеся группы или множественные значения — вынести их в отдельное отношение вместе с первичным ключом исходного.

**Шаг 2. Приведение к 2НФ:**
- Определить все потенциальные ключи.
- Для каждого неключевого атрибута проверить, зависит ли он от части ключа (частичная зависимость). Если да — вынести этот атрибут вместе с той частью ключа, от которой он зависит, в новое отношение.
- Исходное отношение сохраняет первичный ключ и атрибуты, зависящие от всего ключа.

**Шаг 3. Приведение к 3НФ:**
- Проверить наличие транзитивных зависимостей: существует ли ФЗ X → Z и Z → Y, где X — первичный ключ, Z — неключевой атрибут, Y — неключевой атрибут.
- Если да — вынести Z и Y в новое отношение, где Z — первичный ключ. В исходном отношении оставить Z как внешний ключ.

**Пример приведения к 3НФ:**

Исходное отношение: Заказы(НомерЗаказа, Дата, КодКлиента, ФИО_Клиента, Город, Сумма).

**Функциональные зависимости:**
- НомерЗаказа → Дата, КодКлиента, Сумма
- КодКлиента → ФИО_Клиента, Город

Отношение в 1НФ (все значения атомарны).  
Проверка 2НФ: первичный ключ — НомерЗаказа (простой, не составной). Частичных зависимостей нет → отношение во 2НФ.

Проверка 3НФ: есть транзитивная зависимость: НомерЗаказа → КодКлиента → ФИО_Клиента, Город.

**Декомпозиция:**
- Заказы(НомерЗаказа, Дата, КодКлиента, Сумма) — КодКлиента как внешний ключ.
- Клиенты(КодКлиента, ФИО_Клиента, Город).

Оба отношения в 3НФ. Аномалии устранены.
#### 1.6 Проектирование базы данных с использованием метода сущность-связь. Виды сущностей, связей, ключей
Метод сущность-связь - это базовый инструмент для проектирования баз данных и описания информационных систем. Он позволяет наглядно отобразить объекты предметной области, их свойства и то, как они взаимодействуют друг с другом

Сущность — объект предметной области (сильная — существует независимо, слабая — зависит от другой сущности). 
Связь — отношение между сущностями (1:1, 1:M, M:N). Ключи: потенциальный (любой уникальный идентификатор), первичный (основной выбранный), внешний (ссылка на первичный ключ другой сущности), составной (несколько атрибутов). Результат — ER-диаграмма, преобразуемая в реляционную схему.
##### 1.6-1 ==Проектирование базы данных с использованием метода сущность-связь==
**Метод сущность-связь (Entity-Relationship, ER-метод)** — способ концептуального проектирования БД, при котором предметная область описывается в виде сущностей, их атрибутов и связей между ними.

**Этапы проектирования:**
1. Выделение сущностей — объектов, о которых будет храниться информация.
2. Определение атрибутов каждой сущности.
3. Выделение связей между сущностями.
4. Определение типов связей (мощность, степень).
5. Построение ER-диаграммы.
6. Преобразование ER-диаграммы в реляционную схему (таблицы, ключи).

**Обозначения на ER-диаграмме (нотация Чена):**
- Прямоугольник — сущность.
- Ромб — связь.
- Овал — атрибут.
- Линии — соединение сущностей со связями и атрибутами.

Метод позволяет согласовать структуру БД с заказчиком на наглядном уровне, независимо от технической реализации.
##### 1.6-2 ==Виды сущностей==
**Сущность (entity)** — реальный или абстрактный объект предметной области, информация о котором должна храниться в БД. Примеры: Студент, Преподаватель, Курс, Заказ.

**Виды сущностей:**

**По независимости существования:**
- **Сильная (независимая) сущность:** существует сама по себе, не зависит от других сущностей. Имеет собственный уникальный идентификатор. Пример: Студент (существует независимо от группы или курса).
- **Слабая (зависимая) сущность:** не может существовать без связанной с ней сильной сущности. Не имеет собственного уникального идентификатора, использует идентификатор владельца + частичный ключ. На диаграмме — двойной прямоугольник. Пример: Комната (без Здания не существует), Счёт-фактура (без Заказа не существует).

**По роли в иерархии:**
- **Супертип (родительская сущность):** обобщённая сущность, содержащая общие атрибуты группы подтипов. Пример: ТранспортноеСредство.
- **Подтип (дочерняя сущность):** специализированная сущность, наследующая атрибуты супертипа и добавляющая свои. Пример: Автомобиль, Мотоцикл (подтипы ТранспортногоСредства).
##### 1.6-3 ==Виды связей==
**Связь (relationship)** — ассоциация между двумя или более сущностями, значимая для предметной области.

**Классификация по мощности (кардинальности):**
- **Один-к-одному (1:1):** одному экземпляру сущности A соответствует не более одного экземпляра сущности B, и наоборот. Пример: Студент ↔ ЗачётнаяКнижка (у студента одна зачётка, зачётка принадлежит одному студенту).
- **Один-ко-многим (1:M):** одному экземпляру A соответствует множество экземпляров B, но экземпляр B связан не более чем с одним A. Самый распространённый тип. Пример: Группа → Студенты (в группе много студентов, студент в одной группе).
- **Многие-ко-многим (M:N):** экземпляры A и B могут быть связаны в любых комбинациях. Требует создания промежуточной таблицы при реализации. Пример: Студент ↔ Курс (студент посещает много курсов, курс посещают много студентов).

**Классификация по степени (количеству участников):**
- **Унарная (рекурсивная):** сущность связана сама с собой. Пример: Сотрудник → Начальник (оба — Сотрудники).
- **Бинарная:** две сущности (большинство связей).
- **Тернарная:** три сущности. Пример: Поставщик поставляет Деталь для Проекта.
##### 1.6-4 ==Виды ключей==
**Ключ** — атрибут или набор атрибутов, используемый для идентификации кортежей в отношении.

**Потенциальный ключ (candidate key):**  
Любой набор атрибутов, значения которого уникальны для каждого кортежа и который минимален (нельзя удалить атрибут без потери уникальности). В отношении может быть несколько потенциальных ключей. Пример: в таблице Студенты — НомерЗачётки и Email (если у каждого студента уникальный email).

**Первичный ключ (primary key):**  
Один из потенциальных ключей, выбранный в качестве основного идентификатора. Обозначается PRIMARY KEY. Требования:
- Не может содержать NULL.
- Значения должны быть уникальны.
- Выбирается тот, который реже меняется и компактнее.

**Внешний ключ (foreign key):**  
Атрибут(ы) одного отношения, ссылающийся на первичный (или потенциальный) ключ другого отношения. Реализует связи между таблицами. Пример: в таблице Студенты атрибут КодГруппы — внешний ключ к таблице Группы. Обеспечивает ссылочную целостность.

**Составной ключ (composite key):**  
Ключ, состоящий из нескольких атрибутов. Используется, когда одного атрибута недостаточно для уникальной идентификации. Пример: в таблице Успеваемость — {КодСтудента, КодКурса} (оценка однозначно определяется парой студент+курс).

**Суррогатный ключ (surrogate key):**  
Искусственно введённый атрибут (обычно автоинкрементное целое число), используемый как первичный ключ вместо естественного. Преимущества: компактность, неизменность, простота. Пример: Id (целое, автоинкремент) вместо НомерЗачётки.
#### 1.7 Язык SQL. Команда создания таблиц, ведения таблиц, выборки данных
SQL — стандартный язык реляционных БД. 
Делится на подмножества: 
DDL (определение данных: CREATE, ALTER, DROP), 
DML (манипулирование данными: INSERT, UPDATE, DELETE, SELECT), 
DCL (управление доступом: GRANT, REVOKE), 
TCL (управление транзакциями: COMMIT, ROLLBACK). 

CREATE TABLE задаёт имя таблицы, столбцы с типами, ограничения (PRIMARY KEY, FOREIGN KEY, NOT NULL, UNIQUE, CHECK). 
INSERT добавляет строки, UPDATE изменяет, DELETE удаляет. 
SELECT — выборка данных: FROM (источник), WHERE (фильтр), GROUP BY (группировка), HAVING (фильтр групп), ORDER BY (сортировка). 
JOIN соединяет таблицы (INNER, LEFT, RIGHT, FULL).
##### 1.7-1 ==Язык SQL==
**SQL (Structured Query Language)** — структурированный язык запросов, стандартный для реляционных СУБД. Первая версия — IBM, 1970-е. Стандартизирован ANSI и ISO.

**Подмножества SQL:**
- **DDL (Data Definition Language):** создание, изменение, удаление объектов БД. Команды: CREATE, ALTER, DROP, TRUNCATE.
- **DML (Data Manipulation Language):** манипулирование данными. Команды: SELECT, INSERT, UPDATE, DELETE.
- **DCL (Data Control Language):** управление правами доступа. Команды: GRANT, REVOKE.
- **TCL (Transaction Control Language):** управление транзакциями. Команды: COMMIT, ROLLBACK, SAVEPOINT.

SQL — декларативный язык: пользователь указывает, ЧТО получить, а не КАК. Оптимизатор СУБД выбирает способ выполнения.
##### 1.7-2 ==Команда создания таблиц (CREATE TABLE)==
**CREATE TABLE** создаёт новое отношение (таблицу) в БД. Определяет имя таблицы, столбцы, их типы и ограничения.

**Синтаксис:**
```
CREATE TABLE ИмяТаблицы (
    столбец1 ТипДанных [ограничения],
    столбец2 ТипДанных [ограничения],
    ...
    [ограничения уровня таблицы]
);
```
**Основные типы данных SQL:**

- **INTEGER, INT:** целое число.
- **DECIMAL(p, s), NUMERIC(p, s):** число с фиксированной точкой (p — всего цифр, s — после запятой).
- **FLOAT, REAL:** число с плавающей точкой.
- **CHAR(n):** строка фиксированной длины.
- **VARCHAR(n):** строка переменной длины.
- **DATE, TIME, TIMESTAMP:** дата и время.
- **BOOLEAN:** логический тип.
- **BLOB, TEXT:** большие двоичные и текстовые данные.
##### 1.7-3 ==Команды ведения таблиц (INSERT, UPDATE, DELETE)==
**INSERT** — добавление новых строк в таблицу:
```
INSERT INTO Students (StudentID, FullName, GroupID)
VALUES (1, 'Иванов И.И.', 101);
```
Добавление нескольких строк:
```
INSERT INTO Students (StudentID, FullName)
VALUES (2, 'Петров П.П.'), (3, 'Сидоров С.С.');
```
Вставка результата SELECT:
```
INSERT INTO Archive
SELECT * FROM Students WHERE EnrollmentDate < '2020-01-01';
```
**UPDATE** — изменение существующих строк:
```
UPDATE Students
SET GroupID = 102, Email = 'ivanov@example.com'
WHERE StudentID = 1;
```
**DELETE** — удаление строк.
```
DELETE FROM Students WHERE StudentID = 3;
```
##### 1.7-4 ==Команда выборки данных (SELECT)==
**SELECT** — основная команда извлечения данных. Позволяет фильтровать, группировать, сортировать и соединять данные из таблиц.
**Общая структура:**
```
SELECT [DISTINCT] список_столбцов
FROM таблицы_и_соединения
[WHERE условие_фильтрации]
[GROUP BY столбцы_группировки]
[HAVING условие_для_групп]
[ORDER BY столбцы [ASC|DESC]];
```
**Базовый запрос:**
```SELECT FullName, GroupID FROM Students;
SELECT DISTINCT GroupID FROM Students; -- без дубликатов
SELECT * FROM Students; -- все столбцы
```

**WHERE — фильтрация строк:**
```SELECT * FROM Students
WHERE GroupID = 101 AND EnrollmentDate >= '2023-01-01';
```
Операторы: =, <>, >, <, >=, <=, BETWEEN, LIKE, IN, IS NULL, AND, OR, NOT.

**GROUP BY и агрегатные функции:**
```SELECT GroupID, COUNT(*) AS StudentCount, AVG(Age) AS AvgAge
FROM Students
GROUP BY GroupID;
```

**ORDER BY — сортировка:**
```SELECT * FROM Students ORDER BY FullName ASC, GroupID DESC;
```

**JOIN — соединение таблиц:**
- **INNER JOIN:** только совпадающие строки.
- **LEFT JOIN:** все строки левой таблицы + совпадения правой (NULL, если нет).
- **RIGHT JOIN:** все строки правой + совпадения левой.
- **FULL JOIN:** все строки обеих таблиц.
```SELECT Students.FullName, Groups.GroupName
FROM Students
INNER JOIN Groups ON Students.GroupID = Groups.GroupID;
```

**Подзапросы:**
```SELECT FullName FROM Students
WHERE GroupID IN (SELECT GroupID FROM Groups WHERE Faculty = 'Информатика');
```
#### 1.8 Физическая организация баз данных. Методы доступа к данным: последовательный, индексный, производный
Физическая организация БД — способ хранения данных на внешних носителях. Включает файлы данных, журналы транзакций, индексы, страницы памяти. Записи размещаются в блоках (страницах) фиксированного размера. 

Методы доступа: последовательный (полный перебор, эффективен при чтении >10–15% записей), индексный (доступ по ключу через индексную структуру — B-дерево, хэш; быстрый поиск единичных записей), производный (доступ через вычисляемый адрес — хэширование). Выбор метода доступа влияет на производительность запросов.
##### 1.8-1 ==Физическая организация баз данных==
**Физическая организация БД** — уровень представления данных, связанный с их размещением на устройствах внешней памяти (диски, SSD).

**Основные компоненты физической организации:**
- **Файлы данных:** основные файлы, хранящие таблицы и индексы. Могут быть: файлы с неструктурированными записями (куча, heap), файлы с записями фиксированной длины, файлы с записями переменной длины.
- **Блоки (страницы):** минимальная единица обмена между диском и оперативной памятью. Типичный размер — 4–16 КБ. Каждая страница содержит несколько записей. СУБД управляет буферным пулом страниц в оперативной памяти.
- **Индексные структуры:** отдельные файлы или части файлов, ускоряющие поиск данных по ключу.

**Способы размещения записей:**
- **Непосредственное (куча):** записи хранятся в порядке их добавления. Новые записи дописываются в конец.
- **Последовательное:** записи упорядочены по значению ключа. Поиск быстрее, но вставка требует перестроения.
- **Хэшированное:** запись размещается в блоке, номер которого вычисляется хэш-функцией от ключа. Быстрый доступ по точному ключу.
##### 1.8-2 ==Последовательный метод доступа к данным БД==
**Последовательный метод доступа (sequential scan, table scan)** — простейший метод поиска данных: последовательный просмотр всех записей таблицы (или файла) до нахождения нужных.

**Принцип работы:**
- СУБД читает страницы таблицы одна за другой, проверяя каждую запись на соответствие условию.
- При поиске по неиндексированному полю — единственный способ.

**Преимущества:**
- Не требует дополнительных структур (индексов).
- Эффективен при выборке большого процента записей (обычно >10–15% таблицы).
- Минимальные накладные расходы при маленьких таблицах.

**Недостатки:**
- Крайне медленный для поиска единичных записей в больших таблицах.
- Скорость линейно зависит от размера таблицы — O(n).

**Когда применяется:**
- Таблица маленькая.
- Условие WHERE не использует индексированные поля.
- Условие WHERE выбирает значительную часть таблицы (например, все записи за год из многолетней таблицы без индекса по дате).
##### 1.8-3 ==Индексный метод доступа==
**Индексный метод доступа** использует дополнительные структуры (индексы) для ускорения поиска записей по ключу.

**Индекс** — структура данных, ассоциирующая значения ключа с указателями на записи таблицы. Аналогичен предметному указателю в книге.

**Основные типы индексов:**

**B-дерево (B-tree) и B+-дерево:**
- Наиболее распространённый тип. Сбалансированное дерево поиска.
- B+-дерево хранит все данные в листьях, внутренние узлы — только ключи и указатели. Листья связаны в список для быстрого обхода по диапазону.
- Операции поиска, вставки, удаления — O(log n).
- Поддерживает операции: точное совпадение (=), диапазон (BETWEEN), сортировку (ORDER BY), префиксный поиск (LIKE 'abc%').

**Хэш-индекс:**
- Основан на хэш-таблицах. Ключ преобразуется хэш-функцией в адрес блока.
- Очень быстрый для точного поиска (=) — O(1) в среднем.
- **Не поддерживает** диапазонный поиск, сортировку, поиск по части ключа.
##### 1.8-4 ==Производный метод доступа==
**Производный метод доступа** — метод, при котором адрес записи вычисляется на основе значения ключа, а не хранится в отдельной структуре. Основной представитель — хэширование.

**Принцип работы:**
- Значение первичного ключа подаётся на вход хэш-функции.
- Хэш-функция вычисляет номер страницы (блока), где находится запись.
- Прямой доступ к блоку без обхода дерева или сканирования.

**Преимущества:**
- Очень быстрый доступ по точному ключу — O(1) в идеальном случае.
- Не требует отдельного хранения индекса (в классическом статическом хэшировании).

**Недостатки:**
- Проблема коллизий (разные ключи дают одинаковый хэш). Требуются методы разрешения: цепочки переполнения, открытая адресация.
- **Не поддерживает** диапазонный поиск, частичное совпадение, сортировку.
- При изменении размера файла требуется перехэширование (дорогая операция).

**Динамическое хэширование:**  
Современные СУБД используют динамические методы (расширяемое хэширование, линейное хэширование), которые адаптируют структуру при росте данных без полного перехэширования.
#### 1.9 Свойства баз данных: целостность данных, эффективность, восстанавливаемость
База данных должна обладать тремя ключевыми свойствами:

Целостность данных — правильность и непротиворечивость хранимых данных. Обеспечивается ограничениями: PRIMARY KEY (уникальность), NOT NULL, FOREIGN KEY (ссылочная целостность), CHECK, UNIQUE, а также транзакциями. 

Эффективность — скорость выполнения запросов, измеряемая временем отклика и пропускной способностью. Достигается индексами, оптимизацией запросов, кэшированием, нормализацией/денормализацией. 

Восстанавливаемость — способность БД вернуться в согласованное состояние после сбоя. Обеспечивается журнализацией (WAL), резервным копированием, репликацией. Транзакции гарантируют ACID-свойства.
##### 1.9-1 ==Целостность данных (свойства баз данных)==
**Целостность данных** — свойство БД, гарантирующее, что данные в любой момент времени правильны, точны, непротиворечивы и соответствуют предметной области.

**Виды целостности:**

**Целостность сущностей (entity integrity):**  
Каждая таблица должна иметь первичный ключ. Ни один атрибут первичного ключа не может содержать NULL. Обеспечивается ограничением PRIMARY KEY.

**Ссылочная целостность (referential integrity):**  
Значение внешнего ключа должно либо совпадать с существующим значением первичного ключа связанной таблицы, либо быть NULL. Нельзя удалить запись, на которую есть ссылки, без каскадных действий. Обеспечивается FOREIGN KEY с опциями ON DELETE/ON UPDATE: CASCADE, SET NULL, RESTRICT, NO ACTION.

**Доменная целостность (domain integrity):**  
Значения атрибута должны принадлежать заданному домену (тип данных, диапазон, формат). Обеспечивается: тип данных столбца, CHECK, DEFAULT, NOT NULL.

**Пользовательская (бизнес-целостность):**  
Специфические правила предметной области, не покрываемые стандартными ограничениями. Обеспечивается триггерами, хранимыми процедурами, логикой приложения.

**Средства обеспечения целостности:**

- Декларативные ограничения (PRIMARY KEY, FOREIGN KEY, UNIQUE, CHECK, NOT NULL, DEFAULT).
- Триггеры (процедуры, автоматически выполняемые при INSERT/UPDATE/DELETE).
- Транзакции (атомарность и согласованность операций).
- Блокировки (предотвращение конфликтов конкурентного доступа).
##### 1.9-2 ==Эффективность (свойства баз данных)==
**Эффективность БД** — способность обрабатывать запросы с минимальными затратами времени и ресурсов.

**Основные показатели эффективности:**
- **Время отклика (response time):** время от отправки запроса до получения результата.
- **Пропускная способность (throughput):** количество транзакций в единицу времени (TPS — transactions per second).
- **Использование ресурсов:** загрузка CPU, памяти, дисковой подсистемы.

**Методы повышения эффективности:**

**На уровне проектирования:**
- **Нормализация:** устранение избыточности, ускорение модификации. Для аналитики — управляемая денормализация (предварительно вычисленные агрегаты, витрины данных).
- **Выбор типов данных:** минимально достаточные типы (INT вместо BIGINT, DATE вместо DATETIME).

**Индексирование:**
- Создание индексов на полях, участвующих в WHERE, JOIN, ORDER BY, GROUP BY.
- Избегание избыточных индексов (каждый индекс замедляет запись).

**Оптимизация запросов:**
- Избегание SELECT * (выбирать только нужные столбцы).
- Использование EXPLAIN для анализа плана запроса.
- Замена подзапросов на JOIN при необходимости.
- Избегание функций на индексированных полях в WHERE (WHERE YEAR(date) = 2024 не использует индекс).

**Кэширование:**
- Буферный пул СУБД (кэширование страниц в RAM).
- Кэширование результатов запросов на уровне приложения (Redis, Memcached).

**Аппаратные средства:**
- Увеличение объёма RAM (больше данных в буферном пуле).
- Использование SSD вместо HDD.
- Многоядерные процессоры для параллельной обработки запросов.
##### 1.9-3 ==Восстанавливаемость (свойства баз данных)==
**Восстанавливаемость** — способность БД вернуться в согласованное состояние после аппаратного или программного сбоя без потери данных.

**Виды сбоев:**
- **Транзакционные:** ошибка или отмена отдельной транзакции (откат).
- **Системные:** аварийное завершение СУБД (пропадание питания, kernel panic). Данные в оперативной памяти потеряны, дисковые данные могут быть повреждены.
- **Медийные:** физическое повреждение носителя (выход из строя диска).

**Механизмы восстановления:**

**Резервное копирование (backup):**
- **Полное (full backup):** полная копия БД на момент создания.
- **Инкрементное (incremental):** только изменения с момента последнего полного бэкапа.
- **Дифференциальное (differential):** изменения с момента последнего полного бэкапа.
- **Непрерывное архивирование (WAL archiving):** журналы транзакций копируются непрерывно, позволяя восстановление на любой момент времени (PITR — Point-in-Time Recovery).

**Журнализация (logging):**

- **WAL (Write-Ahead Logging):** все изменения записываются сначала в журнал, потом в данные. При сбое незавершённые транзакции откатываются по журналу, завершённые, но не записанные в данные — применяются повторно.
- **Контрольные точки (checkpoint):** периодический сброс изменённых страниц из памяти на диск. При восстановлении анализируется журнал от последней контрольной точки.
