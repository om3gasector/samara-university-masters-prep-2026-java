#### 5.1 Распределенная система. Причины создания, принципы построения, последствия распределенности, требования к распределенным системам
Распределенная система — набор независимых компьютеров, представляющийся пользователю единой системой. 
Причины создания: совместное использование ресурсов, повышение надёжности, масштабируемость, географическая распределённость. 
Принципы построения: прозрачность (доступа, местоположения, параллелизма, отказов), открытость, масштабируемость. 
Последствия: отсутствие единого времени, задержки, частичные отказы, гетерогенность. 
Требования: прозрачность, открытость, масштабируемость, безопасность, отказоустойчивость, управляемость.
##### 5.1-1 ==Распределенная система: определение==
**Распределенная система (Distributed System)** — совокупность автономных вычислительных узлов (компьютеров, серверов, устройств), соединённых сетью и взаимодействующих через обмен сообщениями, которая с точки зрения пользователя выглядит и ведёт себя как единая целостная система.

Ключевой признак: **единая логическая система на физически распределённых компонентах**. Пользователь не знает и не должен знать, на каком именно узле хранятся данные или выполняется его запрос.

Примеры: интернет (глобальная распределённая система), облачные сервисы (AWS, Google Cloud), социальные сети, поисковые системы, распределённые базы данных (Cassandra, MongoDB), блокчейн-сети.
##### 5.1-2 ==Причины создания распределенных систем==
**Совместное использование ресурсов:**  
Объединение вычислительных мощностей, систем хранения, принтеров, данных и программных сервисов. Пользователи разных узлов могут совместно использовать дорогостоящее оборудование и ПО.

**Повышение надёжности и отказоустойчивости:**  
При отказе одного узла его задачи могут быть переданы другим. Дублирование данных на нескольких узлах (репликация) защищает от потери информации. Система продолжает функционировать, возможно, с пониженной производительностью (graceful degradation).

**Масштабируемость:**  
Возможность увеличивать производительность системы добавлением новых узлов (горизонтальное масштабирование), а не заменой существующих на более мощные (вертикальное). Это экономически выгоднее и не имеет жёсткого потолка.

**Географическая распределённость:**  
Бизнес-процессы и пользователи распределены территориально. Локальные узлы обеспечивают быстрый доступ и соблюдение локальных законов (например, о хранении персональных данных).

**Экономическая эффективность:**  
Кластер из множества дешёвых серверов (commodity hardware) часто дешевле, чем один мейнфрейм сопоставимой мощности.
##### 5.1-3 ==Принципы построения распределенных систем==
**Прозрачность (transparency):**  
Сокрытие от пользователя и прикладных программ факта распределённости. Виды прозрачности:

- **Доступа:** единый способ доступа к локальным и удалённым ресурсам.
- **Местоположения:** доступ к ресурсу без знания его физического расположения.
- **Параллелизма:** несколько пользователей одновременно работают с общими ресурсами без взаимных помех.
- **Отказов:** сокрытие факта отказа отдельных компонентов (через репликацию, переподключение).
- **Репликации:** пользователь не знает о существовании копий данных.

**Открытость (openness):**  
Использование открытых стандартов и протоколов, обеспечивающих взаимодействие оборудования и ПО разных производителей. Примеры: HTTP, TCP/IP, REST API, gRPC.

**Масштабируемость (scalability):**  
Способность системы сохранять производительность и управляемость при росте числа узлов, пользователей и объёмов данных. Требует децентрализации алгоритмов и данных.

**Безопасность:**  
Защита данных и ресурсов от несанкционированного доступа. В распределённой среде сложнее: каналы связи прослушиваются, узлы могут быть скомпрометированы. Требуется шифрование, аутентификация, авторизация.
##### 5.1-4 ==Последствия распределенности==
Распределённость создаёт фундаментальные сложности, отличающие такие системы от централизованных:

**Отсутствие единого (глобального) времени:**  
Часы на разных узлах не синхронизированы идеально. Невозможно точно упорядочить события, произошедшие на разных узлах. Это усложняет логирование, отладку, определение причинно-следственных связей.

**Задержки в коммуникациях:**  
Передача сообщений по сети занимает непредсказуемое время. Задержки могут быть значительными (сотни миллисекунд для глобальных сетей). Это влияет на производительность и требует асинхронных моделей взаимодействия.

**Частичные отказы:**  
В распределённой системе часть узлов может выйти из строя, в то время как остальные продолжают работать. Централизованная система либо работает, либо нет. Распределённая — может находиться в промежуточном состоянии. Обнаружение отказов и восстановление — нетривиальная задача.

**Гетерогенность (разнородность):**  
Узлы могут отличаться аппаратно (процессоры, объём памяти), программно (ОС, версии библиотек), сетевыми протоколами. Обеспечение совместимости требует стандартизации (middleware).

**Конкурентный доступ:**  
Множество пользователей и процессов одновременно обращаются к общим ресурсам. Требуются механизмы синхронизации, управления транзакциями, разрешения конфликтов.
##### 5.1-5 ==Требования к распределенным системам==
**Прозрачность (transparency):**  
См. принципы построения. Пользователь и разработчик приложений не должны задумываться о распределённой природе системы.

**Открытость (openness):**  
Использование стандартизированных интерфейсов (API) и протоколов. Обеспечивает интероперабельность (взаимодействие систем разных производителей) и портируемость.

**Масштабируемость (scalability):**  
Возможность роста без деградации производительности. Масштабирование может быть:
- Вертикальным (более мощное железо) — ограничено.
- Горизонтальным (больше узлов) — предпочтительно.

**Безопасность:**  
Обеспечение конфиденциальности, целостности и доступности данных и сервисов. Управление идентификацией, аутентификацией и авторизацией пользователей. Защита каналов связи.

**Отказоустойчивость (fault tolerance):**  
Способность продолжать работу при отказе отдельных компонентов. Достигается через: репликацию данных, избыточность (аппаратную и программную), механизмы обнаружения и восстановления после отказов.

**Управляемость (manageability):**  
Возможность администрирования, мониторинга, обновления и диагностики распределённой системы как единого целого.

**Согласованность (consistency):**  
При репликации данных пользователи должны видеть актуальное и непротиворечивое состояние данных. Баланс между строгой согласованностью (сложно и медленно) и eventual consistency (проще и быстрее).
#### 5.2 Распределенная система. Сложности при реализации, модели распределенных систем, архитектура и основные принципы, преодоление сложности

Распределенная система - это группа независимых компьютеров или узлов, соединенных по сети. Они работают сообща для решения общей задачи, но для конечного пользователя или приложения вся эта сеть выглядит как единое цело

Основные сложности реализации: частичные отказы, задержки сети, отсутствие глобального времени, синхронизация, конкурентный доступ, гетерогенность, безопасность. Модели: клиент-сервер, многоуровневая (N-tier), peer-to-peer (P2P), сервис-ориентированная (SOA, микросервисы), publish-subscribe. Архитектурные принципы: модульность, слабая связанность, stateless-сервисы, асинхронное взаимодействие, репликация, балансировка нагрузки.

Преодоление сложности: middleware, стандартизированные протоколы, контейнеризация и оркестрация, мониторинг, отказоустойчивые алгоритмы (Paxos, Raft), CAP-теорема и осознанный выбор компромиссов.
##### 5.2-1 ==Сложности при реализации распределенных систем==
Перечисленные в вопросе 4.1 последствия распределённости порождают конкретные инженерные сложности:

**Частичные отказы:**
- Нельзя достоверно отличить «узел упал» от «узел медленно отвечает», «сеть не работает» от «узел не работает».
- Необходимы механизмы обнаружения отказов (heartbeat, таймауты) и восстановления.

**Сетевые задержки и негарантированная доставка:**
- Сообщения могут задерживаться, теряться, дублироваться, приходить в другом порядке.
- Требуется проектировать систему с учётом этих возможностей (идемпотентность, повторные отправки).

**Отсутствие глобального времени:**
- Невозможно установить точный порядок событий на разных узлах.
- Используются логические часы (Lamport timestamps) и векторные часы для определения причинно-следственных связей.

**Синхронизация и координация:**
- Сложно обеспечить атомарность действий на нескольких узлах.
- Распределённые транзакции (2PC, 3PC) сложны и имеют проблемы с производительностью и отказами координатора.

**Конкурентный доступ и согласованность:**
- Несколько клиентов одновременно изменяют одни и те же данные.
- Необходимы: распределённые блокировки, управление версиями, алгоритмы консенсуса.

**Гетерогенность среды:**
- Разные ОС, версии библиотек, часовые пояса, кодировки.
- Требуется стандартизация форматов данных и протоколов.
##### 5.2-2 ==Модели распределенных систем==
**Клиент-сервер (Client-Server):**
- Сервер предоставляет ресурсы и сервисы, клиент их запрашивает.
- Сервер пассивен (ждёт запросы), клиент активен.
- Пример: веб-сайт (браузер — веб-сервер).

**Многоуровневая архитектура (N-tier):**
- Развитие клиент-сервер: между клиентом и сервером БД — промежуточные слои (бизнес-логика).
- Три уровня (3-tier): presentation (UI), application (логика), data (БД).
- Повышает модульность, масштабируемость, независимость уровней.

**Peer-to-Peer (P2P):**
- Все узлы равноправны (peer), могут быть и клиентами, и серверами.
- Нет центрального узла (децентрализация).
- Примеры: BitTorrent, блокчейн, IPFS.
- Сложности: поиск ресурсов, безопасность (доверие к пирам).

**Сервис-ориентированная архитектура (SOA) и микросервисы:**
- Функциональность разбивается на независимые слабосвязанные сервисы.
- Взаимодействие через стандартные протоколы (HTTP/REST, gRPC, очереди сообщений).
- Микросервисы — эволюция SOA: сервисы мельче, полностью независимы (каждый со своей БД), развёртываются независимо.
##### 5.2-3 ==Архитектура и основные принципы==
**Слабая связанность (loose coupling):**  
Компоненты системы минимально зависят друг от друга. Изменение одного компонента не требует переделки других. Достигается через чёткие API и асинхронное взаимодействие.

**Stateless-сервисы:**  
Сервер не хранит состояние клиента между запросами. Каждый запрос содержит всю необходимую информацию. Упрощает масштабирование (любой экземпляр сервиса может обработать запрос).

**Репликация:**  
Данные дублируются на нескольких узлах для повышения отказоустойчивости и производительности чтения. Виды: master-slave, multi-master. Требует механизмов синхронизации.

**Балансировка нагрузки:**  
Входящие запросы распределяются между несколькими экземплярами сервиса. Повышает пропускную способность и отказоустойчивость.

**Асинхронное взаимодействие:**  
Отправитель не блокируется в ожидании ответа. Ответ может прийти позже. Повышает производительность и устойчивость. Реализуется через очереди сообщений (Message Queue).

**Шардирование (sharding):**  
Разделение данных на непересекающиеся части (шарды) и размещение на разных узлах. Позволяет горизонтально масштабировать хранение данных.
##### 5.2-4 ==Преодоление сложности (распределенные системы)==
**Middleware (промежуточное ПО):**  
Программный слой между ОС и приложением, скрывающий сложности распределённого взаимодействия. Предоставляет: RPC, распределённые транзакции, очереди сообщений, управление распределёнными объектами. Примеры: Java RMI, CORBA, современные — Kafka, gRPC.

**Контейнеризация и оркестрация:**  
Docker — упаковка сервисов в изолированные контейнеры. Kubernetes — автоматическое развёртывание, масштабирование и управление контейнерами. Упрощает эксплуатацию микросервисов.

**Стандартизированные протоколы:**  
HTTP/REST, gRPC (на основе Protobuf), WebSocket, AMQP (очереди сообщений). Открытые стандарты обеспечивают совместимость и проверены временем.

**Алгоритмы консенсуса:**  
Позволяют группе узлов прийти к единому решению даже при отказах. Paxos (теоретическая основа), Raft (более понятный, используется в etcd, Consul). Применяются для репликации логов, выбора лидера.
#### 5.3 JDBC. Типы драйверов. Основные цели интерфейса JDBC. Примеры использования JDBC. Основы программирования JDBC. Предварительно подготовленные команды. Метаданные
JDBC (Java Database Connectivity) — стандартный Java API для взаимодействия с реляционными БД. Цели: универсальный доступ к разным СУБД через единый интерфейс, независимость от конкретной БД, переносимость. 

Четыре типа драйверов: тип 1 (ODBC-мост), тип 2 (native API), тип 3 (сетевой протокол), тип 4 (чистый Java-драйвер, предпочтителен). Основы программирования: загрузка драйвера, Connection, Statement/PreparedStatement, ResultSet. 

PreparedStatement — предварительно скомпилированный SQL, быстрее и безопаснее (защита от SQL-инъекций). Метаданные: DatabaseMetaData (информация о СУБД) и ResultSetMetaData (структура результата запроса).
##### 5.3-1 ==JDBC: определение и основные цели==
**JDBC (Java Database Connectivity)** — это стандартный Java API (набор интерфейсов и классов в пакете java.sql), предоставляющий универсальный доступ к реляционным базам данных из Java-приложений. JDBC является мостом между Java-кодом и конкретной СУБД.

**Основные цели:**

- **Универсальность:** единый интерфейс для работы с любыми реляционными БД (Oracle, PostgreSQL, MySQL, MSSQL). Разработчик пишет один и тот же код, меняется только драйвер.
- **Независимость от платформы и СУБД:** приложение можно перенести с одной БД на другую, изменив только строку подключения и драйвер (при использовании стандартного SQL).
- **Простота и производительность:** предоставляет простые абстракции (Connection, Statement, ResultSet) и оптимизированные механизмы (пулы соединений, пакетная обработка).
- **Поддержка всех операций SQL:** выполнение запросов (SELECT), модификация данных (INSERT, UPDATE, DELETE), вызов хранимых процедур, управление транзакциями.
##### 5.3-2 ==Типы драйверов JDBC==
- **Тип 1 (JDBC-ODBC Bridge):** преобразует вызовы JDBC в ODBC. Требует ODBC-драйвер на клиенте. Медленный, платформозависимый, устарел (удалён из Java 8+).
- **Тип 2 (Native API):** использует нативное клиентское API СУБД (C/C++). Требует установки клиентской библиотеки. Быстрее типа 1, но платформозависим.
- **Тип 3 (Network Protocol):** использует промежуточный сервер-посредник. Клиент общается с сервером по независимому протоколу, сервер — с СУБД. Не требует клиентских библиотек.
- **Тип 4 (Thin Driver, чистый Java):** полностью на Java, напрямую общается с СУБД по её сетевому протоколу. Не требует установки дополнительного ПО. Самый распространённый и рекомендуемый. Пример: PostgreSQL JDBC Driver, MySQL Connector/J.
##### 5.3-3 ==Основы программирования JDBC==
**Стандартная последовательность работы с JDBC:**

1. **Загрузка драйвера** (с Java 6 — автоматически через ServiceLoader):
```
Class.forName("org.postgresql.Driver"); // для старых версий
```
2. **Установка соединения:**
```
Connection conn = DriverManager.getConnection(
	"jdbc:postgresql://localhost:5432/mydb", "user", "password");
```
3. **Создание Statement или PreparedStatement:**
```
Statement stmt = conn.createStatement();
```
4. **Выполнение SQL-запроса:**
```
ResultSet rs = stmt.executeQuery("SELECT * FROM users");
```

**Основные интерфейсы:**
- **Connection:** соединение с БД, управление транзакциями (commit, rollback, setAutoCommit).
- **Statement:** выполнение статического SQL (executeQuery, executeUpdate, execute).
- **PreparedStatement:** предварительно скомпилированный SQL, параметризованные запросы.
- **CallableStatement:** вызов хранимых процедур.
- **ResultSet:** курсор для чтения результатов SELECT. Методы: next(), getInt(), getString() и т.д.
##### 5.3-4 ==Предварительно подготовленные команды (PreparedStatement)==
**PreparedStatement** — подынтерфейс Statement, представляющий предварительно скомпилированный SQL-запрос с параметрами (плейсхолдерами `?`).

**Преимущества перед обычным Statement:**
- **Защита от SQL-инъекций:** значения параметров автоматически экранируются, злоумышленник не может внедрить вредоносный SQL-код.
- **Производительность:** запрос компилируется один раз, выполняется многократно с разными параметрами (полезно в циклах).
- **Читаемость кода:** отделение SQL от значений.

**Пример:**
```
String sql = "INSERT INTO users (name, email) VALUES (?, ?)";
PreparedStatement pstmt = conn.prepareStatement(sql);
pstmt.setString(1, "Иванов");
pstmt.setString(2, "ivanov@example.com");
int rows = pstmt.executeUpdate();
```

Методы установки параметров: setInt(), setString(), setDouble(), setDate(), setNull() и т.д. Параметры нумеруются с 1.
##### 5.3-5 ==Метаданные (JDBC)==
JDBC предоставляет два типа метаданных — информацию о структуре БД и результатах запросов.

**DatabaseMetaData — информация о СУБД и БД:**  
Получается через `conn.getMetaData()`. Предоставляет:
- Название СУБД и версию.
- Список таблиц, схем, каталогов.
- Информацию о поддерживаемых типах данных, функциях, возможностях.
- Информацию о первичных и внешних ключах, индексах.
##### 5.3-6 ==Примеры использования JDBC==
**Основные сферы применения:**
- **Веб-приложения:** Серверные приложения используют JDBC для извлечения данных из БД (например, профилей пользователей) и последующего отображения их на веб-страницах.
- **Системы обработки транзакций:** Банковские системы используют JDBC (транзакции `Connection.setAutoCommit(false)` и `commit()`) для безопасного перевода средств между счетами.
- **Enterprise-фреймворки:** Такие технологии, как Hibernate или Spring Data JPA, скрывают рутинную работу под капотом, но внутри себя для связи с базой данных они всё равно используют низкоуровневый JDBC.
- **Массовые операции (Batching):** Пакетная загрузка большого количества строк в хранилище данных выполняется эффективнее с применением метода `addBatch()`.
#### 5.4 XML. Отличия XML от HTML. Структура XML-документа. Пример XML. Создание XML-документов. Пример
XML (eXtensible Markup Language) — расширяемый язык разметки, предназначенный для хранения и передачи структурированных данных. В отличие от HTML, который описывает отображение данных, XML описывает сами данные и их структуру. Теги в XML не предопределены, разработчик создаёт собственные. Структура: пролог, корневой элемент, вложенные элементы с атрибутами. XML-документ должен быть well-formed (синтаксически правильным). Применяется для обмена данными между системами, конфигурационных файлов, хранения данных.
##### 5.4-1 ==XML: определение и назначение==
**XML (eXtensible Markup Language)** — расширяемый язык разметки, рекомендованный консорциумом W3C в 1998 году. Это текстовый формат для представления структурированных данных в виде, удобном для чтения как человеком, так и машиной.

**Основное назначение XML:**
- Хранение и передача данных между различными системами и платформами (основа веб-сервисов, SOAP, REST/XML).
- Конфигурационные файлы (web.xml, pom.xml, hibernate.cfg.xml).
- Обмен данными между приложениями (экспорт/импорт).
- Документооборот (форматы Office Open XML, OpenDocument).

XML не выполняет никаких действий — он только описывает данные. Для обработки XML используются парсеры (DOM, SAX, StAX), языки запросов (XPath, XQuery) и преобразований (XSLT).
##### 5.4-2 ==Отличия XML от HTML==
XML и HTML оба являются языками разметки и синтаксически похожи (теги в угловых скобках), но имеют принципиально разные цели и свойства.

|Характеристика|XML|HTML|
|---|---|---|
|Назначение|Описание структуры и хранение данных|Отображение данных (разметка веб-страниц)|
|Теги|Произвольные, определяются разработчиком|Предопределённый набор стандартных тегов|
|Регистр тегов|Чувствителен к регистру (`<Name>` ≠ `<name>`)|Нечувствителен (`<P>` = `<p>`)|
|Закрытие тегов|Обязательно для всех тегов (включая пустые)|Допускается незакрытие некоторых тегов (`<br>`, `<img>`)|
|Атрибуты|Значения обязательно в кавычках|Допускается без кавычек|
|Строгость синтаксиса|Документ должен быть well-formed (строгие правила)|Браузеры терпимы к ошибкам (исправляют)|
|Ориентация|Данные (data)|Представление (presentation)|

XML не заменяет HTML. HTML используется для веб-интерфейсов, XML — для передачи данных за кулисами. Современный HTML5 не основан на XML (хотя существует XHTML — вариант HTML, соответствующий правилам XML).
##### 5.4-3 ==Структура XML-документа==
XML-документ состоит из следующих частей (сверху вниз):

**1. Пролог (необязателен):**
- XML-декларация: `<?xml version="1.0" encoding="UTF-8"?>`. Указывает версию XML и кодировку.
- Может содержать объявление DTD или ссылку на схему.

**2. Корневой элемент (root element):**
- Единственный элемент верхнего уровня, заключающий весь документ.
- Все остальные элементы — его потомки.

**3. Элементы (elements):**
- Основные строительные блоки XML.
- Состоят из открывающего и закрывающего тегов (или пустой тег `<tag/>`).
- Могут содержать: текст, другие элементы (вложенность), атрибуты.
- Древовидная иерархическая структура: родительские, дочерние, сестринские элементы.

**4. Атрибуты (attributes):**
- Пары имя="значение" внутри открывающего тега.
- Содержат дополнительные сведения об элементе.
- Значение всегда в кавычках.

**5. Текстовые узлы:**
- Символьные данные между тегами.

**6. Комментарии:**
- `<!-- комментарий -->`.
##### 5.4-4 ==Создание XML-документов. Пример==
XML-документ можно создать в любом текстовом редакторе. Файл сохраняется с расширением `.xml`. Специальных инструментов не требуется, но существуют XML-редакторы с подсветкой синтаксиса и валидацией.

**Способы создания в Java:**
- Ручная генерация строки (не рекомендуется для сложных документов).
- DOM-парсер (построение дерева в памяти с последующей записью в файл).
- StAX (потоковая запись).
- JAXB (сериализация Java-объектов в XML по аннотациям).
- Библиотеки: Jackson XML, XStream.

**Пример XML-документа (конфигурация сервера):**
```
<?xml version="1.0" encoding="UTF-8"?>
<server>
    <name>WebServer</name>
    <port>8080</port>
    <security>
        <ssl enabled="true">
            <certificate>/etc/ssl/cert.pem</certificate>
        </ssl>
    </security>
    <databases>
        <database type="postgresql" host="localhost" port="5432" name="mydb"/>
        <database type="redis" host="localhost" port="6379"/>
    </databases>
</server>
```
Этот пример иллюстрирует: пролог с кодировкой, корневой элемент `<server>`, вложенные элементы (`<security>`, `<databases>`), атрибуты (`enabled="true"`, `type="postgresql"`), пустой элемент (`<database ... />`), текстовые узлы (`WebServer`, `8080`).
#### 5.5 DTD и XML-схема. Примеры
DTD (Document Type Definition) и XML Schema — способы описания структуры XML-документа. 
DTD — устаревший, простой синтаксис (не XML), описывает элементы, атрибуты, порядок вложенности. Не поддерживает типы данных и пространства имён. 

XML Schema (XSD) — современный стандарт, написан на самом XML, поддерживает типы данных (string, integer, date), пространства имён, наследование, ограничения значений. DTD достаточно для простых конфигураций, XSD применяется в веб-сервисах, сложных форматах данных и валидации.
##### 5.5-1 ==DTD (определение типа документа)==
**DTD (Document Type Definition)** — определение типа документа. Это набор правил, описывающих допустимую структуру XML-документа: какие элементы могут в нём присутствовать, в каком порядке, какие у них могут быть атрибуты.

**Способы подключения DTD:**
- **Внутреннее DTD:** правила записываются непосредственно внутри самого XML-документа в секции `<!DOCTYPE>`.
- **Внешнее DTD:** правила хранятся в отдельном файле (обычно `.dtd`), а в XML-документе указывается ссылка на этот файл.

**Основные конструкции DTD:**
- **Объявление элемента:** `<!ELEMENT имя (содержимое)>`. Содержимое может быть: `(#PCDATA)` — текст, `(дочерние_элементы)` — вложенные элементы, `EMPTY` — пустой элемент, `ANY` — любое содержимое. Порядок и количество задаются символами: запятая — строгая последовательность, `|` — выбор, `?` — ноль или один, `*` — ноль или более, `+` — один или более.
- **Объявление атрибута:** `<!ATTLIST элемент атрибут тип_атрибута умолчание>`. Типы: `CDATA` (строка), `ID` (уникальный идентификатор), `IDREF` (ссылка на ID), перечисление значений в скобках. Умолчание: `#REQUIRED` (обязателен), `#IMPLIED` (необязателен), конкретное значение по умолчанию.

**Недостатки DTD:**
- Не является XML-документом (требует изучения отдельного синтаксиса).
- Не поддерживает типы данных (числа, даты).
- Нет поддержки пространств имён.
- Слабая расширяемость.
##### 5.5-2 ==XML-схема (XML Schema)==
**XML Schema (XSD — XML Schema Definition)** — стандарт, рекомендованный W3C для описания структуры и типов данных XML-документов. Сама схема представляет собой XML-документ и подчиняется правилам XML.

**Преимущества перед DTD:**
- Написана на самом XML (единый синтаксис для данных и их описания).
- Поддерживает пространства имён.
- Богатая система типов данных: string, integer, decimal, boolean, date, time, и возможность создавать пользовательские типы (простые и комплексные).
- Возможность задавать ограничения на значения (диапазон, длина, шаблон регулярного выражения).
- Поддержка наследования и переопределения типов.

**Основные понятия XSD:**
- **Элементы:** объявляются тегом `<xs:element name="имя" type="тип"/>`.
- **Типы данных:**
    - Простые типы (`<xs:simpleType>`) — элементы, содержащие только текст (без вложенных элементов). Могут иметь ограничения (`<xs:restriction>`): минимум/максимум, длина, перечисление (`<xs:enumeration>`).
    - Комплексные типы (`<xs:complexType>`) — элементы, содержащие вложенные элементы и/или атрибуты. Задают последовательность (`<xs:sequence>`), выбор (`<xs:choice>`), произвольный набор (`<xs:all>`).
- **Атрибуты:** объявляются внутри `<xs:complexType>` тегом `<xs:attribute name="имя" type="тип" use="required|optional"/>`.
- **Ограничения частоты:** атрибуты `minOccurs` и `maxOccurs` (по умолчанию 1).
##### 5.5-3 ==Отличия DTD от XML-схемы==
|Характеристика|DTD|XML Schema (XSD)|
|---|---|---|
|Синтаксис|Собственный (не XML)|XML|
|Типы данных|Только строки (CDATA)|Богатый набор (string, int, date и др.)|
|Пространства имён|Не поддерживает|Поддерживает|
|Ограничения значений|Нет|Диапазон, длина, шаблон, перечисление|
|Наследование типов|Нет|Расширение и ограничение типов|
|Элементы и атрибуты|Базовые возможности|Гибкое управление порядком и частотой|
|Современный статус|Устарел, но используется в простых случаях|Современный стандарт|
##### 5.5-4 ==Примеры DTD и XML-схемы==
**Пример DTD:**

Файл `library.dtd`:
```
<!ELEMENT library (book+)>
<!ELEMENT book (title, author, year)>
<!ATTLIST book isbn CDATA #REQUIRED>
<!ELEMENT title (#PCDATA)>
<!ELEMENT author (#PCDATA)>
<!ELEMENT year (#PCDATA)>
```

Подключение в XML: `<!DOCTYPE library SYSTEM "library.dtd">`.

**Пример XML Schema (XSD):**

Файл `library.xsd`:
```
<?xml version="1.0" encoding="UTF-8"?>
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema">
    <xs:element name="library">
        <xs:complexType>
            <xs:sequence>
                <xs:element name="book" maxOccurs="unbounded">
                    <xs:complexType>
                        <xs:sequence>
                            <xs:element name="title" type="xs:string"/>
                            <xs:element name="author" type="xs:string"/>
                            <xs:element name="year" type="xs:integer"/>
                        </xs:sequence>
                        <xs:attribute name="isbn" type="xs:string" use="required"/>
                    </xs:complexType>
                </xs:element>
            </xs:sequence>
        </xs:complexType>
    </xs:element>
</xs:schema>
```
Подключение в XML:
```
<library xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:noNamespaceSchemaLocation="library.xsd">
```
#### 5.6 Dom и SAX. Примеры
DOM (Document Object Model) и SAX (Simple API for XML) — два основных подхода к разбору XML в Java. DOM читает весь документ, строит в памяти дерево объектов — можно перемещаться, изменять, удалять узлы. Минус: потребляет много памяти на больших файлах. 

SAX — событийный парсер: читает документ последовательно, генерирует события (начало элемента, конец элемента, текст), которые обрабатываются методами обработчика. Плюс: низкое потребление памяти, быстрый. Минус: нельзя вернуться назад, только чтение. Выбор: 
DOM — когда нужен произвольный доступ и изменение структуры. SAX — для однократного чтения больших файлов или потоковой обработки.
##### 5.6-1 ==DOM (Document Object Model): принцип и особенности==
**DOM (Document Object Model)** — объектная модель документа. При разборе XML DOM-парсер читает весь документ и строит в оперативной памяти полное древовидное представление. Каждый узел (элемент, атрибут, текст, комментарий) становится объектом, иерархически связанным с родителями и потомками.

**Принцип работы:**
1. XML-файл полностью загружается в память.
2. Строится дерево узлов (объектов класса Node и его наследников: Element, Attr, Text).
3. Разработчик программно перемещается по дереву: от корня к дочерним узлам, ищет элементы по имени, читает и изменяет значения, добавляет и удаляет узлы.
4. После модификации дерево можно сериализовать обратно в XML (записать в файл).

**Особенности:**
- **Полный доступ:** можно читать, изменять, удалять, создавать любые части документа.
- **Произвольная навигация:** переходы между любыми узлами в любом направлении.
- **Потребление памяти:** для больших XML-файлов (десятки и сотни мегабайт) потребление RAM может быть неприемлемым — дерево занимает в 3–10 раз больше памяти, чем исходный файл.
- **Относительно медленная загрузка:** перед началом работы нужно дождаться построения всего дерева.
##### 5.6-2 ==SAX (Simple API for XML): принцип и особенности==
**SAX (Simple API for XML)** — событийно-ориентированный парсер. Он не строит никакой модели в памяти, а последовательно читает XML-файл и вызывает методы обработчика при наступлении определённых событий.

**Принцип работы:**
1. Создаётся обработчик — объект, реализующий интерфейс `org.xml.sax.helpers.DefaultHandler`.
2. Обработчик содержит методы, вызываемые парсером: `startDocument()` (начало документа), `startElement()` (открывающий тег), `characters()` (текстовое содержимое), `endElement()` (закрывающий тег), `endDocument()` (конец документа).
3. SAX-парсер читает XML-файл последовательно, от начала до конца. При обнаружении начала элемента вызывает `startElement()`, при обнаружении текста — `characters()`, при закрытии элемента — `endElement()`.
4. Разработчик в этих методах реализует нужную логику: накапливает данные, заполняет свои структуры.

**Особенности:**
- **Низкое потребление памяти:** в памяти находится только текущий обрабатываемый фрагмент. Подходит для файлов гигабайтного размера.
- **Высокая скорость:** не тратит время на построение дерева объектов.
- **Только чтение:** нельзя изменить документ, вернуться к предыдущему элементу.
- **Сложность обработки:** разработчик сам должен отслеживать контекст (в каком элементе находимся, на каком уровне вложенности). Код обработчика для сложной структуры становится громоздким.
- **Последовательная обработка:** для извлечения нужных данных требуется один полный проход документа.
##### 5.6-3 ==Отличия DOM от SAX==
|Характеристика|DOM|SAX|
|---|---|---|
|Модель работы|Объектное дерево в памяти|События (callbacks)|
|Направление|Произвольный доступ, навигация|Только последовательно вперёд|
|Изменение XML|Полная поддержка (чтение + запись)|Только чтение|
|Потребление памяти|Высокое|Низкое|
|Скорость загрузки|Медленнее (построение дерева)|Быстрее|
|Сложность кода|Проще для сложной навигации|Сложнее (ручное отслеживание контекста)|
|Типичный сценарий|Конфигурационные файлы, небольшие документы, редактирование|Большие файлы, потоковая обработка, однократное извлечение данных|
**Компромиссный вариант — StAX (Streaming API for XML):**  
StAX занимает промежуточную позицию. Работает по принципу pull: разработчик сам запрашивает следующий элемент (типа итератора), а не получает события (как в SAX). Позволяет и читать, и писать XML. Потребление памяти низкое, код проще, чем SAX. В современных проектах StAX часто предпочтительнее SAX.
##### 5.6-4 ==Примеры использования DOM и SAX==
**Пример использования DOM (на языке Python)**

DOM загружает весь файл в оперативную память. С ним можно работать как с привычной структурой данных: искать элементы, добавлять новые теги или удалять узлы.

```
import xml.dom.minidom

# Загружаем и парсим весь XML-документ в память
domtree = xml.dom.minidom.parse("items.xml")
collection = domtree.documentElement
items = collection.getElementsByTagName("item")

# Навигация по дереву DOM
for item in items:
    if item.hasAttribute("id"):
        print(f"ID товара: {item.getAttribute('id')}")
        
    # Извлечение текста из дочерних тегов
    name = item.getElementsByTagName('name')[0].childNodes[0].data
    price = item.getElementsByTagName('price')[0].childNodes[0].data
    print(f"Название: {name}, Цена: {price}")
```

**Пример использования SAX (на языке Python)**

SAX работает как поток событий. Он не создает структуру в памяти. Вместо этого вы задаете класс-обработчик с методами, которые вызываются в моменты, когда парсер натыкается на открывающий тег, текст внутри него или закрывающий тег.

```
import xml.sax

# Создаем обработчик событий
class ItemHandler(xml.sax.ContentHandler):
    def __init__(self):
        self.CurrentData = ""
        self.name = ""
        self.price = ""

    # Срабатывает при начале тега
    def startElement(self, tag, attributes):
        self.CurrentData = tag
        if tag == "item":
            print(f"--- Товар с ID: {attributes.get('id', '')} ---")

    # Срабатывает при обнаружении текста между тегами
    def characters(self, content):
        if self.CurrentData == "name":
            self.name = content
        elif self.CurrentData == "price":
            self.price = content

    # Срабатывает при закрытии тега
    def endElement(self, tag):
        if self.CurrentData == "name":
            print(f"Название: {self.name}")
        elif self.CurrentData == "price":
            print(f"Цена: {self.price}")

# Передаем обработчик парсеру
parser = xml.sax.make_parser()
Handler = ItemHandler()
parser.setContentHandler(Handler)
parser.parse("items.xml")
```
#### 5.7 ORM. Достоинства и недостатки. Hibernate. Файл hibernate.cfg.xml. Пример
ORM (Object-Relational Mapping) — технология отображения объектов приложения на таблицы реляционной БД. 
Достоинства: ускорение разработки, абстрагирование от SQL, автоматическое управление связями, кэширование, переносимость между СУБД. 
Недостатки: снижение производительности на сложных запросах, риск N+1 problem, потеря контроля над SQL, порог входа.
##### 5.7-1 ==ORM: определение, достоинства и недостатки==
**ORM (Object-Relational Mapping)** — это технология программирования, которая связывает объектно-ориентированную модель приложения с реляционной моделью базы данных. ORM автоматически преобразует объекты в строки таблиц (и обратно), генерирует SQL-запросы, управляет связями между объектами (one-to-one, one-to-many, many-to-many).

**Достоинства:**
- **Скорость разработки:** разработчик работает с объектами и их методами, а не пишет SQL вручную. Меньше шаблонного кода.
- **Абстрагирование от СУБД:** смена базы данных (например, с PostgreSQL на MySQL) часто требует только изменения диалекта в конфигурации, без переписывания кода.
- **Автоматическое управление связями:** ORM автоматически загружает связанные объекты (ленивая или жадная загрузка), каскадно сохраняет и удаляет зависимые сущности.
- **Кэширование:** встроенные механизмы кэша первого уровня (сессия) и второго уровня (между сессиями) снижают количество обращений к БД.
- **Миграции и валидация:** многие ORM поддерживают автоматическое создание и обновление схемы БД по описанию сущностей.

**Недостатки:**
- **Снижение производительности:** генерируемый SQL не всегда оптимален. Для сложных аналитических запросов и массовых операций ORM проигрывает ручному SQL.
- **Проблема N+1 запросов:** ORM может выполнить один запрос для получения списка сущностей, а затем N дополнительных запросов для загрузки связанных данных для каждой сущности. Лечится явным указанием способа загрузки (JOIN FETCH, Entity Graph).
- **Потеря контроля:** разработчик не видит, какой именно SQL выполняется, и не всегда может его оптимизировать без глубокого понимания внутренностей ORM.
- **Высокий порог входа:** требуется изучить саму ORM, её конфигурацию, управление сессиями, транзакциями, жизненным циклом сущностей.
- **Неудобство для legacy-схем:** если БД спроектирована без учёта ORM (составные ключи, нетипичные связи), маппинг становится сложным.
##### 5.7-2 ==Hibernate: назначение и особенности==
**Hibernate** — наиболее популярная реализация ORM для Java. Также является реализацией спецификации JPA (Jakarta Persistence API) — стандартного Java API для ORM.

**Назначение:** полное объектно-реляционное отображение, управление персистентностью Java-объектов, генерация SQL.

**Ключевые особенности:**
- **Автоматическое создание схемы БД:** на основе маппинга сущностей может генерировать DDL-скрипты (create, update, validate).
- **Кэш первого уровня:** встроенный кэш сессии (по умолчанию).
- **Кэш второго уровня:** подключаемый (Ehcache, Hazelcast), кэширует данные между сессиями.
- **Ленивая загрузка (lazy loading):** связанные сущности подгружаются только при обращении к ним. Реализована через прокси-объекты.
- **Управление версиями (optimistic locking):** через поле-версию (`@Version`) предотвращает конфликты параллельного доступа.
##### 5.7-3 ==Файл hibernate.cfg.xml==
**hibernate.cfg.xml** — основной конфигурационный файл Hibernate. Обычно размещается в classpath (`src/main/resources`).

**Основные секции конфигурации:**
- **Настройки подключения к БД:** драйвер (hibernate.connection.driver_class), URL (hibernate.connection.url), пользователь, пароль, размер пула соединений.
- **Диалект SQL (hibernate.dialect):** указывает Hibernate, с какой СУБД он работает, чтобы генерировать оптимальный SQL (PostgreSQL, MySQL, Oracle, H2 и т.д.).
- **Управление схемой (hibernate.hbm2ddl.auto):** что делать со схемой БД при старте — validate (проверить), update (обновить), create (создать заново, удаляя данные), create-drop (создать и удалить при остановке).
- **Настройки кэша:** кэш второго уровня (hibernate.cache.use_second_level_cache), провайдер кэша, кэш запросов.
- **Логирование SQL:** показывать ли SQL-запросы в логах (hibernate.show_sql), форматировать ли их (hibernate.format_sql).
- **Маппинг сущностей:** перечисление классов-сущностей или указание пакетов для сканирования (`<mapping class="..."/>`).
##### 5.7-4 ==Пример использования Hibernate==
**Основные шаги работы с Hibernate:**

1. **Создание SessionFactory** (один раз при старте приложения, тяжёлый объект):
```
SessionFactory sessionFactory = new Configuration()
    .configure("hibernate.cfg.xml")
    .buildSessionFactory();
```
2. **Открытие сессии и транзакции:**
```
Session session = sessionFactory.openSession();
Transaction tx = session.beginTransaction();
```
3. **CRUD-операции с сущностями:**
- Сохранение: `session.save(user);` / `session.persist(user);`
- Загрузка по ID: `User u = session.get(User.class, id);`
- Обновление: `session.update(user);` / `session.merge(user);`
- Удаление: `session.delete(user);`
- Запрос HQL: `session.createQuery("FROM User WHERE age > :age", User.class).setParameter("age", 18).list();`

4. **Фиксация транзакции и закрытие сессии:**
```
tx.commit();
session.close();
```
#### 5.8 Servlet, Servlet container, Distributed servlet container, Servlet context, Servlet mapping. Жизненный цикл сервлетов. Особенности сервлетов. Объекты запроса и отклика. HTTP Servlet. Пример сервлета. Поддержка сессий. Общий дескриптор развёртывания web.xml. Фильтрация
Сервлет — Java-класс, обрабатывающий HTTP-запросы на сервере. Контейнер сервлетов (Tomcat, Jetty) управляет их жизненным циклом: загрузка → init() → service() (вызывается при каждом запросе) → destroy(). 
ServletContext — общее пространство для всех сервлетов приложения. Servlet mapping связывает URL с конкретным сервлетом. 
HttpServlet — базовый класс для обработки HTTP (методы doGet, doPost). Объекты запроса (HttpServletRequest) и отклика (HttpServletResponse) инкапсулируют данные запроса и формируют ответ. 
Сессии поддерживаются через HttpSession (getSession()). web.xml — дескриптор развёртывания (конфигурация сервлетов, маппинг, фильтры). Фильтры перехватывают запросы до и после сервлета (логирование, аутентификация, сжатие).
##### 5.8-1 ==Servlet: определение и особенности==
**Сервлет (Servlet)** — это Java-класс, выполняющийся на стороне сервера и обрабатывающий входящие запросы от клиентов (как правило, HTTP-запросы от браузера). Сервлет является фундаментальным строительным блоком Java-веб-приложений (Jakarta Servlet API, ранее Java Servlet API).

**Особенности:**
- **Платформонезависимость:** работает на любой платформе с JVM.
- **Производительность:** в отличие от CGI (Common Gateway Interface), сервлет не создаёт новый процесс на каждый запрос. Один экземпляр сервлета обслуживает множество запросов в разных потоках.
- **Простота API:** набор интерфейсов и абстрактных классов для обработки запросов и формирования ответов.
- **Интеграция с Java-экосистемой:** доступ к JDBC, JPA, EJB, JMS и другим технологиям.
- **Поддержка сессий и состояния:** встроенные механизмы для отслеживания пользовательских сессий.
##### 5.8-2 ==Servlet container, Distributed servlet container==
**Контейнер сервлетов (Servlet Container, веб-контейнер)** — компонент веб-сервера, который управляет жизненным циклом сервлетов, преобразует HTTP-запросы в объекты Java и передаёт их сервлетам, а также преобразует ответы сервлетов в HTTP-ответы, отправляемые клиенту.

**Функции контейнера:**
- Загрузка и инициализация сервлетов.
- Маршрутизация запросов к нужному сервлету (servlet mapping).
- Управление многопоточностью (каждый запрос в отдельном потоке).
- Поддержка сессий, фильтров, слушателей.
- Обеспечение безопасности (аутентификация, авторизация).

**Примеры контейнеров:** Apache Tomcat, Jetty, WildFly (JBoss), GlassFish.

**Распределённый контейнер сервлетов (Distributed Servlet Container):**  
Позволяет развернуть веб-приложение на нескольких JVM, возможно, на разных физических серверах. Контейнеры могут быть объединены в кластер. Особенности:
- **Репликация сессий:** данные сессии пользователя копируются между узлами кластера, чтобы отказ одного сервера не приводил к потере сессии.
- **Балансировка нагрузки:** запросы распределяются между узлами.
- **Общий контекст:** приложение ведёт себя как единое целое, несмотря на распределённость.
- Требуется настройка в дескрипторе (`<distributable/>` в web.xml).
##### 5.8-3 ==Servlet context и Servlet mapping==
**ServletContext:**  
Интерфейс, представляющий контекст веб-приложения. По одному экземпляру ServletContext на всё приложение (общий для всех сервлетов). Предоставляет:
- Доступ к параметрам инициализации приложения (из web.xml).
- Возможность обмена данными между сервлетами (атрибуты контекста через `setAttribute`/`getAttribute`).
- Доступ к файловым ресурсам приложения (`getResource`, `getResourceAsStream`).
- Информацию о сервере и версии API.
- Возможность программной регистрации сервлетов и фильтров (в Servlet 3.0+).

**Servlet Mapping:**  
Механизм связывания URL-шаблонов с конкретными сервлетами. Определяет, какой сервлет будет обрабатывать запрос по данному URL. Настраивается в web.xml или через аннотацию `@WebServlet`. Примеры шаблонов:
- Точное совпадение: `/login` → сервлет LoginServlet.
- Префиксный путь: `/api/*` → все URL, начинающиеся с /api/.
- По расширению: `*.html` → все запросы к .html-файлам.
##### 5.8-4 ==Жизненный цикл сервлетов==
Контейнер управляет жизненным циклом сервлета, вызывая его методы в строго определённом порядке:
1. **Загрузка и инстанцирование:** при старте приложения или первом запросе контейнер загружает класс сервлета и создаёт его единственный экземпляр (синглтон).
2. **init(ServletConfig config):** вызывается однократно после создания. Сервлет инициализируется, получает конфигурацию. Здесь открываются ресурсы (соединения с БД). Без успешного init сервлет не обслуживает запросы.
3. **service(ServletRequest req, ServletResponse res):** вызывается при каждом запросе. Определяет тип запроса и направляет его в соответствующий метод (doGet, doPost для HttpServlet). Этот метод многопоточный — может вызываться одновременно для разных запросов.
4. **destroy():** вызывается однократно при выгрузке сервлета (остановка приложения или перезагрузка). Здесь освобождаются ресурсы, закрываются соединения.

После destroy сервлет становится кандидатом на сборку мусора.
##### 5.8-5 ==Объекты запроса (HttpServletRequest) и отклика (HttpServletResponse)==
**HttpServletRequest:**  
Инкапсулирует HTTP-запрос, поступивший от клиента. Основные методы:
- `getParameter(String name)` — значение параметра запроса (из URL или тела POST).
- `getHeader(String name)` — значение HTTP-заголовка.
- `getSession()` — получение или создание сессии.
- `getCookies()` — массив cookie.
- `getInputStream()` / `getReader()` — тело запроса (для POST/PUT).
- `getRequestDispatcher(String path)` — для перенаправления запроса другому ресурсу (forward/include).

**HttpServletResponse:**  
Инкапсулирует HTTP-ответ, формируемый сервером. Основные методы:
- `setContentType(String type)` — MIME-тип ответа (например, "text/html").
- `getWriter()` — PrintWriter для текстового ответа.
- `getOutputStream()` — ServletOutputStream для бинарного ответа.
- `setStatus(int code)` — HTTP-статус (200, 404, 500).
- `sendRedirect(String url)` — перенаправление клиента на другой URL.
- `addCookie(Cookie)` — установка cookie.
- `setHeader(String name, String value)` — установка заголовка ответа.
##### 5.8-6 ==HTTP Servlet. Пример сервлета==
**HttpServlet** — абстрактный класс, наследник GenericServlet, предназначенный для обработки HTTP-запросов. Разработчик переопределяет методы, соответствующие HTTP-методам:

- `doGet(HttpServletRequest req, HttpServletResponse resp)` — обработка GET.
- `doPost(HttpServletRequest req, HttpServletResponse resp)` — обработка POST.
- `doPut`, `doDelete`, `doHead` и т.д.
##### 5.8-7 ==Поддержка сессий (сервлеты)==
HTTP — протокол без сохранения состояния (stateless). Каждый запрос изолирован. Для сохранения информации о пользователе между запросами используются сессии.

**HttpSession:**  
Объект сессии создаётся при первом обращении `req.getSession()`. Сервер связывает сессию с клиентом через идентификатор (JSESSIONID), передаваемый в cookie или в URL. Методы:
- `setAttribute(String name, Object value)` — сохранить данные в сессии.
- `getAttribute(String name)` — получить данные из сессии.
- `removeAttribute(String name)` — удалить данные.
- `invalidate()` — завершить сессию (logout).
- `getId()` — идентификатор сессии.
- `setMaxInactiveInterval(int seconds)` — время неактивности до автоматического завершения.
##### 5.8-8 ==Дескриптор развёртывания web.xml==
**web.xml** — XML-файл (дескриптор развертывания) описывающий конфигурацию веб-приложения. Файл строится на базе XML. Корневым элементом всегда выступает `<web-app>`, внутри которого располагаются все остальные конфигурационные теги.

**Основные секции:**
- `<servlet>` — объявление сервлета: имя, класс, параметры инициализации.
- `<servlet-mapping>` — связь сервлета с URL-шаблоном.
- `<filter>` и `<filter-mapping>` — объявление и маппинг фильтров.
- `<listener>` — слушатели событий (старт/стоп приложения, создание сессии).
- `<context-param>` — параметры контекста, доступные всем сервлетам.
- `<session-config>` — настройки сессий (таймаут).
- `<welcome-file-list>` — файлы по умолчанию (index.html).
- `<error-page>` — кастомные страницы ошибок.
- `<distributable/>` — пометка приложения как распределённого.
##### 5.8-9 ==Фильтрация (сервлеты)==
**Фильтр (Filter)** — компонент, перехватывающий запросы до того, как они достигнут сервлета, и/или ответы после обработки сервлетом. Интерфейс `javax.servlet.Filter` (Jakarta Servlet API).

**Методы:**
- `init(FilterConfig)` — инициализация фильтра.
- `doFilter(ServletRequest, ServletResponse, FilterChain)` — обработка запроса. Вызвав `chain.doFilter(req, res)`, фильтр передаёт управление дальше по цепочке.
- `destroy()` — освобождение ресурсов.

**Применение:**
- Аутентификация и авторизация.
- Логирование запросов.
- Сжатие ответа (GZip).
- Установка кодировки.
- Защита от XSS, CSRF.
- Трансформация запроса/ответа.
### 5.9 JSP. Принцип работы, обработка ошибок, виды JSP, элементы JSP-страницы. Пример
SP (Java Server Pages) — технология создания динамических веб-страниц на Java. JSP-страница — текстовый файл с HTML и вставками Java-кода. Принцип работы: при первом запросе контейнер преобразует JSP в сервлет (фаза трансляции), компилирует его и загружает; при последующих запросах вызывается метод service сервлета (фаза выполнения). Обработка ошибок: директива `page errorPage` указывает страницу для обработки исключений; на целевой странице `isErrorPage="true"`. Элементы JSP: директивы, скриптовые элементы (скриптлеты, выражения, объявления), действия, EL-выражения.
##### 5.9-1 ==Принцип работы JSP==
JSP-страница — это файл с расширением `.jsp`, содержащий HTML-разметку и JSP-теги. Контейнер сервлетов обрабатывает её в два этапа.

**Фаза трансляции (translation phase):**  
Происходит при первом обращении к JSP-странице (или при её изменении). Контейнер преобразует JSP в исходный код сервлета: HTML-код оборачивается в операторы `out.write()`, JSP-теги преобразуются в Java-код. Полученный сервлет компилируется в `.class`-файл и загружается.

**Фаза выполнения (execution phase):**  
При последующих запросах контейнер вызывает метод `_jspService()` скомпилированного сервлета, передавая ему объекты запроса и ответа. Сервлет формирует HTML-ответ, который отправляется клиенту.

**Жизненный цикл JSP-страницы (аналогичен сервлету):**
1. Трансляция и компиляция.
2. Инициализация (`jspInit()`).
3. Обслуживание запросов (`_jspService()`).
4. Уничтожение (`jspDestroy()`).
##### 5.9-2 ==Обработка ошибок в JSP==
Обработка исключений в JSP настраивается двумя способами.

**Способ 1. Директива page с атрибутом errorPage:**  
На странице, где может возникнуть ошибка, указывается:
```
<%@ page errorPage="error.jsp" %>
```
При возникновении неперехваченного исключения контейнер перенаправляет запрос на указанную страницу.

**Способ 2. Страница ошибки с isErrorPage="true":**  
На целевой странице указывается:
```
<%@ page isErrorPage="true" %>
```
Это даёт доступ к неявному объекту `exception` (тип Throwable), содержащему информацию об ошибке. Можно вывести сообщение: `<%= exception.getMessage() %>`.
##### 5.9-3 ==Виды JSP и элементы JSP-страницы==
JSP-страница может содержать следующие виды элементов.

**1. Директивы (directives):** `<%@ директива атрибуты %>`  
Управляют структурой сервлета, генерируемого из JSP. Виды:
- `page` — настройки страницы (импорты, кодировка, сессия, буфер, errorPage).
- `include` — статическая вставка содержимого другого файла на этапе трансляции.
- `taglib` — подключение библиотек пользовательских тегов.

**2. Скриптовые элементы:**
- **Скриптлеты (scriptlets):** `<% код %>` — произвольный Java-код, помещаемый в метод `_jspService()`.
- **Выражения (expressions):** `<%= выражение %>` — вычисляется и выводится в ответ (без `;`).
- **Объявления (declarations):** `<%! объявление %>` — поля и методы класса сервлета (не метода `_jspService()`).

**3. Действия (actions):** `<jsp:имяДействия ... />`  
Стандартные теги для выполнения операций: `include`, `forward`, `useBean`, `setProperty`, `getProperty`.
**4. EL-выражения (Expression Language):** `${выражение}`  
Упрощённый доступ к данным без Java-кода (рассматривается в вопросе 5.11).
**5. Пользовательские теги (custom tags) и JSTL:**  
Библиотеки тегов, заменяющие скриптлеты (рассматриваются в вопросе 5.11).
##### 5.9-4 ==Пример JSP-страницы==
```
<%@ page language="java" contentType="text/html; charset=UTF-8"
    pageEncoding="UTF-8" errorPage="error.jsp" %>
<%@ page import="java.util.Date" %>
<!DOCTYPE html>
<html>
<head><title>Пример JSP</title></head>
<body>
    <h1>Текущее время: <%= new Date() %></h1>
    <%
        String name = request.getParameter("name");
        if (name != null && !name.isEmpty()) {
            out.println("<p>Привет, " + name + "!</p>");
        } else {
            out.println("<p>Введите имя в URL (?name=Имя)</p>");
        }
    %>
</body>
</html>
```
Этот пример содержит: директивы page и import, выражение, скриптлет, работу с неявными объектами request и out.
### 5.10 Директива include. Стандартные объекты. Теги <jsp:…>. Теги <jsp:include>. Теги <jsp:forward>. Взаимодействие с HTML-формами. Примеры.
Директива `<%@ include %>` выполняет статическую вставку содержимого файла на этапе трансляции JSP в сервлет. Стандартные неявные объекты JSP: request, response, out, session, application, config, pageContext, page, exception. Теги действий `<jsp:...>` выполняют операции во время выполнения запроса: `<jsp:include>` — динамическая вставка результата другого ресурса, `<jsp:forward>` — перенаправление запроса другому ресурсу без возврата, `<jsp:useBean>`, `<jsp:setProperty>`, `<jsp:getProperty>` — работа с JavaBeans. Взаимодействие с HTML-формами: параметры извлекаются через `request.getParameter()`, результаты могут быть записаны в JavaBean и отображены на странице.
##### 5.10-1 ==Директива include (JSP (JavaServer Pages))==
Директива `<%@ include %>` выполняет **статическую вставку** содержимого другого файла в текущую JSP-страницу. Вставка происходит на этапе трансляции JSP в сервлет, до компиляции.

**Синтаксис:**
```
<%@ include file="путь_к_файлу" %>
```
**Как работает:**  
	Контейнер берёт содержимое указанного файла и вставляет его непосредственно в исходный код генерируемого сервлета вместо директивы. После этого получившийся единый файл компилируется. Изменения в подключаемом файле не будут замечены до перекомпиляции основной страницы.

**Типичное применение:**  
Подключение общих частей страниц — заголовков, подвалов, меню навигации:
```
<%@ include file="header.jsp" %>
<!-- основное содержимое -->
<%@ include file="footer.jsp" %>
```
**Отличие от `<jsp:include>`:**  
	Директива `include` — статическая, на этапе трансляции, один скомпилированный сервлет.  
	Тег `<jsp:include>` — динамический, на этапе выполнения, отдельный запрос к ресурсу.
##### 5.10-2 ==Стандартные объекты (неявные объекты JSP)==
JSP предоставляет девять предопределённых (неявных) объектов, доступных в скриптлетах и выражениях без явного объявления.
- **request** (HttpServletRequest) — запрос клиента, содержит параметры, заголовки, cookies.
- **response** (HttpServletResponse) — ответ клиенту, позволяет устанавливать заголовки, перенаправлять.
- **out** (JspWriter) — поток вывода, аналог PrintWriter в сервлете. Буферизирован.
- **session** (HttpSession) — сессия пользователя. Доступна, если в директиве page `session="true"`.
- **application** (ServletContext) — контекст веб-приложения, общий для всех сервлетов и JSP.
- **config** (ServletConfig) — конфигурация сервлета, параметры инициализации.
- **pageContext** (PageContext) — контекст текущей JSP-страницы. Предоставляет доступ ко всем остальным неявным объектам, атрибутам разных областей видимости.
- **page** (Object) — ссылка на сам сервлет (this).
- **exception** (Throwable) — исключение, переданное на страницу ошибки. Доступен только если `isErrorPage="true"`.
##### 5.10-3 ==Теги `<jsp:include>`(JavaServer Pages)==
Тег `<jsp:include>` выполняет **динамическую вставку** содержимого другого ресурса. В отличие от директивы `include`, вставка происходит во время выполнения запроса.

**Синтаксис:**
```
<jsp:include page="URL_ресурса" />
```
или с передачей параметров:
```
<jsp:include page="URL_ресурса">
    <jsp:param name="имя" value="значение" />
</jsp:include>
```

**Как работает:**  
При обработке запроса контейнер выполняет отдельный запрос к указанному ресурсу (JSP, сервлет, HTML), получает результат и вставляет его в текущий вывод. Ресурс может быть динамическим — его содержимое может меняться между запросами.
##### 5.10-4 ==Теги `<jsp:forward>`(JavaServer Pages)==
Тег `<jsp:forward>` передаёт обработку запроса другому ресурсу. Текущая JSP-страница прекращает вывод, и управление полностью переходит к указанному ресурсу. Клиент не видит смены URL — это серверное перенаправление.

**Синтаксис:**
```
<jsp:forward page="URL_ресурса" />
```
или с параметрами:
```
<jsp:forward page="URL_ресурса">
    <jsp:param name="имя" value="значение" />
</jsp:forward>
```

**Как работает:**  
`Буфер вывода очищается (если не был сброшен), запрос передаётся другому ресурсу. После `<jsp:forward>` код текущей страницы не выполняется.
##### 5.10-5 ==Теги для работы с JavaBeans==
JSP предоставляет три стандартных действия для работы с JavaBeans — классами, удовлетворяющими соглашению (конструктор без параметров, геттеры/сеттеры).

**`<jsp:useBean>`** — объявляет или находит экземпляр бина:
```
<jsp:useBean id="user" class="com.example.User" scope="session" />
```

Если бин с таким id существует в указанной области — используется существующий, иначе создаётся новый.

**`<jsp:setProperty>`** — устанавливает свойства бина, в том числе из параметров запроса:
```
<jsp:setProperty name="user" property="username" value="Иван" />
<jsp:setProperty name="user" property="*" /> <!-- автоматическое соответствие -->
```

При `property="*"` контейнер сопоставляет параметры запроса с одноимёнными свойствами бина.

**`<jsp:getProperty>`** — читает свойство бина и выводит его:
```
<jsp:getProperty name="user" property="username" />
```
##### 5.10-6 ==Взаимодействие JSP с HTML-формами. Примеры==
HTML-форма отправляет данные на сервер методом GET или POST. JSP получает параметры через неявный объект `request`.

**Стандартный сценарий обработки формы:**

**Форма (HTML/JSP):**
```
<form action="process.jsp" method="post">
    Имя: <input type="text" name="username" />
    Email: <input type="email" name="email" />
    <input type="submit" value="Отправить" />
</form>
```

**Обработчик (process.jsp):**
```
<%@ page contentType="text/html; charset=UTF-8" %>
<%
    String username = request.getParameter("username");
    String email = request.getParameter("email");
    if (username == null || username.isEmpty()) {
        out.println("<p style='color:red'>Имя обязательно!</p>");
    } else {
        out.println("<p>Привет, " + username + "!</p>");
        out.println("<p>Ваш email: " + email + "</p>");
    }
%>
```
#### 5.11 Недостатки раннего JSP. Основные идеи смены парадигмы. Expression Language. Виды выражений EL. Пример. Литералы, операторы, объекты доступа к другим объектам. Custom tags. Библиотеки тегов. JSP Standard Tag Library. Библиотеки JSTL.
Ранний JSP (Model 1) страдал от смешения бизнес-логики и представления: громоздкие скриптлеты делали страницы нечитаемыми и неподдерживаемыми. 
Переход к MVC Model 2 разделил приложение на контроллер (сервлет), модель (JavaBean/POJO) и представление (JSP без Java-кода). Expression Language (EL) заменил скриптлеты лаконичными `${...}`-выражениями. 

EL поддерживает литералы, операторы, неявные объекты для доступа к параметрам, заголовкам, атрибутам сессии. 
Custom tags и JSTL (Core, Formatting, Functions) полностью вытесняют Java-код из JSP, заменяя его тегами для циклов, условий, форматирования.
##### 5.11-1 ==Недостатки раннего JSP==
Ранние JSP-приложения использовали архитектуру **Model 1**: JSP-страница одновременно принимала запрос, обрабатывала бизнес-логику и формировала ответ.

**Основные недостатки:**
- **Смешение логики и представления:** скриптлеты содержали код работы с БД, проверки условий, циклы. Дизайнер не мог редактировать такую страницу без знания Java.
- **Дублирование кода:** одинаковые операции (подключение к БД, проверка прав) копировались с одной JSP на другую.
- **Сложность отладки и тестирования:** ошибки в скриптлетах выдавали неинформативные stack trace прямо в браузер.
- **Низкая сопровождаемость:** изменение бизнес-логики требовало правки десятков JSP-файлов.
- **Раздувание страниц:** файлы превращались в простыни из HTML и Java-кода, понять структуру было почти невозможно.
##### 5.11-2 ==Основные идеи смены парадигмы (MVC Model 2)==
Решением стала архитектура **MVC Model 2**, которая разделяет приложение на три компонента.

**Model (Модель):**  
JavaBeans, POJO, сервисные классы. Содержат данные и бизнес-логику. Ничего не знают о том, как данные будут отображаться. Модель не зависит от контроллера и представления.

**View (Представление):**  
JSP-страницы без Java-кода. Только отображение данных, полученных от контроллера. Используют EL и JSTL вместо скриптлетов. Могут редактироваться веб-дизайнерами.

**Controller (Контроллер):**  
Сервлет (один или несколько). Принимает запрос, извлекает параметры, вызывает методы модели, помещает результат в атрибуты request/session, передаёт управление на JSP для отображения.

**Поток обработки запроса в Model 2:**
1. Запрос поступает на сервлет-контроллер.
2. Контроллер извлекает параметры, вызывает бизнес-логику (модель).
3. Результат сохраняется в request.setAttribute().
4. Контроллер делает forward на JSP.
5. JSP извлекает данные через EL и отображает их.

**Преимущества Model 2:**
- Чёткое разделение ответственности.
- Логика тестируется независимо от представления.
- JSP-страницы чисты, содержат только HTML и минимум кода.
- Один и тот же контроллер может обслуживать разные представления.
##### 5.11-3 ==Expression Language (EL): назначение и синтаксис==
**Expression Language (EL)** — язык выражений, встроенный в JSP (начиная с JSP 2.0). Предоставляет лаконичный синтаксис для доступа к данным без использования скриптлетов.

**Синтаксис:**  
`${выражение}` или `#{выражение}` (для отложенных выражений в JSF).

**Что заменяет EL:**  
Вместо громоздкого:
```
<%= ((User) session.getAttribute("user")).getName() %>
```
Пишется:
```
${sessionScope.user.name}
```

**Основные возможности:**
- Доступ к свойствам бинов через точку или скобки: `${bean.property}`, `${bean["property"]}`.
- Доступ к элементам массивов, списков, мап: `${list[0]}`, `${map["key"]}`.
- Автоматическое приведение типов.
- Устойчивость к null: если объект или свойство null, EL возвращает пустую строку, а не NullPointerException.
- Поддержка арифметических, логических операций и операций сравнения.
- Встроенные неявные объекты для доступа к параметрам, заголовкам, кукам, областям видимости
##### 5.11-4 ==Виды выражений EL. Пример==
**Немедленное выполнение (immediate evaluation):** `${выражение}`  
Вычисляется сразу при обработке страницы. Стандартный способ.

**Отложенное выполнение (deferred evaluation):** `#{выражение}`  
Вычисляется позже (используется в JSF, JavaServer Faces).

**Основные виды выражений EL:**

**Доступ к свойству бина:**
```
${user.name}           <!-- эквивалент user.getName() -->
${user["name"]}        <!-- то же самое, скобочная нотация -->
```

**Доступ к элементам коллекций:**
```
${products[0]}         <!-- первый элемент списка -->
${productMap["key"]}   <!-- значение по ключу в Map -->
```

**Доступ к областям видимости:**
```
${requestScope.user}   <!-- атрибут запроса -->
${sessionScope.cart}   <!-- атрибут сессии -->
${applicationScope.config} <!-- атрибут контекста -->
```
##### 5.11-5 ==Литералы, операторы и объекты доступа в EL==
**Литералы (константы):**
- Числовые: `${42}`, `${3.14}`
- Строковые: `${"hello"}`, `${'world'}`
- Булевы: `${true}`, `${false}`
- null: `${null}`

**Операторы EL:**
Арифметические: `+`, `-`, `*`, `/`, `%`, `div`, `mod`.  
	Пример: `${price * quantity}`

Сравнения: `==`, `!=`, `<`, `>`, `<=`, `>=`, `eq`, `ne`, `lt`, `gt`, `le`, `ge`.  
	Пример: `${user.age ge 18}` (greater or equal)

Логические: `&&`, `||`, `!`, `and`, `or`, `not`.  
	Пример: `${user.admin and user.active}`

Проверка на пустоту: `empty`. Возвращает true для null, пустой строки, пустой коллекции.  
	Пример: `${empty cart.items}`

Тернарный оператор: `условие ? значение1 : значение2`.  
	Пример: `${user.vip ? "VIP-клиент" : "Обычный клиент"}`

**Неявные объекты EL:**
- `param` — параметр запроса: `${param.username}`
- `paramValues` — массив параметров: `${paramValues.colors[0]}`
- `header` — HTTP-заголовок: `${header["User-Agent"]}`
- `cookie` — значение куки: `${cookie.JSESSIONID.value}`
- `pageScope`, `requestScope`, `sessionScope`, `applicationScope` — атрибуты областей видимости.
- `pageContext` — доступ к сервлетному PageContext.
##### 5.11-6 ==Custom tags: назначение и принцип==
**Пользовательские теги (custom tags)** — механизм расширения JSP собственными XML-тегами, за которыми стоит Java-код. Позволяют упаковать повторяющуюся логику в переиспользуемый компонент, полностью скрывая Java-код.

**Составные части custom tag:**
- **Класс-обработчик тега (tag handler):** Java-класс, реализующий интерфейс `Tag` или наследующий `SimpleTagSupport`. Содержит логику тега в методе `doTag()`.
- **TLD-файл (Tag Library Descriptor):** XML-файл, описывающий тег: имя, класс-обработчик, атрибуты, тело тега.
- **Подключение в JSP:** директива `<%@ taglib prefix="my" uri="..." %>`. Использование: `<my:customTag attr="value" />`.

**Пример custom tag для форматирования даты:**  
Класс-обработчик задаёт формат, тег в JSP вызывает `<my:formatDate date="${order.date}" />`. Дизайнеру не нужно знать Java.

Custom tags были популярны до появления JSTL. Сейчас в большинстве случаев достаточно JSTL, но custom tags по-прежнему актуальны для узкоспециализированных библиотек.
##### 5.11-7 ==Библиотеки тегов. JSP Standard Tag Library (JSTL)==
**JSTL (JSP Standard Tag Library)** — стандартная библиотека тегов, предоставляющая готовые решения для типовых задач. Полностью заменяет скриптлеты. Входит в Jakarta Standard Tag Library.

**Основные библиотеки JSTL:**

| Библиотека | Префикс | Назначение                             |
| ---------- | ------- | -------------------------------------- |
| Core       | c       | Условные операторы, циклы, вывод       |
| Formatting | fmt     | Форматирование чисел, дат, локализация |
| Functions  | fn      | Строковые функции                      |
| SQL        | sql     | Работа с БД (не рекомендуется в MVC)   |
| XML        | x       | Обработка XML                          |

**Основные теги Core:**
- `<c:out value="${expr}" />` — безопасный вывод (экранирует HTML).
- `<c:set var="name" value="${expr}" scope="session" />` — установка переменной.
- `<c:if test="${condition}"> ... </c:if>` — условный блок.
- `<c:choose>` / `<c:when test="...">` / `<c:otherwise>` — множественный выбор (switch).
- `<c:forEach var="item" items="${list}"> ... </c:forEach>` — цикл по коллекции.
- `<c:forEach var="i" begin="1" end="10"> ... </c:forEach>` — цикл со счётчиком.

**Основные теги Formatting:**
- `<fmt:formatDate value="${date}" pattern="dd.MM.yyyy" />`
- `<fmt:formatNumber value="${price}" type="currency" />`
- `<fmt:setLocale value="ru_RU" />`
##### 5.11-8 ==Примеры использования JSTL (JavaServer Pages Standard Tag Library)==
**Пример 1: Вывод списка товаров без скриптлетов:**

```
<%@ taglib prefix="c" uri="jakarta.tags.core" %>
<table>
    <tr><th>Название</th><th>Цена</th></tr>
    <c:forEach var="product" items="${products}">
        <tr>
            <td><c:out value="${product.name}" /></td>
            <td><c:out value="${product.price}" /> руб.</td>
        </tr>
    </c:forEach>
</table>
```

JavaScript — интерпретируемый язык программирования, выполняемый в браузере. 
Области применения: клиентская интерактивность, валидация форм, анимации, AJAX, серверная разработка (Node.js). 

Размещается внутри HTML-тега `<script>` или во внешнем .js-файле. 
Браузер выполняет JS последовательно, блокируя парсинг HTML при встрече `<script>`. 
Комментарии: `//` и `/* */`. 
Переменные: `var` (устаревший), `let` (блочная видимость), `const` (константа). 
Типы: number, string, boolean, undefined, null, object, symbol, bigint. 
Управление выполнением: `if/else`, `switch`, `for`, `while`, `do...while`, `break`, `continue`.
##### 5.9-1 ==JavaScript: области применения==
**JavaScript** — мультипарадигменный язык программирования, изначально созданный для оживления веб-страниц. Стандартизирован как ECMAScript.

**Области применения:**
- **Клиентский веб (основная):** динамическое изменение содержимого страниц (DOM), обработка событий (клики, ввод), анимации, валидация форм до отправки, создание SPA (Single Page Application) на React, Angular, Vue.
- **Серверная разработка:** Node.js позволяет выполнять JavaScript на сервере, обрабатывать запросы, работать с БД, файловой системой.
- **Мобильная разработка:** React Native, NativeScript — создание нативных мобильных приложений на JavaScript.
- **Десктопная разработка:** Electron — создание настольных приложений (VS Code, Slack, Discord).
- **Расширения браузера:** Chrome Extensions, Firefox Add-ons.
- **Игры:** браузерные игры с Canvas, WebGL.
- **IoT и робототехника:** Johnny-Five, Espruino.
##### 5.9-2 ==Размещение JavaScript в HTML-документе==
**Внутренний скрипт** — код пишется прямо внутри HTML в теге `<script>`:
```
<script>
    console.log("Привет");
</script>
```

**Внешний скрипт** — код в отдельном файле `.js`, подключается с атрибутом `src`:
```
<script src="script.js"></script>
```

**Атрибуты тега `<script>`:**
- `async` — скрипт загружается асинхронно и выполняется сразу после загрузки, не дожидаясь парсинга HTML.
- `defer` — скрипт загружается асинхронно, но выполняется только после полного парсинга HTML, в порядке их подключения.

**Рекомендации по размещению:** скрипты лучше подключать в конце `<body>` или с атрибутом `defer`, чтобы не блокировать отображение страницы.
##### 5.9-3 ==Обработка JavaScript браузером==
Браузер обрабатывает HTML последовательно, сверху вниз. Когда он встречает тег `<script>`:
1. Парсинг HTML приостанавливается.
2. Загружается (если внешний) и выполняется JavaScript.
3. После выполнения скрипта парсинг HTML продолжается.

Именно поэтому скрипты без `defer`/`async`, размещённые в `<head>`, не видят элементы DOM ниже по коду — они ещё не распарсены.

Браузерный движок JavaScript (V8 в Chrome, SpiderMonkey в Firefox, JavaScriptCore в Safari) компилирует JS в машинный код «на лету» (JIT-компиляция) и выполняет его.

**Комментарии:**
```
// однострочный комментарий

```

**Литералы** — фиксированные значения в коде:

- `null` — отсутствие значения (явно)
- `undefined` — значение не присвоено
##### 5.9-5 ==Управление порядком выполнения (JavaScript)==
**Условные операторы:**
```
if (a > b) {
    // ...
} else if (a < b) {
    // ...
} else {
    // ...
}
switch (x) {
    case 1: ...; break;
    case 2: ...; break;
    default: ...;
}
```

**Циклы:**
```
// цикл с известным числом итераций
for (let i = 0; i < 10; i++) { }
// перебор свойств объекта (ключи)
for (let key in obj) { }
// перебор итерируемых (массивы, строки, Map, Set) — значения

```
**Управление циклом:**
- `break` — выход из цикла.
- `continue` — переход к следующей итерации.

**Обработка ошибок:**
```
try {
    throw new Error("Сообщение");
} catch (e) {
    console.error(e.message);
} finally {
    // выполняется всегда
}
```
#### 5.13 JavaScript. Функции. Объекты. Функции-конструкторы. Встроенные объекты. Наследование. Выбрасывание и обработка исключений. Пользовательские массивы. Методы пользовательских массивов
В JavaScript функции — объекты первого класса: их можно присваивать, передавать, возвращать. 
Объекты — наборы пар ключ-значение. Функции-конструкторы (с new) создают объекты с общим прототипом. Встроенные объекты: 
Object, Array, String, Number, Date, Math, JSON, RegExp. 
Наследование — прототипное: объект ссылается на прототип, от которого наследует свойства. 
Исключения: throw new Error(), try/catch/finally. Массивы — объекты с числовыми индексами и свойством length. Методы массивов: push, pop, shift, unshift, slice, splice, map, filter, reduce, forEach.
##### 5.10-1 ==Функции (JavaScript)==
Функция — блок кода, который можно вызывать многократно. В JavaScript функции — объекты первого класса.

**Способы объявления:**
- Function Declaration: `function sum(a, b) { return a + b; }` — доступна до объявления (всплытие).
- Function Expression: `const sum = function(a, b) { return a + b; };` — доступна после присваивания.
- Стрелочная функция (ES6): `const sum = (a, b) => a + b;` — короче, не имеет своего this.

**Параметры:**
- Можно передать любое количество аргументов, независимо от объявленных параметров.
- Лишние игнорируются, недостающие — undefined.
- Параметры по умолчанию: `function greet(name = "Гость")`.
- Остаточные параметры: `function f(...args)` — собирает все аргументы в массив.
##### 5.10-2 ==Объекты и функции-конструкторы (JavaScript)==
**Объект** — неупорядоченная коллекция свойств (ключ-значение). Ключ — строка или Symbol, значение — любой тип.

**Создание объекта:**
- Литерал: `const obj = { name: "Иван", age: 25 };`
- Конструктор Object: `const obj = new Object();`
- Object.create(proto): создание с указанным прототипом.

**Доступ к свойствам:**
- Точечная нотация: `obj.name`
- Квадратные скобки: `obj["name"]`

**Функции-конструкторы** — функции, предназначенные для создания множества однотипных объектов. Вызываются с `new`. Соглашение: имя с большой буквы.

```
function Person(name, age) {
    this.name = name;
    this.age = age;
}
const person = new Person("Иван", 25);
```

При вызове с `new`:
1. Создаётся пустой объект.
2. `this` внутри функции ссылается на этот объект.
3. Объект связывается с прототипом (Person.prototype).
4. Функция возвращает объект (если явно не возвращён другой).
##### 5.10-3 ==Встроенные объекты (JavaScript)==
**Object:** базовый объект, предок всех объектов. Методы: keys(), values(), entries(), assign(), freeze(), seal().
**Array:** упорядоченные списки. Методы: push/pop, shift/unshift, slice, splice, map, filter, reduce, find, sort.
**String:** строки. Методы: length, charAt(), indexOf(), slice(), substring(), replace(), split(), toLowerCase(), toUpperCase(), trim().
**Number:** числа. Свойства: MAX_VALUE, MIN_VALUE, NaN, POSITIVE_INFINITY. Методы: toFixed(), toString(), isNaN() (глобальная).
**Math:** математические константы и функции (не конструктор). Math.PI, Math.random(), Math.round(), Math.floor(), Math.ceil(), Math.max(), Math.min().
**Date:** работа с датами. new Date(), getFullYear(), getMonth(), getDay(), getTime().
**JSON:** сериализация/десериализация. JSON.stringify(obj), JSON.parse(str).
**RegExp:** регулярные выражения. /pattern/flags. Методы: test(), exec().
##### 5.10-4 ==Наследование (JavaScript)==
JavaScript использует **прототипное наследование**. Каждый объект имеет внутреннюю ссылку `[[Prototype]]` (доступ через `__proto__` или `Object.getPrototypeOf()`).

**Принцип:** если свойство не найдено в самом объекте, поиск идёт вверх по цепочке прототипов до Object.prototype и далее до null.

**Установка прототипа:**
- При создании через конструктор: `Person.prototype` становится прототипом экземпляров.
- Object.create(proto).
- class (ES6) — синтаксический сахар над прототипным наследованием.

**Установка через конструктор:**
```
function Animal(name) { this.name = name; }
Animal.prototype.say = function() { console.log(this.name); };
function Dog(name, breed) {
    Animal.call(this, name);
    this.breed = breed;
}
Dog.prototype = Object.create(Animal.prototype);
Dog.prototype.constructor = Dog;
```
##### 5.10-5 ==Выбрасывание и обработка исключений (JavaScript)==
**Выбрасывание исключения:**
```
throw new Error("Сообщение");
throw "Строка-ошибка"; // можно, но плохая практика
throw { message: "Ошибка", code: 404 };
```
**Типы встроенных ошибок:**
- Error — базовый.
- TypeError — неверный тип.
- ReferenceError — обращение к несуществующей переменной.
- SyntaxError — ошибка синтаксиса.
- RangeError — выход за допустимый диапазон.

**Обработка:**
```
try {
    // потенциально опасный код
} catch (e) {
    console.error(e.message);
} finally {
    // выполняется в любом случае (очистка ресурсов)
}
```
Блок `catch` можно опустить, если есть `finally`. Начиная с ES2019 можно писать `try { } catch { }` без указания переменной ошибки, если она не нужна.
##### 5.10-6 ==Пользовательские массивы и их методы (JavaScript)==
**Массив в JavaScript** — это объект с числовыми индексами и автоматически обновляемым свойством `length`. Не фиксированного размера, может содержать элементы любых типов.

**Создание:**
```
const arr = [1, 2, 3];
const arr2 = new Array(10); // массив длины 10
```

**Основные методы массивов:**

**Мутирующие (изменяют исходный массив):**
- `push(...items)` — добавить в конец.
- `pop()` — удалить с конца.
- `shift()` — удалить с начала.
- `unshift(...items)` — добавить в начало.
- `splice(start, deleteCount, ...items)` — удалить/вставить элементы.
- `sort(fn)` — сортировка на месте.
- `reverse()` — перевернуть.

**Немутирующие (возвращают новый массив):**
- `slice(start, end)` — копия части массива.
- `concat(...arrays)` — объединение массивов.
- `map(fn)` — преобразование каждого элемента.
- `filter(fn)` — отбор элементов по условию.
- `reduce(fn, initial)` — свёртка в одно значение.
#### 5.14 JavaScript. Виды объектов. BOM и DOM. Объект window. Создание и закрытие новых окон. Запуск новых потоков. Свойства window. Объект document. Прямая запись в документ. События объектов. Работа с формами. Программирование гиперссылок
В браузерном JavaScript все объекты делятся на: встроенные (Array, Date), объекты браузерной среды — BOM (Browser Object Model: window, navigator, location, history) и DOM (Document Object Model: document, элементы страницы). window — глобальный объект, представляет окно браузера. 

Методы: open() (новое окно), close(), setTimeout()/setInterval() (запуск отложенного/периодического кода). 
Свойства window: innerWidth/innerHeight, location, history, navigator. document — объект DOM-дерева. 
Прямая запись: document.write() (устаревший). 
События: onclick, onchange, onsubmit, addEventListener. 
Формы: доступ через document.forms, чтение значений элементов. 
Гиперссылки: программный переход через location.href, перехват клика.
##### 5.11-1 ==Виды объектов (JavaScript): BOM и DOM==
В браузерном окружении все объекты JavaScript делятся на три категории:

**Встроенные объекты языка:**  
Object, Array, String, Number, Date, Math, RegExp, Function, Error и др. Не зависят от среды выполнения, доступны везде.

**BOM (Browser Object Model):**  
Объекты, предоставляемые браузером для управления окном и взаимодействия с окружением, не связанные напрямую с содержимым страницы. Основные:
- **window** — глобальный объект, окно браузера.
- **navigator** — информация о браузере и ОС.
- **location** — URL текущей страницы.
- **history** — история посещений.
- **screen** — параметры экрана.
- **frames** — управление фреймами.

**DOM (Document Object Model):**  
Объектная модель HTML-документа. Представляет страницу как дерево узлов (элементы, атрибуты, текст). Основные:
- **document** — корневой узел, вход в DOM-дерево.
- Узлы-элементы (Element): `<div>`, `<p>`, `<a>`.
- Узлы-атрибуты (Attr): `id`, `class`, `href`.
- Текстовые узлы (Text).
##### 5.11-2 ==Объект window: создание и закрытие окон, свойства==
**window** — глобальный объект, представляющий вкладку или окно браузера. Все глобальные переменные и функции — его свойства и методы. `window.document` — то же, что `document`.

**Создание нового окна:**
```
const newWin = window.open("https://example.com", "_blank", "width=500,height=400");
```

Параметры: URL, имя окна (специальное или `_blank`), строка с параметрами окна (размеры, полосы прокрутки).

**Закрытие окна:**
```
newWin.close(); // закрыть созданное окно
window.close(); // закрыть текущее (работает только на окнах, открытых скриптом)
```

**Основные свойства window:**
- `innerWidth` / `innerHeight` — ширина/высота области содержимого (включая полосы прокрутки).
- `outerWidth` / `outerHeight` — ширина/высота всего окна браузера.
- `location` — объект с данными о текущем URL.
- `history` — объект для навигации (back(), forward(), go()).
- `navigator` — информация о браузере (userAgent, platform, language).
- `localStorage` / `sessionStorage` — хранилища данных.
- `console` — отладка.
##### 5.11-3 ==Запуск новых потоков (JavaScript)==
JavaScript — однопоточный язык. Настоящих потоков (threads) в браузере нет. Под «запуском новых потоков» понимается асинхронное выполнение кода.

**Основные механизмы:**
- **setTimeout(fn, delay):** выполнить функцию один раз через delay миллисекунд. Возвращает идентификатор таймера. `clearTimeout(id)` отменяет.
- **setInterval(fn, interval):** выполнять функцию периодически через interval. `clearInterval(id)` останавливает.
- **requestAnimationFrame(fn):** выполнить fn перед следующей перерисовкой кадра (для анимаций).
- **Promise и async/await:** асинхронные операции без блокировки основного потока.
- **Web Workers:** настоящая многопоточность — скрипт в отдельном потоке, общение через postMessage. Не имеет доступа к DOM.

**Очередь событий (Event Loop):**  
Синхронный код выполняется сразу. Асинхронные коллбэки (таймеры, события, промисы) попадают в очередь и выполняются, когда основной поток свободен. Это создаёт иллюзию параллельности без настоящих потоков.
##### 5.11-4 ==Объект document и прямая запись в документ==
**document** — корневой объект DOM, предоставляющий доступ к содержимому HTML-страницы.

**Основные методы доступа к элементам:**
- `document.getElementById("id")` — по идентификатору.
- `document.getElementsByClassName("class")` — по классу.
- `document.getElementsByTagName("div")` — по тегу.
- `document.querySelector(".class")` — первый по CSS-селектору.
- `document.querySelectorAll("div.class")` — все по CSS-селектору.

**Прямая запись в документ:**
```
document.write("<h1>Текст</h1>");
```

Выводит HTML в текущую позицию парсинга. Если вызван после полной загрузки страницы — перезаписывает весь документ. Считается устаревшим. Вместо него используют манипуляции с DOM:
```
const div = document.createElement("div");
div.textContent = "Привет";
document.body.appendChild(div);
```
##### 5.11-5 ==События объектов (JavaScript)==
События позволяют реагировать на действия пользователя и другие инциденты.

**Назначение обработчиков:**
- В HTML-атрибуте: `<button onclick="handler()">`.
- Через свойство DOM-элемента: `element.onclick = function() {}`.
- Через addEventListener (рекомендуемый):

```
element.addEventListener("click", function(event) {
    console.log(event.target); // элемент, на котором сработало событие
});
```

**Основные типы событий:**
- **Мыши:** click, dblclick, mousedown, mouseup, mousemove, mouseover, mouseout
- **Клавиатуры:** keydown, keyup, keypress.
- **Формы:** submit, change, focus, blur, input.
- **Документа:** DOMContentLoaded, load, beforeunload.
- **Сенсорные:** touchstart, touchend, touchmove.
##### 5.11-6 ==Работа с формами (JavaScript)==
Формы (`<form>`) содержат элементы ввода: `<input>`, `<select>`, `<textarea>`, `<button>`.

**Доступ к формам:**
- `document.forms` — коллекция всех форм на странице.
- `document.forms[0]` — первая форма.
- `document.forms.myForm` — форма с name="myForm".

**Доступ к элементам формы:**
- `form.elements` — коллекция элементов формы.
- `form.elements.myInput` — элемент с name="myInput".

**Чтение и установка значений:**
```
const val = form.elements.myInput.value;
form.elements.myCheckbox.checked; // для чекбоксов
form.elements.mySelect.selectedIndex; // для выпадающих списков
```

**Событие submit:**
```
form.addEventListener("submit", function(event) {
    event.preventDefault(); // отменить отправку для валидации
    // проверка данных
    if (valid) form.submit(); // программная отправка
});
```

**Валидация форм:** проверка на пустоту, формат email, длину пароля, равенство паролей — всё на JavaScript перед отправкой, с выводом сообщений об ошибках пользователю.
##### 5.11-7 ==Программирование гиперссылок (JavaScript)==
Гиперссылки (`<a href="...">`) можно контролировать из JavaScript.

**Программный переход:**
```
window.location.href = "https://example.com"; // переход с сохранением в истории
window.location.replace("https://example.com"); // переход без сохранения в истории
window.location.reload(); // перезагрузка страницы
```

**Перехват клика по ссылке:**
```
document.querySelectorAll("a").forEach(link => {
    link.addEventListener("click", function(event) {
        event.preventDefault(); // отменить переход
        const url = this.href;
        // какая-то логика: проверка, подтверждение, анимация
        window.location.href = url; // или программный переход
    });
});
```

Это используется для: подтверждения ухода со страницы, плавной навигации в SPA (Single Page Application), отправки аналитики при клике, проверки прав доступа перед переходом.
