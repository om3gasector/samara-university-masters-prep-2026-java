#### 4.1 Три направления развития платформы Java. Характерные особенности языка Java. Три принципа ООП. Пример. Достоинства и недостатки ООП. Классы и объекты. Свойства объектов. Пример.
**Java развивается в трех направлениях:**
Java SE (Standard Edition) — базовые API для настольных приложений, 
Java EE (Enterprise Edition) — серверные компоненты и веб-сервисы, 
Java ME (Micro Edition) — мобильные и встраиваемые устройства. 

Особенности языка: кроссплатформенность (JVM), объектно-ориентированность, строгая типизация, автоматическая сборка мусора, многопоточность, безопасность (байт-код верификация). 

**Три принципа ООП:** 
инкапсуляция — «внутреннее состояние объекта должно быть защищено от изменений изве и должно производиться только с помощью методов самого объекта», 

наследование — «принцип, который позволяет всем наследникам какого то класса наследовать его свойства», 

полиморфизм — «возможность обращаться со всеми наследниками объекта одинаковым образом, однако поведение разное»

Класс — шаблон, описывающий состояние и поведение. 
Объект — экземпляр класса. 
Свойства объектов: это набор значений всех полей объекта, например: состояние (поля), поведение (методы), идентичность (уникальность).
##### 4.1-1 ==Три направления развития платформы Java==
Java как платформа включает три редакции, каждая для своего сегмента разработки:

- **Java SE (Standard Edition):** содержит базовые API: коллекции, ввод-вывод, сеть, многопоточность, Swing/JavaFX для GUI. На ней строятся настольные приложения и она является фундаментом для других редакций.
- **Java EE (Enterprise Edition, сейчас Jakarta EE):** расширяет SE компонентами для серверной разработки: сервлеты, JSP, EJB, JPA, веб-сервисы (SOAP/REST). Предназначена для корпоративных распределенных систем.
- **Java ME (Micro Edition):** урезанная версия для устройств с ограниченными ресурсами: телефоны, датчики, встраиваемые системы. Включает профили (MIDP) под разные классы устройств.
##### 4.1-2 ==Характерные особенности языка Java==
- **Кроссплатформенность:** исходный код компилируется в байт-код (.class), исполняемый JVM. «Write once, run anywhere». JVM для каждой платформы своя.
- **Объектно-ориентированность:** всё является объектом, кроме примитивных типов (int, boolean и др.).
- **Строгая статическая типизация:** тип каждой переменной известен на этапе компиляции.
- **Автоматическая сборка мусора (Garbage Collector):** программист не управляет памятью вручную, JVM освобождает неиспользуемые объекты.
- **Многопоточность:** встроенные средства для создания и управления потоками (Thread, synchronized, java.util.concurrent).
- **Безопасность:** байт-код верифицируется перед выполнением, SecurityManager контролирует доступ к ресурсам.
##### 4.1-3 ==Три принципа ООП. Пример==
**Инкапсуляция** — сокрытие внутренних деталей реализации и предоставление доступа только через публичные методы. Поля класса private, доступ через get/set. Защищает данные от некорректного изменения.

**Наследование** — создание нового класса на основе существующего с автоматическим получением его членов. Позволяет повторно использовать код и строить иерархии. В Java — через ключевое слово extends.

**Полиморфизм** — способность объекта использовать метод, конкретная реализация которого определяется во время выполнения. Достигается через переопределение методов. Переменная типа родительского класса может ссылаться на объект дочернего, и вызов метода выполнит дочернюю реализацию.

**Пример (все три принципа):**  
**Наследование(берем готовое и достраиваем):**
	У нас есть класс `Животное`, которое умеет `есть()` и `спать()`. Мы хотим создать класс `Собака`. Вместо того чтобы переписывать «есть» и «спать» заново, мы говорим: `Собака extends Животное`. Собака автоматически получает эти два метода плюс мы добавляем новый — `лаять()`.
```
class Animal {
    void eat() { }
    void sleep() { }
}

class Dog extends Animal {
    void bark() { } // только у собаки
}
```
Объект `Dog` умеет есть, спать и лаять. Код `eat()` написан один раз в `Animal`.

**Инкапсуляция (скрываем данные, даём контролируемый доступ):**
	Представим класс `Счёт-В-Банке`. Поле `баланс` нельзя сделать public, потому что тогда любой код извне сможет написать `счёт.баланс = -100500;` и уйти в минус без проверки. Мы прячем баланс (private) и даём публичный метод `снятьДеньги(сумма)`, который внутри проверяет: хватает ли средств, не нулевая ли сумма. Внешний код вызывает только метод, не touching поле напрямую. Данные защищены.
``` 
class Account {
    private int balance; // никто снаружи не видит

    public void withdraw(int amount) { // единственный способ снять
        if (amount > 0 && amount <= balance) {
            balance = balance - amount;
        }
    }
}
```

**Полиморфизм(один интерфейс, разное поведение):**
	У родителя `Животное` есть метод `издатьЗвук()`. Каждое конкретное животное реализует его по-своему: собака лает, кошка мяукает. Мы пишем код, который работает с типом `Животное`, не зная, кто именно перед ним — собака или кошка. Во время выполнения Java сама определяет, чей метод вызвать.
```
Animal a = new Dog();
a.makeSound(); // "Гав" — вызовется метод Dog, хотя тип переменной Animal
a = new Cat();
a.makeSound(); // "Мяу" — та же переменная, та же строка кода, другой результат
```
##### 4.1-4 ==Достоинства и недостатки ООП==
**Достоинства:**
- Повторное использование кода (наследование).
- Модульность и локализация изменений.
- Упрощение сопровождения и масштабирования.
- Соответствие реальной предметной области (объекты = сущности).

**Недостатки:**
- Более высокий порог входа по сравнению с процедурным подходом.
- Избыточность для простых задач.
- Накладные расходы на динамическое связывание и создание объектов.
- Сложность проектирования глубоких иерархий (наследование ради наследования).
##### 4.1-5 ==Классы и объекты. Свойства объектов. Пример==
**Класс** — это описание (шаблон), определяющее:
- Поля (переменные) — описывают состояние.
- Методы — описывают поведение.
- Конструкторы — определяют способ создания экземпляра.

**Объект** — конкретный экземпляр класса, обладающий:
- **Состоянием** (state): совокупность значений полей в данный момент.
- **Поведением** (behavior): набор методов, определяющих, что объект может делать.
- **Идентичностью** (identity): уникальность, отличающая один объект от другого даже при одинаковом состоянии.

**Пример:** 
класс Car (поля: brand, speed; методы: accelerate(), brake()). 
Объект: Car myCar = new Car("Toyota", 0). 
Состояние: brand = "Toyota", speed = 0. 
Поведение: myCar.accelerate(20) — скорость становится 20. 
Идентичность: ссылка myCar указывает на конкретный экземпляр в памяти, другой new Car("Toyota", 0) — другой объект с той же ссылкой.
#### 4.2 Члены класса. Модификаторы объявления класса. Пакеты. Пространства имен. Модуль компиляции.
Члены класса — поля, методы, конструкторы, блоки инициализации, вложенные классы и интерфейсы. 

Модификаторы объявления класса: 
public (класс доступен везде), 
default/package-private (доступен в своем пакете), 
abstract (нельзя создать экземпляр), 
final (нельзя менять/наследовать) 

Пакеты — способ организации классов в логические группы и создания пространств имен, предотвращающих конфликты имен. 

Модуль компиляции — это файл .java, который может содержать опциональный оператор package, опциональные операторы import и объявления классов или интерфейсов.
##### 4.2-1 ==Члены класса==
Членами класса в Java называются компоненты, объявленные внутри тела класса. К ним относятся:

- **Поля (fields):** переменные, хранящие состояние объекта или класса (статические поля). Бывают примитивных типов и ссылочных типов.
- **Методы (methods):** функции, определяющие поведение объекта или класса. Включают обычные методы, статические методы, абстрактные методы (в абстрактных классах и интерфейсах).
- **Конструкторы (constructors):** специальные блоки кода, вызываемые при создании объекта. Имя конструктора совпадает с именем класса, возвращаемый тип не указывается.
- **Блоки инициализации:** обычные (экземпляра) и статические. Выполняются при создании объекта или загрузке класса соответственно.
- **Вложенные классы и интерфейсы:** классы, объявленные внутри другого класса. Могут быть статическими и нестатическими (внутренними).

Порядок объявления членов внутри класса не регламентирован, но принято группировать: сначала поля, затем конструкторы, затем методы.
##### 4.2-2 ==Модификаторы объявления класса==
При объявлении класса перед ключевым словом class могут быть указаны модификаторы, определяющие видимость и свойства класса.

**Модификаторы доступа (видимости):**
- **public:** класс доступен из любого другого класса в любом пакете. В одном файле .java может быть только один public-класс, и имя файла должно совпадать с его именем.
- **default (package-private):** модификатор отсутствует. Класс доступен только внутри своего пакета. Это уровень доступа по умолчанию.

**Другие модификаторы класса:**
- **abstract:** класс не может быть инстанцирован (нельзя создать его экземпляр). Может содержать абстрактные методы (без тела) и обычные методы. Используется как базовый класс для наследования.
- **final:** класс не может иметь подклассов (запрещено наследование). Все методы final-класса неявно являются final.
##### 4.2-3 ==Пакеты. Пространства имен==
**Пакет (package)** — это механизм группировки связанных классов и интерфейсов в единую структурную единицу. Аналогичен папке в файловой системе.

**Назначение пакетов:**
- Организация кода в логические модули.
- Создание пространств имен, предотвращающих конфликты имен классов.
- Управление доступом: классы с default-видимостью доступны только внутри пакета.

**Объявление пакета:** 
	оператор package помещается первой строкой файла .java (кроме комментариев). Пример: `package com.example.project;`. Если package не указан, класс попадает в безымянный пакет (не рекомендуется для реальных проектов).

**Импорт:** 
	для использования классов из других пакетов применяется оператор import. Можно импортировать конкретный класс (`import java.util.ArrayList;`) или все классы пакета (`import java.util.*;`). Классы пакета java.lang импортируются автоматически.

**Соглашение об именовании:** 
	имена пакетов строятся на основе доменного имени организации в обратном порядке и записываются строчными буквами. Пример: `ru.samara.ssau.department`.
##### 4.2-4 ==Модуль компиляции==
**Модуль компиляции** — это один файл с исходным кодом на Java с расширением .java. Это минимальная единица, обрабатываемая компилятором javac.

**Структура модуля компиляции (сверху вниз):**
1. Оператор package (опционально, но рекомендуется). Может быть только один.
2. Операторы import (опционально, любое количество).
3. Объявления классов, интерфейсов, перечислений (одно или несколько).

**Правила модуля компиляции:**
- В одном файле может быть только один публичный класс/интерфейс, имя которого должно совпадать с именем файла.
- Дополнительно могут присутствовать непубличные классы в том же файле.
- После компиляции каждый класс компилируется в отдельный .class-файл. Если в файле объявлено три класса, будет создано три .class-файла.

**Пример структуры файла:**
```
package ru.example.app;

import java.util.List;
import java.io.*;

public class Main {
    // тело класса
}

class Helper {
    // непубличный вспомогательный класс`
}
```
#### 4.3 Поля. Модификаторы доступа. Методы. Модификаторы доступа. Метод main.
Поля — переменные, объявленные в теле класса. Хранят состояние объекта (экземплярные) или класса (статические). 

Модификаторы доступа: 
private (только внутри класса), 
default (внутри пакета), 
protected (внутри пакета + наследники), public (везде). 

Методы — блоки кода, выполняющие действия. 
Модификаторы доступа методов те же. 
Метод main — точка входа в программу: `public static void main(String[] args)`. Должен быть public, static, void, принимать массив String.
##### 4.3-1 ==Поля. Модификаторы доступа==
**Поля (fields)** — переменные, объявленные непосредственно в теле класса (не внутри методов). Каждое поле имеет тип и имя. Поля бывают:
- **Экземплярные (instance fields):** у каждого объекта свое значение. Объявляются без static.
- **Статические (static fields, class variables):** одно значение на весь класс. Объявляются с static.

**Модификаторы доступа полей:**
- **private:** поле доступно только внутри своего класса. Основа инкапсуляции — скрытие внутреннего состояния от внешнего мира.
- **default (package-private, модификатор отсутствует):** доступ внутри своего пакета.
- **protected:** доступ внутри пакета + доступ наследникам (даже в других пакетах).
- **public:** доступ отовсюду.

**Другие модификаторы полей:**
- **static:** поле принадлежит классу, а не экземпляру.
- **final:** поле нельзя изменить после инициализации (константа).
- **transient:** поле не участвует в сериализации.
- **volatile:** поле синхронизируется между потоками, чтение и запись атомарны.

Рекомендуемая практика: поля объявлять private, доступ — через public get/set-методы (геттеры и сеттеры).
##### 4.3-2 ==Методы. Модификаторы доступа==
**Методы** — блоки кода, объявленные в теле класса, определяющие поведение. Состоят из сигнатуры (модификаторы, возвращаемый тип, имя, параметры) и тела (фигурные скобки). Методы бывают экземплярные и статические (static).

**Модификаторы доступа методов** те же, что у полей: private, default, protected, public. Принципы те же: private — вспомогательные методы внутри класса, public — интерфейс класса.

**Другие модификаторы методов:**
- **static:** метод принадлежит классу, вызывается без создания объекта.
- **final:** метод нельзя переопределить в наследниках.
- **abstract:** метод без тела (в абстрактных классах и интерфейсах). Реализация — в наследниках.
- **synchronized:** метод может выполняться только одним потоком одновременно.
- **native:** метод написан на другом языке (C/C++), реализация — в нативной библиотеке.

**Перегрузка методов (overloading):** в одном классе можно объявить несколько методов с одинаковым именем, но разной сигнатурой (разные типы/количество параметров). Возвращаемый тип не участвует в различении.
##### 4.3-3 ==Метод main==
Метод main — точка входа в Java-приложение. JVM ищет именно этот метод для запуска программы.

**Стандартная сигнатура:**
```
public static void main(String[] args)
```
**Пояснение каждого элемента:**
- **public:** JVM должна иметь доступ к методу извне класса.
- **static:** метод вызывается без создания объекта класса. JVM не создает экземпляр, а вызывает метод напрямую.
- **void:** метод не возвращает значение. По завершении программа заканчивается.
- **main:** зарезервированное имя, которое ищет JVM.
- **String[] args:** массив строковых аргументов командной строки, передаваемых при запуске.

**Допустимые вариации:**
- `public static void main(String... args)` — varargs.
- `public static void main(String args[])` — скобки после имени параметра.
#### 4.4 Создание объектов. Конструкторы. Блоки инициализации. Статическая инициализация.
Объект создается оператором `new`, который выделяет память в куче и вызывает конструктор. 
Конструктор — специальный метод, имя совпадает с именем класса, не имеет возвращаемого типа. Может быть несколько (перегрузка). Если конструктор не объявлен, компилятор создает конструктор по умолчанию без параметров. 
Блоки инициализации: нестатические (instance initializer) — выполняются при каждом создании объекта перед телом конструктора; статические (static initializer) — выполняются один раз при загрузке класса. 
Порядок инициализации: статические поля и блоки (по порядку объявления) → нестатические поля и блоки → тело конструктора.
##### 4.4-1 ==Создание объектов==
Объект в Java создается с помощью оператора `new`. Процесс состоит из трех этапов:

1. **Выделение памяти:** JVM выделяет область в куче (heap) под новый объект. Память инициализируется значениями по умолчанию (0 для чисел, false для boolean, null для ссылок).
2. **Инициализация объекта:** выполняются блоки инициализации экземпляра и инициализация полей в порядке их объявления. Затем вызывается тело конструктора.
3. **Возврат ссылки:** оператор `new` возвращает ссылку на созданный объект, которая может быть присвоена переменной.

**Синтаксис:**
```
ClassName obj = new ClassName(аргументы);
```
Объекты живут в куче. Когда на объект больше нет ссылок, он становится кандидатом на удаление сборщиком мусора (Garbage Collector). Явного удаления объектов в Java нет.
##### 4.4-2 ==Конструкторы==
**Конструктор** — специальный блок кода, вызываемый при создании объекта. Основное назначение — инициализация состояния объекта (полей).

**Свойства конструктора:**
- Имя конструктора всегда совпадает с именем класса (включая регистр).
- Не имеет возвращаемого типа (даже void не пишется).
- Может иметь модификаторы доступа (public, protected, default, private).
- Можно объявить несколько конструкторов (перегрузка) с разным набором параметров.

**Конструктор по умолчанию:**  
Если в классе не объявлено ни одного конструктора, компилятор автоматически создает конструктор по умолчанию (default constructor). Он public, без параметров, с пустым телом: `public ClassName() {}`. Если в классе объявлен любой конструктор (с параметрами), конструктор по умолчанию уже не создается.

**Вызов другого конструктора того же класса:**  
Через `this(параметры)` в первой строке конструктора. Позволяет избежать дублирования кода.

**Вызов конструктора родительского класса:**  
Через `super(параметры)` в первой строке конструктора. Если явно не указан, компилятор вставляет `super()` — вызов конструктора родителя без параметров. Это важно: родитель инициализируется до дочернего класса.

**Private конструктор:** используется в паттерне Singleton или в классах только со статическими методами (запрещает создание экземпляров извне).
##### 4.4-3 ==Блоки инициализации==
Блок инициализации — это фрагмент кода в фигурных скобках непосредственно в теле класса (вне методов). Может содержать любой код: присваивания, вызовы методов, условия, циклы.

**Нестатический блок инициализации (instance initializer):**
```
{
    // код
}
```
- Выполняется при каждом создании объекта.
- Выполняется до тела конструктора.
- Если блоков несколько, выполняются в порядке объявления сверху вниз.
- Используется, когда инициализация сложнее, чем простое присваивание, и общая для всех конструкторов. Позволяет избежать дублирования кода.
##### 4.4-4 ==Статическая инициализация==
**Статический блок инициализации (static initializer):**
```
static {
    // код
}
```
- Выполняется один раз при первой загрузке класса в JVM (до создания первого объекта, до вызова любого статического метода или обращения к статическому полю).
- Если статических блоков несколько, выполняются в порядке объявления.
- Используется для инициализации статических полей, требующих сложной логики (например, заполнение статической коллекции).
- Статические поля инициализируются либо при объявлении (`static int x = 10;`), либо в статическом блоке. Порядок — по порядку объявления.

**Порядок инициализации при создании объекта (полный):**

1. Статические поля и статические блоки родительского класса (по порядку).
2. Статические поля и статические блоки дочернего класса (по порядку).
3. Нестатические поля и блоки родительского класса (по порядку).
4. Конструктор родительского класса.
5. Нестатические поля и блоки дочернего класса (по порядку).
6. Конструктор дочернего класса.
#### 3.5 Исключения. Родительский класс исключений. Выбрасывание исключений. Объявляемые и не объявляемые исключения. Пример. Синхронные и асинхронные исключения. Пример. Предложение throws. try, catch и finally.
Исключение — событие, нарушающее нормальный ход выполнения программы. Все исключения — наследники класса Throwable. Две ветви: Error (критические ошибки JVM, не обрабатываются) и Exception (исключения приложения). 
Exception делится на checked (объявляемые, компилятор требует обработки: IOException, SQLException) и unchecked (необъявляемые: RuntimeException и его наследники — NullPointerException, ArithmeticException). 
Обработка: try (блок с потенциальной ошибкой), catch (перехват исключения), finally (выполняется всегда). throws — предупреждение в сигнатуре метода о возможном исключении. 
Синхронные исключения — результат действия программы (деление на ноль). 
Асинхронные — не связаны с кодом (сбой JVM).
##### 4.5-1 ==Исключения. Родительский класс исключений==
**Исключение (exception)** — это событие, возникающее во время выполнения программы и прерывающее нормальный поток инструкций. Исключение представляет собой объект, который создается при возникновении ошибки и передается системе исполнения (runtime). Процесс передачи называется выбрасыванием исключения.

**Иерархия классов исключений (корень — Throwable):**
```
Throwable
├── Error
└── Exception
    └── RuntimeException
```
- **Throwable** — корневой класс для всех исключительных ситуаций. Имеет методы: `getMessage()` (текст ошибки), `printStackTrace()` (трассировка стека), `getCause()` (причина).
- **Error** — серьезные ошибки уровня JVM, не связанные с логикой приложения (OutOfMemoryError, StackOverflowError, NoClassDefFoundError). Не требуют обработки, программа обычно не может их восстановить.
- **Exception** — исключения уровня приложения, которые можно и нужно обрабатывать.
##### 4.5-2 ==Выбрасывание исключений==
Исключение создается и выбрасывается с помощью оператора **throw**.

**Синтаксис:**
throw new ТипИсключения("сообщение");

Throw работает следующим образом: создается объект исключения, выполнение текущего метода немедленно прекращается, JVM ищет подходящий обработчик (catch) в стеке вызовов. Если обработчик не найден, поток завершается аварийно.

**Пример:**
```
if (age < 0) {
    throw new IllegalArgumentException("Возраст не может быть отрицательным");
}
```
Исключения могут быть выброшены не только явно через throw, но и неявно — средой выполнения (JVM), например, при выходе за границы массива.
##### 4.5-3 ==Объявляемые и не объявляемые исключения. Пример==
**Объявляемые исключения (checked exceptions):**
- Наследники класса Exception, кроме RuntimeException и его потомков.
- Компилятор требует, чтобы метод либо обработал такое исключение (try-catch), либо объявил его в сигнатуре (throws).
- Представляют ситуации, которые можно предвидеть и восстановить (файл не найден, сетевое соединение разорвано).
- Примеры: IOException, SQLException, ClassNotFoundException, FileNotFoundException.

**Необъявляемые исключения (unchecked exceptions):**
- Наследники RuntimeException и Error.
- Компилятор не требует ни обработки, ни объявления в throws.
- Обычно указывают на ошибки программирования (RuntimeException: NullPointerException, IndexOutOfBoundsException, ArithmeticException, IllegalArgumentException) или критические сбои (Error).
- Проверка компилятором для них отключена.

**Пример:**
```
// checked — нужно обрабатывать или объявлять
public void readFile() throws IOException {
    FileReader fr = new FileReader("file.txt"); // может бросить FileNotFoundException
}

// unchecked — обработка не требуется
public int divide(int a, int b) {
    return a / b; // может бросить ArithmeticException, если b == 0
}
```
##### 4.5-4 ==Синхронные и асинхронные исключения. Пример==
**Синхронные исключения:**
- Возникают непосредственно в результате выполнения инструкций программы.
- Связаны с конкретным действием: деление на ноль, обращение к null-ссылке, выход за границы массива, явный throw.
- Происходят в том же потоке управления, что и вызвавший их код.
- Примеры: NullPointerException при вызове метода у null, ArrayIndexOutOfBoundsException при обращении к несуществующему индексу.
    

**Асинхронные исключения:**
- Не связаны напрямую с выполняющимся кодом.
- Могут возникнуть в любой момент выполнения программы независимо от текущей инструкции.
- В основном это Error: OutOfMemoryError (закончилась память в куче), StackOverflowError (переполнение стека вызовов), InternalError (внутренняя ошибка JVM).
- Также асинхронным может быть ThreadDeath (устаревший, не используется).

**Пример:**
```
// синхронное — прямое следствие кода
String s = null;
int len = s.length(); // NullPointerException — синхронное

// асинхронное — может возникнуть в любом месте
// бесконечная рекурсия рано или поздно вызовет StackOverflowError
public void recursive() {
    recursive(); // StackOverflowError — асинхронное
}
```
##### 4.5-5 ==Предложение throws==
**throws** — ключевое слово, используемое в сигнатуре метода для объявления того, что метод может выбросить checked-исключения указанных типов. Синтаксис:
```
модификаторы Тип имяМетода(параметры) throws Тип1, Тип2 {
    // тело метода
}
```
Назначение throws: предупредить вызывающий код о потенциальной опасности. Вызывающий метод обязан либо обработать эти исключения (try-catch), либо пробросить их дальше, также указав в своем throws.
```
public void processFile(String path) throws IOException, SQLException {
    // чтение файла и запись в БД
}
```
##### 4.5-6 ==try, catch и finally==
Конструкция try-catch-finally предназначена для перехвата и обработки исключений.

**Синтаксис:**
```
try {
    // код, который может выбросить исключение
} catch (Тип1 e) {
    // обработка исключения Тип1
} catch (Тип2 e) {
    // обработка исключения Тип2
} finally {
    // код, выполняемый в любом случае
}
```
**try-блок:** содержит потенциально опасный код. Если исключение выбрасывается, выполнение try немедленно прекращается, управление передается в подходящий catch.

**catch-блок:** перехватывает исключения указанного типа и его наследников. Блоков catch может быть несколько, проверяются сверху вниз. Важно: более общие типы должны быть ниже (catch Exception после catch IOException). Начиная с Java 7 можно перехватывать несколько типов в одном catch: `catch (IOException | SQLException e)`.

**finally-блок:** содержит код, который выполняется всегда, независимо от того, было исключение или нет, перехвачено оно или проброшено дальше, был ли return в try/catch. Даже если в try/catch выполнился return, finally выполнится до возврата из метода. Типичное использование: закрытие ресурсов (файлов, соединений).
#### 4.6 Интерфейсы. Модификаторы в объявлениях интерфейсов. Пример простого интерфейса. Объявление интерфейса. Константы и методы в интерфейсах.
Интерфейс — это контракт, определяющий набор методов без их реализации. Класс, реализующий интерфейс (implements), обязан предоставить реализацию всех его методов. Интерфейс может содержать: абстрактные методы (неявно public abstract), константы (неявно public static final), методы с реализацией default и static (Java 8+), private методы (Java 9+). 

Модификаторы интерфейса: public или package-private, abstract (неявно). Интерфейс не может быть final. Множественное наследование интерфейсов разрешено. Константы в интерфейсах — это поля, неявно public static final, должны быть инициализированы при объявлении.
##### 4.6-1 ==Интерфейсы. Модификаторы в объявлениях интерфейсов==
**Интерфейс (interface)** — это ссылочный тип, определяющий контракт: набор сигнатур методов, которые класс-реализатор обязан предоставить. До Java 8 интерфейсы содержали только абстрактные методы и константы. Начиная с Java 8 добавлены методы с реализацией (default и static), с Java 9 — private методы.

**Модификаторы в объявлении интерфейса:**
- **public:** интерфейс доступен из любого пакета.
- **package-private (default):** интерфейс доступен только внутри своего пакета.
- **abstract:** интерфейс неявно является abstract, указывать явно не требуется.

Интерфейс не может быть объявлен как **final** (иначе его нельзя было бы реализовать).

Интерфейс может расширять (extends) несколько других интерфейсов. Это называется множественным наследованием интерфейсов. Класс может реализовать (implements) несколько интерфейсов. Это основной механизм множественного наследования поведения в Java.
##### 4.6-2 ==Пример простого интерфейса Java==
```
public interface Drawable {
    void draw();           // абстрактный метод
    double getArea();      // абстрактный метод
}
```

**Класс, реализующий интерфейс, обязан предоставить реализацию всех методов:**
```
public class Circle implements Drawable {
private double radius;
    public Circle(double radius) {
        this.radius = radius;
    }

    @Override
    public void draw() {
        System.out.println("Рисуем круг радиусом " + radius);
    }

    @Override
    public double getArea() {
        return Math.PI * radius * radius;
    }
}
```
##### 4.6-3 ==Объявление интерфейса==
Объявление интерфейса включает: необязательные модификаторы, ключевое слово `interface`, имя интерфейса, необязательную часть с расширением (extends) и тело интерфейса в фигурных скобках.

**Синтаксис:**
```
\[public] interface ИмяИнтерфейса \[extends Интерфейс1, Интерфейс2, ...] {
    // тело интерфейса
}
```
Тело интерфейса может содержать:

- **Абстрактные методы:** неявно public abstract. Сигнатура метода, заканчивающаяся точкой с запятой.
- **default-методы (Java 8+):** методы с телом, помеченные ключевым словом default. Классы-реализаторы могут переопределить, но не обязаны.
- **static-методы (Java 8+):** методы с телом, принадлежащие интерфейсу. Вызываются через `ИмяИнтерфейса.метод()`. Не наследуются классами.
- **private-методы (Java 9+):** вспомогательные методы внутри интерфейса, не видны снаружи. Могут быть статическими.
- **Константы:** поля, неявно public static final.
##### 4.6-4 ==Константы и методы в интерфейсах==
**Константы в интерфейсах:**  
Любое поле, объявленное в интерфейсе, автоматически становится:
- **public** — доступно везде.
- **static** — принадлежит интерфейсу, а не экземпляру.
- **final** — нельзя изменить после инициализации.

Эти модификаторы указывать необязательно, они подразумеваются. Константа должна быть инициализирована сразу при объявлении. По соглашению имена констант пишутся ЗАГЛАВНЫМИ_БУКВАМИ_С_ПОДЧЕРКИВАНИЕМ.

**Пример:**
```
public interface Constants {
    int MAX_SIZE = 100;           // неявно public static final
    String DEFAULT_NAME = "User"; // неявно public static final
}
```
#### 4.7 Расширение интерфейсов. Наследование и сокрытие констант. Наследование, переопределение и перегрузка методов. Пустые интерфейсы. Пример. Отличия абстрактного класса от интерфейса.
Интерфейс может расширять (extends) несколько интерфейсов, наследуя их методы и константы. Константы наследуются, но не переопределяются — при объявлении константы с тем же именем в наследнике происходит сокрытие (hiding). 

Методы интерфейса могут быть переопределены в наследующем интерфейсе (уточнение сигнатуры или default-реализации). 

Перегрузка методов — несколько методов с одним именем, но разными параметрами. 

Пустые интерфейсы (маркеры) — интерфейсы без методов (Serializable, Cloneable, Remote), используются для пометки классов и разрешения операций. 
Абстрактный класс может иметь состояние (поля), конструкторы, обычные методы; интерфейс — только константы и методы. Класс наследует только один абстрактный класс, но может реализовать много интерфейсов.
##### 4.7-1 ==Расширение интерфейсов==
Интерфейс может наследовать другие интерфейсы с помощью ключевого слова `extends`. В отличие от классов, интерфейс может наследовать **несколько** интерфейсов одновременно (множественное наследование). 

**Синтаксис:**
```
public interface A extends B, C, D {
    // методы
}
```
Наследующий интерфейс получает все методы и константы родительских интерфейсов. Он может:
- Оставить абстрактные методы без изменений.
- Добавить новые методы.
- Переопределить default-методы как абстрактные (заставив классы-реализаторы предоставить свою реализацию).
- Предоставить новую default-реализацию для унаследованного метода.

Если два родительских интерфейса содержат default-метод с одинаковой сигнатурой, наследующий интерфейс обязан явно разрешить конфликт, переопределив метод.
##### 4.7-2 ==Наследование и сокрытие констант==
Константы интерфейса (неявно public static final) наследуются наследующим интерфейсом и классом-реализатором. Доступ к унаследованной константе возможен по имени интерфейса или через имя класса/интерфейса-наследника.

**Сокрытие (hiding):** если в наследующем интерфейсе объявляется поле с тем же именем, что и в родительском, это **не переопределение**, а **сокрытие**. Родительская константа продолжает существовать, но для доступа к ней нужно указывать квалифицированное имя родительского интерфейса. Аналогичная ситуация возникает при реализации интерфейса классом.
##### 4.7-3 ==Наследование, переопределение и перегрузка методов==
В отношении интерфейсов эти три понятия работают так:

**Наследование методов:** наследующий интерфейс автоматически получает все методы родительского. Абстрактные методы остаются абстрактными, default-методы сохраняют реализацию.

**Переопределение методов (overriding):**
- В интерфейсе-наследнике можно изменить default-реализацию, предоставив новую с тем же именем и сигнатурой (с ключевым словом @Override).
- Можно сделать default-метод снова абстрактным, переопределив его без тела.
- Сигнатура метода (имя + параметры) должна совпадать. Возвращаемый тип может быть сужен до подтипа (ковариантность).

**Перегрузка методов (overloading):**
- Объявление нескольких методов с одинаковым именем, но разными параметрами (количество, типы, порядок).
- Перегрузка возможна и в одном интерфейсе, и между родительским и дочерним интерфейсами.
- Возвращаемый тип может быть любым, перегрузка только по параметрам.

**Пример:**
```
interface Base {
    void process(int x);          // метод 1
    default void log(String msg) { // default
        System.out.println(msg);
    }
}

interface Extended extends Base {
    void process(String s);       // перегрузка метода 1
    @Override
    void log(String msg);         // переопределение — теперь снова абстрактный
}
```
##### 4.7-4 ==Пустые интерфейсы. Пример==
**Пустой интерфейс (маркерный интерфейс, marker interface)** — интерфейс, не содержащий ни методов, ни констант. Он используется только для пометки класса как обладающего некоторым свойством или правом на определенную операцию.

Назначение: предоставить информацию среде выполнения (JVM) или библиотеке о том, что объект этого класса можно подвергнуть специальной обработке. Проверка осуществляется через оператор `instanceof`.

**Стандартные примеры из Java API:**
- **java.io.Serializable:** помечает классы, объекты которых могут быть сериализованы (сохранены в поток байтов). Если класс не реализует Serializable, при попытке сериализации будет выброшено NotSerializableException.
- **java.lang.Cloneable:** помечает классы, для которых разрешено клонирование через метод Object.clone(). Без этой пометки clone() бросает CloneNotSupportedException.
- **java.rmi.Remote:** помечает интерфейсы, методы которых могут вызываться удаленно.

**Пример:**
```
if (obj instanceof Serializable) {
    // можно безопасно сериализовать
    outputStream.writeObject(obj);
}
```
##### 4.7-5 ==Отличия абстрактного класса от интерфейса==

| Характеристика               | Абстрактный класс (abstract class)                         | Интерфейс (interface)                                                                  |
| ---------------------------- | ---------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| Ключевое слово               | abstract class                                             | interface                                                                              |
| Множественное наследование   | Класс может наследовать только один абстрактный класс      | Класс может реализовать много интерфейсов; интерфейс может расширять много интерфейсов |
| Поля (состояние)             | Может иметь любые поля (экземплярные и статические)        | Только константы (public static final)                                                 |
| Конструкторы                 | Может иметь конструкторы                                   | Не может иметь конструкторов                                                           |
| Модификаторы доступа методов | Любые (private, protected, public, default)                | Все методы неявно public (private — только с Java 9 для внутренних)                    |
| Назначение                   | Моделирует иерархию "is-a" с общим состоянием и поведением | Определяет контракт "can-do" без привязки к иерархии                                   |
| final                        | Может быть объявлен final (запрет наследования)            | Не может быть final                                                                    |
| Когда использовать           | Есть общее состояние и поведение для группы классов        | Нужно определить общий контракт для несвязанных классов                                |

**Пример выбора:** абстрактный класс `Animal` (общее состояние: `name`, `age`; общее поведение: `eat()`, абстрактное: `makeSound()`) и интерфейс `Flyable` (контракт способности летать, который могут реализовать и `Bird`, и `Airplane` — классы из разных иерархий).
#### 4.8 Потоки данных. Байтовые потоки. Базовые абстрактные классы байтовых потоков. Символьные потоки. Базовые абстрактные классы символьных потоков. Примеры байтовых и символьных классов потоков. Стандартные потоки. InputStreamReader и OutputStreamWriter.
Потоки данных (streams) — абстракция для чтения/записи данных из источника в приемник. Делятся на байтовые (InputStream/OutputStream, работают с байтами) и символьные (Reader/Writer, работают с символами Unicode). 

Базовые абстрактные классы: InputStream (методы read, close), OutputStream (методы write, flush, close), Reader (read), Writer (write, flush). Стандартные потоки: System.in (InputStream, клавиатура), System.out (PrintStream, консоль), System.err (PrintStream, ошибки). InputStreamReader и OutputStreamWriter — мосты между байтовыми и символьными потоками, преобразуют байты в символы и обратно с указанием кодировки.
##### 4.8-1 ==Потоки данных==
Поток (stream) в Java — это абстракция, представляющая последовательность данных, передаваемых от источника к приемнику. Потоки обеспечивают единообразный доступ к различным источникам данных: файлам, сетевым соединениям, массивам байтов, консоли.

Характеристики потоков:
- **Однонаправленность:** потоки либо входные (чтение), либо выходные (запись).
- **Последовательный доступ:** данные читаются/пишутся последовательно, произвольный доступ поддерживается не всеми потоками.
- **Буферизация:** многие потоки используют внутренний буфер для повышения производительности.

Два основных типа потоков:
- **Байтовые потоки (byte streams):** работают с единицами данных размером 1 байт (8 бит). Используются для любых данных: изображений, звука, бинарных файлов.
- **Символьные потоки (character streams):** работают с символами Unicode (2 байта, char). Используются для текстовых данных, автоматически учитывают кодировку.
##### 4.8-2 ==Байтовые потоки. Базовые абстрактные классы байтовых потоков==
Байтовые потоки работают с данными на уровне отдельных байтов (8 бит). Подходят для любых типов данных: текст, изображения, аудио, бинарные файлы.

**InputStream (входной):**
- int read() — читает один байт (0–255), возвращает -1 в конце.
- int read(byte[] buf) — читает массив байтов.
- void close() — закрывает поток.

**OutputStream (выходной):**
- void write(int b) — записывает один байт (младшие 8 бит).
- void write(byte[] buf) — записывает массив байтов.
- void flush() — сбрасывает буфер.
- void close() — закрывает поток.

Оба класса абстрактные. Конкретные реализации (FileInputStream, ByteArrayInputStream и др.) наследуют их и реализуют методы под конкретный источник/приемник данных.
##### 4.8-3 ==Символьные потоки. Базовые абстрактные классы символьных потоков==
Символьные потоки работают с 16-битными символами Unicode. Автоматически преобразуют байты в символы (и обратно) с учетом кодировки.

**Reader (входной):**
- int read() — читает один символ (0–65535), возвращает -1 в конце.
- int read(char[] buf) — читает массив символов.
- void close() — закрывает поток.

**Writer (выходной):**
- void write(int c) — записывает один символ.
- void write(char[] buf) — записывает массив символов.
- void write(String str) — записывает строку.
- void flush() — сбрасывает буфер.
- void close() — закрывает поток.

Главное отличие от байтовых: внутри используют кодировку для преобразования символов в байты и обратно.
##### 4.8-4 ==Примеры байтовых и символьных классов потоков==
**Байтовые:**

- FileInputStream / FileOutputStream — чтение/запись файлов как байтов.
- ByteArrayInputStream / ByteArrayOutputStream — работа с массивом байтов в памяти.
- BufferedInputStream / BufferedOutputStream — добавляют буферизацию к любому потоку (декораторы).
- DataInputStream / DataOutputStream — чтение/запись примитивных типов (int, double и др.).
- ObjectInputStream / ObjectOutputStream — сериализация/десериализация объектов.

**Символьные:**
- FileReader / FileWriter — чтение/запись текстовых файлов.
- StringReader / StringWriter — работа со строками в памяти как с потоками.
- BufferedReader / BufferedWriter — буферизация + полезные методы readLine() и newLine().
- PrintWriter — форматированный вывод текста (аналог System.out для файлов)

**Принцип:** большинство продвинутых потоков являются декораторами — оборачивают базовый поток и добавляют функциональность. Пример: `new BufferedReader(new FileReader("file.txt"))`.
##### 4.8-5 ==Стандартные потоки==
Java предоставляет три предопределенных потока, доступных через класс **System**:

- **System.in:** объект типа **InputStream**. По умолчанию связан с клавиатурой. Используется для чтения ввода пользователя. Обычно оборачивается: `BufferedReader reader = new BufferedReader(new InputStreamReader(System.in));`
- **System.out:** объект типа **PrintStream** (наследник OutputStream). По умолчанию связан с консолью. Используется для обычного вывода: `System.out.println("текст");`
- **System.err:** объект типа **PrintStream**. По умолчанию связан с консолью. Предназначен для вывода сообщений об ошибках. Выводится в тот же поток, что и System.out, но может быть перенаправлен отдельно: `System.err.println("ошибка");`

Стандартные потоки можно перенаправить с помощью методов System.setIn(), System.setOut(), System.setErr(). Это используется для тестирования или перенаправления вывода в файл.
##### 4.8-6 ==InputStreamReader и OutputStreamWriter==
Это классы-мосты, соединяющие байтовые и символьные потоки.

**InputStreamReader:**
- Наследует **Reader**.
- Конструктор принимает **InputStream** и (опционально) имя кодировки.
- Читает байты из байтового потока и преобразует их в символы согласно указанной кодировке.
- Если кодировка не указана, используется системная кодировка по умолчанию.
- Пример: `InputStreamReader isr = new InputStreamReader(new FileInputStream("file.txt"), "UTF-8");`

**OutputStreamWriter:**
- Наследует **Writer**.
- Конструктор принимает **OutputStream** и (опционально) имя кодировки.
- При записи преобразует символы в байты согласно указанной кодировке и передает в байтовый поток.
- Пример: `OutputStreamWriter osw = new OutputStreamWriter(new FileOutputStream("file.txt"), "UTF-8");`

**Зачем нужны:** позволяют работать с текстовыми данными в нужной кодировке, когда доступен только байтовый поток (например, сетевой сокет или сжатый файл). FileReader/FileWriter внутри себя используют эти классы, но всегда с системной кодировкой, поэтому для явного указания кодировки рекомендуется использовать InputStreamReader/OutputStreamWriter напрямую с FileInputStream/FileOutputStream.
#### 4.9 Сериализация объектов. Подготовка классов к сериализации. Порядок сериализации и десериализации. Настройка механизма сериализации. Контроль версий объектов.
Сериализация — процесс преобразования объекта в последовательность байтов для сохранения в файл или передачи по сети. 
Десериализация — обратный процесс восстановления объекта из байтов. Для сериализации класс должен реализовать интерфейс Serializable (маркерный). 

Поля с transient не сериализуются. При десериализации конструктор не вызывается, объект восстанавливается из байтового представления. 

Для настройки можно определить методы writeObject/readObject. Контроль версий — через статическое поле serialVersionUID, которое должно совпадать при сериализации и десериализации, иначе — InvalidClassException.
##### 4.9-1 ==Сериализация объектов==
**Сериализация** — это процесс преобразования графа объектов в последовательность байтов. Может использоваться для:
- Сохранения состояния объекта в файл.
- Передачи объекта по сети.
- Глубокого клонирования (deep copy) через сериализацию/десериализацию.
- Кэширования объектов.

**Десериализация** — обратный процесс: восстановление графа объектов из последовательности байтов. При этом создаются новые объекты с теми же значениями полей.

Основные классы для сериализации в Java:
- **ObjectOutputStream** — записывает объекты в поток (метод writeObject).
- **ObjectInputStream** — читает объекты из потока (метод readObject).

При сериализации сохраняется весь граф объектов, достижимых по ссылкам из сериализуемого объекта. Если один и тот же объект встречается несколько раз, он будет записан только один раз (сохраняется целостность ссылок).
##### 4.9-2 ==Подготовка классов к сериализации==
Чтобы объекты класса можно было сериализовать, класс должен **реализовать интерфейс java.io.Serializable**. Это маркерный интерфейс (без методов). Если класс не реализует Serializable, при попытке сериализации будет выброшено NotSerializableException.

Шаги по подготовке класса:
1. Добавить `implements Serializable` в объявление класса.
2. Убедиться, что все поля класса также Serializable (или примитивные). Если поле — объект, его класс тоже должен реализовать Serializable.
3. Поля, которые не нужно или нельзя сериализовать, пометить **transient** (например, пароль, соединение с БД, поток ввода-вывода). Значения transient-полей при десериализации будут инициализированы значениями по умолчанию.
4. По желанию объявить **serialVersionUID** (рекомендуется всегда).

Вложенные статические классы Serializable, но внешний класс тоже должен быть Serializable. Для нестатических внутренних классов сериализация сложнее, так как они неявно ссылаются на внешний объект.
##### 4.9-3 ==Порядок сериализации и десериализации==
**Порядок сериализации:**

1. Создается ObjectOutputStream, связанный с выходным потоком (например, FileOutputStream).
2. Вызывается метод writeObject() для корневого объекта.
3. JVM рекурсивно обходит граф объектов, записывая:
    - Информацию о классе (имя, serialVersionUID, поля).    
    - Значения примитивных полей.    
    - Сериализованные объекты ссылочных полей.
4. При обнаружении уже сериализованного объекта записывается ссылка на него (обработка циклических зависимостей).
5. Вызывается flush() и close().

**Порядок десериализации:**

1. Создается ObjectInputStream.
2. Вызывается метод readObject() — возвращает Object, нужно явное приведение типа.
3. JVM читает информацию о классе и создает **новый объект без вызова конструктора**. Память выделяется напрямую.
4. Восстанавливаются значения полей (кроме transient — получают значения по умолчанию).
5. Если класс имеет метод readObject(), он вызывается для дополнительной инициализации.
6. Восстанавливается граф объектов с сохранением ссылочной целостности.
##### 4.9-4 ==Настройка механизма сериализации==
Сериализацию можно настраивать, определив в классе специальные методы (не переопределяя, так как они не наследуются):

- **private void writeObject(ObjectOutputStream out) throws IOException** — вызывается при сериализации. Внутри можно выполнить дополнительные действия перед записью полей (out.defaultWriteObject()) или вместо неё.
- **private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException** — вызывается при десериализации. Позволяет выполнить дополнительную инициализацию после восстановления полей (in.defaultReadObject()).
- **private void readObjectNoData() throws ObjectStreamException** — вызывается, если сериализованный объект имеет версию класса, в которой нет данных для этого объекта.

Также можно полностью управлять форматом сериализации, реализовав интерфейс **Externalizable** (вместо Serializable). В этом случае класс должен реализовать методы writeExternal и readExternal, и вся ответственность за сохранение/восстановление полей ложится на программиста. Конструктор без параметров обязателен.
##### 4.9.5 ==Контроль версий объектов==
При сериализации вместе с данными сохраняется идентификатор версии класса — **serialVersionUID**. Это статическое поле типа long. При десериализации JVM сравнивает serialVersionUID класса в байтовом потоке с serialVersionUID загруженного класса.

- Если значения совпадают — десериализация продолжается.
- Если не совпадают — выбрасывается **InvalidClassException**.

Если serialVersionUID не объявлен явно, компилятор вычисляет его автоматически на основе структуры класса (имени, полей, методов, модификаторов). Даже незначительное изменение класса (добавление метода, изменение модификатора поля) приводит к новому значению, и ранее сериализованные объекты не смогут быть десериализованы.
#### 4.10 Расширенный класс. Конструкторы расширенных классов. Порядок выполнения конструкторов. Перегрузка и переопределение методов. Совместимость. Явное преобразование типов.
Расширенный класс (subclass) — класс, наследующий другой класс с помощью extends. Конструктор расширенного класса обязан первой строкой вызвать конструктор родителя: явно через super() или неявно (компилятор вставит super()). 

Порядок: статические блоки родителя → статические блоки наследника → нестатические блоки и конструктор родителя → нестатические блоки и конструктор наследника. Перегрузка (overloading) — методы с одним именем, но разными параметрами в одном классе. 

Переопределение (overriding) — метод наследника с той же сигнатурой, что у родителя. Совместимость типов: ссылке родителя можно присвоить объект наследника (восходящее преобразование, автоматическое). Явное преобразование (нисходящее) — приведение ссылки родителя к типу наследника через (Тип), требует проверки instanceof.
##### 4.10-1 ==Расширенный класс==
Расширенный класс (subclass, наследник, дочерний класс) — класс, который наследует другой класс с помощью ключевого слова **extends**. В Java каждый класс может иметь только одного прямого родителя (одиночное наследование), за исключением Object, который не имеет родителя.

**Синтаксис:**
```
public class Child extends Parent {
    // новые поля и методы
}
```
Наследник получает:
- Все поля и методы родителя (кроме private — они не видны напрямую, но присутствуют в объекте).
- Возможность использовать унаследованные члены (public, protected, package-private в том же пакете).
- Возможность переопределить методы родителя.
- Обязанность реализовать абстрактные методы, если родитель абстрактный.
##### 4.10-2 ==Конструкторы расширенных классов==
Конструкторы не наследуются. Каждый конструктор наследника **обязан** первой инструкцией вызвать конструктор родителя через ключевое слово **super**.

Способы вызова:
- **Явный вызов:** `super(параметры);` — вызывает конкретный конструктор родителя.
- **Неявный вызов:** если `super()` не написано, компилятор автоматически вставляет `super();` — вызов конструктора родителя без параметров.

Если у родителя нет конструктора без параметров (он явно объявлен только с параметрами), то наследник обязан явно вызвать `super(параметры)`, иначе — ошибка компиляции.

Также конструктор может вызвать другой конструктор того же класса через **this(параметры)**. В любом случае цепочка вызовов должна завершиться вызовом конструктора родителя.
##### 4.10-3 ==Порядок выполнения конструкторов==
При создании объекта наследника инициализация происходит в строгом порядке:

1. Статические поля и статические блоки родителя (по порядку объявления).
2. Статические поля и статические блоки наследника (по порядку).
3. Нестатические поля и блоки инициализации родителя (по порядку).
4. Тело конструктора родителя.
5. Нестатические поля и блоки инициализации наследника (по порядку).
6. Тело конструктора наследника.

Принцип: родитель должен быть полностью инициализирован до того, как начнется инициализация наследника. Это гарантирует, что методы наследника, вызванные из конструктора родителя, не обратятся к еще не инициализированным полям наследника (хотя это все равно плохая практика).
##### 4.1-4 ==Перегрузка и переопределение методов==
**Перегрузка (overloading):**
- Несколько методов с **одинаковым именем**, но **разными параметрами** (количество, тип, порядок).
- Происходит в пределах **одного класса** или между родителем и наследником.
- Решается на этапе **компиляции** (статическое связывание).
- Возвращаемый тип может быть любым, он не участвует в различении.
- Модификатор доступа может быть любым.

**Пример перегрузки:**
```
void print(int x) { }
void print(String s) { }
void print(int x, int y) { }
```
**Переопределение (overriding):**
- Метод наследника с **такой же сигнатурой** (имя + параметры), что и у родителя.
- Возвращаемый тип должен быть **таким же или подтипом** (ковариантность).
- Модификатор доступа не может быть более ограничивающим (private → default → protected → public — можно расширять).
- Решается на этапе **выполнения** (динамическое связывание).
- Нельзя переопределить: static-методы (они скрываются), final-методы, private-методы (они не видны).
- Рекомендуется использовать аннотацию @Override.

**Пример переопределения:**
```
class Parent {
    void speak() { System.out.println("Parent"); }
}
class Child extends Parent {
    @Override
    void speak() { System.out.println("Child"); }
}
```
##### 4.10.5 ==Совместимость. Явное преобразование типов==
Java — строго типизированный язык, но наследование создает совместимость типов (is-a отношение).

**Восходящее преобразование (upcasting):**
- Присвоение ссылки на объект наследника переменной типа родителя.
- Всегда безопасно, выполняется автоматически (неявно).
- Объект не меняется, меняется только тип ссылки.
- Через родительскую ссылку доступны только члены, объявленные в родителе.
- Пример: `Parent p = new Child();`

**Нисходящее преобразование (downcasting):**
- Приведение ссылки типа родителя к типу наследника.
- Требует **явного** указания типа в скобках.
- Опасно: если реальный объект не является экземпляром указанного типа, выбрасывается **ClassCastException**.
- Перед приведением рекомендуется проверять тип через **instanceof**.

**Пример:**
```
Parent p = new Child();
if (p instanceof Child) {
    Child c = (Child) p;  // явное приведение
    c.childMethod();
}
```
#### 3.11 Сокрытие полей. Доступ к унаследованным членам. Возможность доступа и переопределение. Сокрытие статических членов. Служебное слово super. Проверка типа.
Сокрытие полей — объявление в наследнике поля с тем же именем, что у родителя. Родительское поле не исчезает, но доступно только через super или ссылку родительского типа. 
Доступ к унаследованным членам: public и protected доступны всегда, default — только в том же пакете, private — не видны. 
Статические методы не переопределяются, а сокрываются — выбор метода зависит от типа ссылки, а не объекта. 
Ключевое слово super используется для обращения к членам родительского класса (полям, методам, конструкторам). Проверка типа — instanceof, возвращает true для объекта и его родительских классов/интерфейсов.
##### 4.11-1 ==Сокрытие полей==
Сокрытие (hiding) происходит, когда в наследнике объявляется поле с тем же именем, что и поле в родителе. Это **не переопределение** (поля не переопределяются). Родительское поле продолжает существовать в объекте, но поле наследника его «заслоняет».

Поведение зависит от типа ссылки:
- Через ссылку типа наследника доступно поле наследника.
- Через ссылку типа родителя — поле родителя (даже если реальный объект — наследник).
##### 4.11-2 ==Доступ к унаследованным членам==
Унаследованные члены (поля и методы) доступны в наследнике в зависимости от их модификатора доступа:
- **public:** доступны везде, в том числе в наследниках из любых пакетов.
- **protected:** доступны в наследниках всегда (даже из другого пакета), а также внутри своего пакета.
- **default (package-private):** доступны только если наследник находится в том же пакете.
- **private:** не доступны напрямую. Но они присутствуют в объекте и могут влиять на его поведение через публичные методы родителя.

Доступ к унаследованному члену:
- Просто по имени (если нет конфликта имен).
- Через `this.имя`.
- Через `super.имя` (для обращения к родительской версии при сокрытии).
##### 4.11-3 ==Возможность доступа и переопределение==
При переопределении методов действуют правила доступа:

- Модификатор доступа в наследнике может быть **таким же или более открытым**, но не более закрытым. Порядок: private → default → protected → public. Например, protected-метод родителя можно переопределить как public, но нельзя как private.
- Методы с модификатором **private** не переопределяются (они не видны наследнику). Если в наследнике объявить метод с той же сигнатурой, это будет новый метод, не связанный с родительским.
- Методы с модификатором **final** нельзя переопределить.
- Методы с модификатором **static** не переопределяются, а сокрываются (см. 3.11.4).
- Возвращаемый тип при переопределении может быть сужен до подтипа (ковариантность).
##### 4.11-4 ==Сокрытие статических членов==
Статические методы **не переопределяются**, а **сокрываются** (hiding). Это принципиальное отличие от экземплярных методов.

При сокрытии выбор вызываемого метода зависит от **типа ссылки**, а не от реального типа объекта (статическое связывание). Для экземплярных методов — наоборот (динамическое связывание).

Статические поля также сокрываются, аналогично полям экземпляра (см. 3.11.1). Рекомендуется всегда вызывать статические методы через имя класса, а не через ссылку, чтобы избежать путаницы: `Parent.print()`, `Child.print()`.
##### 4.11-5 ==Служебное слово super==
**super** — ключевое слово, используемое для обращения к членам непосредственного родительского класса.

Три варианта использования:

**1. Вызов конструктора родителя:**
super(параметры); // должна быть первой строкой конструктора

**2. Обращение к полю родителя (при сокрытии):**
super.fieldName;

**3. Вызов метода родителя (при переопределении/сокрытии):**
super.methodName(параметры);

super позволяет обратиться к родительской версии метода или поля, даже если она переопределена или сокрыта в текущем классе. Цепочка super не может быть длиннее одного уровня: нельзя написать `super.super.method()`.
##### 4.11-6 ==Проверка типа==
Для проверки реального типа объекта во время выполнения используется оператор **instanceof**.

Синтаксис: 
```
объект instanceof Тип
```
Возвращает:

- **true**, если объект является экземпляром указанного класса или его наследника, либо реализует указанный интерфейс.
- **false**, если объект равен null (instanceof для null всегда false).

**Пример:**
```
Animal a = new Dog();
if (a instanceof Dog) {
    Dog d = (Dog) a;
}
if (a instanceof Animal) { // тоже true — Dog является Animal
}
```
**Альтернативный метод:** `getClass()` возвращает объект Class, представляющий точный класс объекта. Сравнение через == проверяет точное совпадение класса (не учитывает наследование, в отличие от instanceof). `a.getClass() == Dog.class` — true только если a точно Dog, не его наследник.
#### 4.12 Методы и классы final. Методы и классы abstract. Класс Object. Методы класса Object. Клонирование объектов.
final-класс нельзя наследовать (String, Math). 
final-метод нельзя переопределить в наследниках. abstract-класс нельзя инстанцировать, он служит базой для наследования; может содержать abstract-методы без тела. 

Класс Object — корень иерархии всех классов, содержит методы: toString(), equals(Object), hashCode(), getClass(), clone(), finalize(), wait(), notify(), notifyAll(). 

Клонирование — создание копии объекта через метод clone(). Класс должен реализовать интерфейс Cloneable, иначе — CloneNotSupportedException. По умолчанию clone() делает поверхностное копирование; для глубокого нужно переопределить.
##### 4.12-1 ==Методы и классы final==
**final-класс** — класс, от которого **нельзя наследовать**. Объявляется с модификатором `final`. Все его методы неявно становятся final (но не поля). Примеры: String, Math, Integer, System. Используется, когда наследование может нарушить безопасность или логику работы (например, String — для неизменяемости и безопасности).

**final-метод** — метод, который **нельзя переопределить** в наследниках. Позволяет зафиксировать реализацию и гарантировать, что поведение метода не изменится в подклассах. Перегружать final-метод можно (другая сигнатура). Пример: методы, вызываемые из конструктора (во избежание вызова переопределенной версии до инициализации наследника).

**final-поле** — поле, значение которого нельзя изменить после инициализации. Примитив — константа. Ссылочный тип — нельзя переназначить ссылку, но объект по ссылке можно менять. Инициализируется либо при объявлении, либо в конструкторе, либо в блоке инициализации. final-поле не имеет значения по умолчанию и должно быть инициализировано ровно один раз.
##### 4.12-2 ==Методы и классы abstract==
**abstract-класс** — класс, который **нельзя инстанцировать** (нельзя создать через new). Содержит слово `abstract` в объявлении. Может содержать:

- Абстрактные методы (без тела, с `;`).
- Обычные методы с реализацией.
- Поля и конструкторы.

Наследник абстрактного класса обязан реализовать все абстрактные методы или тоже быть объявлен как abstract.

**abstract-метод** — метод без тела (без {}), заканчивается точкой с запятой. Объявляется только в абстрактных классах и интерфейсах. Наследник обязан его реализовать (переопределить). Может быть protected и public, но не private (иначе наследник не увидит). Не может быть static, final, synchronized, native.
##### 4.12-3 ==Класс Object==
**java.lang.Object** — корень иерархии классов в Java. Каждый класс, включая массивы, неявно наследует Object. Если класс не указывает extends, компилятор добавляет `extends Object`. Все объекты обладают методами, определенными в Object.

Ссылочная переменная типа Object может указывать на любой объект: `Object obj = new Dog();`.
##### 4.12-4 ==Методы класса Object==
Основные методы, доступные всем объектам:

- **String toString()** — строковое представление объекта. По умолчанию: `имяКласса@хэшКод`. Рекомендуется переопределять.
- **boolean equals(Object obj)** — сравнение объектов. По умолчанию сравнивает ссылки (\==). Рекомендуется переопределять для логического сравнения. Контракт: рефлексивность, симметричность, транзитивность, консистентность, null возвращает false. Всегда переопределять hashCode() вместе с equals().
- **int hashCode()** — целочисленный хэш объекта. Контракт: одинаковые по equals() объекты должны иметь одинаковый hashCode(); разные объекты могут иметь одинаковый (коллизии). Используется в HashMap, HashSet.
- **Class\<?> getClass()** — возвращает объект Class, представляющий точный класс объекта. Используется для рефлексии.
- **protected Object clone()** — создает копию объекта. Требует реализацию Cloneable.
- **void finalize()** — устаревший (deprecated с Java 9). Вызывался сборщиком мусора. Не рекомендуется использовать.
- Методы для многопоточности: **wait()**, **notify()**, **notifyAll()** — управляют синхронизацией потоков. Должны вызываться только внутри synchronized-блока.
##### 4.12-5 ==Клонирование объектов==
**Клонирование** — создание точной копии объекта. В Java реализуется через метод **clone()** класса Object.

**Условия для клонирования:**
1. Класс должен реализовать маркерный интерфейс **Cloneable**. Без него clone() бросает CloneNotSupportedException.
2. Метод clone() нужно переопределить как public (в Object он protected).
3. Вызвать super.clone() и при необходимости выполнить дополнительное копирование.

**Поверхностное клонирование (shallow copy):** super.clone() копирует значения всех полей. Примитивы копируются полностью. Ссылочные поля — копируются ссылки (оба объекта указывают на один и тот же объект-поле). Подходит для классов без изменяемых ссылочных полей.

**Глубокое клонирование (deep copy):** после super.clone() нужно вручную скопировать все изменяемые объекты по ссылкам. Требует дополнительного кода.
#### 4.13 Рефлексия. Возможности и участники механизма рефлексии. Получение представления класса. Возможности класса Class. Передача параметров в методы. Создание экземпляров классов. Вызов методов. Вызов статического метода.
Рефлексия (reflection) — механизм, позволяющий программе во время выполнения исследовать и модифицировать собственную структуру: получать информацию о классах, полях, методах, конструкторах, создавать объекты, вызывать методы, изменять поля, в том числе private. 

Основные классы пакета java.lang.reflect: Class, Field, Method, Constructor. Получить Class можно через: obj.getClass(), ClassName.class, Class.forName("полное.имя"). Class дает доступ к имени, пакету, родителю, интерфейсам, модификаторам, членам класса. 

Вызов метода: method.invoke(объект, аргументы), для static: method.invoke(null, аргументы). Создание экземпляра: class.newInstance() (устаревший) или constructor.newInstance(аргументы).
##### 4.13-1 ==Рефлексия. Возможности и участники механизма рефлексии==
**Рефлексия** — возможность программы исследовать и изменять свое поведение во время выполнения. Пакет **java.lang.reflect** предоставляет классы для анализа структуры классов.

**Основные возможности рефлексии:**
- Получить полное имя класса, пакет, модификаторы.
- Узнать родительский класс и реализуемые интерфейсы.
- Получить список полей (включая private), их типы, имена.
- Получить список методов, конструкторов, их параметры, возвращаемые типы, исключения.
- Создавать экземпляры классов (даже без известного имени на этапе компиляции).
- Вызывать методы, изменять значения полей (включая private, с отключением проверки доступа через setAccessible(true)).
- Работать с аннотациями.

**Основные участники (классы):**
- **Class\<?>** — представляет класс или интерфейс во время выполнения.
- **Field** — поле класса.
- **Method** — метод класса.
- **Constructor** — конструктор класса.
- **Modifier** — утилитарный класс для разбора модификаторов доступа.
- **AccessibleObject** — базовый класс для Field, Method, Constructor. Содержит setAccessible(boolean).
##### 4.13-2 ==Получение представления класса. Возможности класса Class==
**Способы получить объект Class:**

1. Вызов `getClass()` на объекте:
```
String s = "hello";
Class\<?> c = s.getClass();
```
2. Через литерал .class:
```
Class\<String> c = String.class;
```
3. Через статический метод `Class.forName()` (требует полного имени класса, бросает ClassNotFoundException):
```
Class\<?> c = Class.forName("java.lang.String");
```

**Основные методы класса Class:**
- **getName()** — полное имя класса с пакетом.
- **getSimpleName()** — имя класса без пакета.
- **getPackage()** — пакет (объект Package).
- **getSuperclass()** — родительский класс (Class).
- **getInterfaces()** — массив реализуемых интерфейсов.
##### 4.13-3 ==Передача параметров в методы==
При вызове метода через рефлексию параметры передаются в виде массива объектов `Object[]`. Метод `invoke()` принимает:
- Первый аргумент — объект, на котором вызывается метод (null для статических).
- Второй аргумент — varargs Object... — значения параметров

Примитивные типы автоматически упаковываются в обертки (autoboxing). Типы параметров должны соответствовать сигнатуре метода, иначе — IllegalArgumentException.

**Пример:**
```
Method m = Math.class.getMethod("max", int.class, int.class);
int result = (int) m.invoke(null, 5, 10); // 10
```
##### 4.13-4 ==Создание экземпляров классов==
Два способа создания объекта через рефлексию:

**1. Через Class.newInstance() (устаревший с Java 9):**
Class\<String> c = String.class;
String s = c.newInstance(); // вызывает конструктор без параметров

Недостаток: не передать аргументы, не обработать checked-исключения конструктора. Заменен на Constructor.newInstance().

**2. Через Constructor.newInstance() (рекомендуемый):**
- Получаем Constructor через getConstructor() или getDeclaredConstructor().
- Вызываем newInstance() с аргументами.

**Пример:**
```
Constructor\<String> con = String.class.getConstructor(String.class);
String s = con.newInstance("hello");
```
##### 4.13-5 ==Вызов методов. Вызов статического метода==
**Вызов обычного (экземплярного) метода:**
1. Получить Method: `getMethod("имя", типы_параметров...)` или `getDeclaredMethod()`.
2. Вызвать `method.invoke(объект, аргументы...)`.
3. Если метод private — вызвать `method.setAccessible(true)` перед invoke.

**Пример:**
```
class Person {
    public void say(String message) {
        System.out.println(message);
    }
}
Person p = new Person();
Method m = Person.class.getMethod("say", String.class);
m.invoke(p, "Привет"); // эквивалент p.say("Привет")
```

**Вызов статического метода:**
1. Получить Method аналогично.
2. Вызвать `method.invoke(null, аргументы...)` — первый аргумент null, так как статический метод не привязан к объекту.

**Пример:**
```
Method m = Math.class.getMethod("sqrt", double.class);
double result = (double) m.invoke(null, 16.0); // 4.0, эквивалент Math.sqrt(16.0)
```

**Обработка возвращаемого значения:** invoke() возвращает Object. Если метод void, возвращается null. Примитивные типы автоупаковываются, их нужно явно распаковывать (приведение к int, double и т.д.).
#### 4.14 Статический импорт. Автоупаковка и автораспаковка (автобоксинг). Аргументы переменной длины. Улучшенный цикл for.
Статический импорт (import static) позволяет использовать статические члены класса без указания имени класса. Автоупаковка (autoboxing) — автоматическое преобразование примитива в объект-обертку (int → Integer), автораспаковка (unboxing) — обратный процесс. Аргументы переменной длины (varargs) позволяют методу принимать произвольное количество параметров одного типа (синтаксис: тип... имя). Улучшенный цикл for (for-each) — компактный перебор элементов массива или коллекции без индекса или итератора.
##### 4.14-1 ==Статический импорт==
Статический импорт позволяет импортировать статические члены класса (поля, методы) и использовать их без указания имени класса. Вводится конструкцией `import static`.

**Синтаксис:**
```
import static пакет.Класс.статическийЧлен;
import static пакет.Класс.*; // все статические члены
```
**Пример без статического импорта:**
```
double x = Math.sqrt(16);
double pi = Math.PI;
```
**Пример со статическим импортом:**
```
import static java.lang.Math.sqrt;
import static java.lang.Math.PI;
double x = sqrt(16);
double pi = PI;
```
Преимущества: код короче, читаемость выше при частом использовании (например, константы из библиотек, методы Assert в тестах). Недостатки: при чрезмерном использовании ухудшает читаемость, так как неясно, откуда взялся метод; возможны конфликты имен. Рекомендуется использовать умеренно, в основном для констант и утилитных методов тестирования.
##### 4.14-2 ==Автоупаковка и автораспаковка (автобоксинг)==
**Автоупаковка (autoboxing)** — автоматическое преобразование примитивного типа в соответствующий объект-обертку, выполняемое компилятором Java (с версии 5). Например, int → Integer, double → Double, boolean → Boolean.

**Автораспаковка (unboxing)** — обратный процесс: автоматическое извлечение примитивного значения из объекта-обертки.

**Где применяется:**
- Присваивание примитива переменной ссылочного типа (и наоборот).
- Передача параметров в методы, ожидающие обертку (или примитив).
- Работа с коллекциями (коллекции хранят только объекты).
- Арифметические операции с обертками.
##### 4.14-3 ==Аргументы переменной длины==
Varargs (variable arguments) — механизм, позволяющий методу принимать произвольное количество аргументов одного типа. Синтаксис: многоточие после типа параметра `тип... имя`.

Правила varargs:
- В методе может быть только один varargs-параметр.
- Varargs-параметр должен быть последним в списке параметров.
- Внутри метода varargs ведет себя как массив указанного типа.
- Можно передать любое количество аргументов (включая 0) или сразу массив.

**Перегрузка методов с varargs:** компилятор сначала пытается найти метод без varargs. Если есть и метод с фиксированными параметрами, и метод с varargs, предпочтение отдается фиксированному. Два метода с varargs одного типа создают неоднозначность.
##### 4.14-4 ==Улучшенный цикл for==
Улучшенный цикл for (for-each) появился в Java 5. Предоставляет компактный синтаксис для перебора элементов массива или коллекции без явного индекса или итератора.

**Синтаксис:**
```
for (Тип переменная : коллекция_или_массив) {
    // тело цикла
}
```
**Применение к массивам:**
```
int[] numbers = {1, 2, 3, 4, 5};
for (int n : numbers) {
    System.out.println(n);
}
```

**Ограничения:**
- Нельзя изменить текущий элемент примитивного массива (работаем с копией).
- Нельзя удалить элемент коллекции напрямую — ConcurrentModificationException. Для удаления нужен явный Iterator.
- Нет доступа к индексу текущего элемента.
- Не подходит для изменения массива/коллекции во время обхода.
- Не подходит для перебора нескольких коллекций параллельно.

**Преимущества:** чище и короче кода, исключает ошибки выхода за границы и ошибки с индексами.
#### 4.15 Настраиваемые типы и их особенности. Ограниченные типы. Метасимвольный аргумент. Метасимвол с ограничениями. Настраиваемые методы, конструкторы, интерфейсы. Примеры.
Настраиваемые типы (Generics) позволяют создавать классы, интерфейсы и методы, работающие с разными типами с проверкой на этапе компиляции. Синтаксис: `Class<T>`. 
Ограниченные типы (`T extends Parent`) сужают допустимые типы до наследников указанного класса/интерфейса. Метасимвол `?` (wildcard) означает любой тип. 

Настраиваемые методы объявляют свой типовой параметр перед возвращаемым типом. Конструкторы могут использовать типовые параметры класса, но не объявлять свои. 

Интерфейсы могут быть настраиваемыми (Comparable\<T>, List\<T>). После компиляции информация о типовых параметрах стирается (type erasure), заменяясь на Object или ограничивающий тип.
##### 4.15-1 ==Настраиваемые типы и их особенности==
**Настраиваемые типы (Generics, обобщения)** появились в Java 5. Позволяют создавать классы, интерфейсы и методы, параметризованные типом. Обеспечивают типобезопасность на этапе компиляции без приведения типов.

**Синтаксис объявления настраиваемого класса:**
```
public class Box<T> {
    private T value;
    public void set(T value) { this.value = value; }
    public T get() { return value; }
}
```
**Использование:**
```
Bo<String> box = new Box<>();
box.set("Hello");
String s = box.get(); // без приведения типа
```
**Особенности:**
- Проверка типов на этапе компиляции (нельзя положить Integer в Box\<String>).
- Отсутствие необходимости явного приведения типов.
- Параметр типа может быть только ссылочным типом (не примитивом — используется автобоксинг).
- Статические поля не могут иметь тип параметра класса (общие для всех экземпляров).
##### 4.15-2 ==Ограниченные типы==
Ограниченные типовые параметры сужают множество допустимых типов с помощью ключевого слова **extends** (для интерфейсов тоже extends, не implements).

**Синтаксис:**
```
public class NumericBox\<T extends Number> {
    private T value;
    // можно вызывать методы Number на value
}
```
Здесь T может быть только Number или его наследниками (Integer, Double, Float и т.д.). Попытка `NumericBox<String>` вызовет ошибку компиляции.

**Преимущества:**
- Доступ к методам ограничивающего типа: для `T extends Number` доступен `value.doubleValue()`.
- Сужение допустимых типов, повышение безопасности.
##### 4.15-3 ==Метасимвольный аргумент==
**Wildcard (метасимвол) `?`** означает «любой тип». Используется, когда конкретный тип не важен или неизвестен.

**Применение:** в сигнатурах методов, где не нужно знать точный тип. Не используется при создании экземпляров (new).
```
public void printList(List\<?> list) {
    for (Object obj : list) {
        System.out.println(obj);
    }
}
```
Метод примет `List<String>`, `List<Integer>`, `List<Object>`.

**Ограничения wildcard:**
- Нельзя добавлять элементы в `List<?>` (кроме null), так как неизвестен точный тип. Читать можно (возвращается Object).
- Для добавления элементов нужны ограниченные wildcard
##### 4.15-4 ==Метасимвол с ограничениями==
Различают два типа ограничений wildcard:

**`? extends T` — ограничение сверху (upper bounded).**
- Означает «T или наследник T».
- Подходит для **чтения** (read).
- Методы, возвращающие значение, возвращают T.
- Нельзя добавлять (кроме null), так как неизвестен конкретный наследник.

**`? super T` — ограничение снизу (lower bounded).**
- Означает «T или суперкласс T».
- Подходит для **записи** (write).
- Можно добавлять объекты типа T (и его наследников).
- Чтение возвращает Object.
##### 4.15-5 ==Настраиваемые методы, конструкторы, интерфейсы. Примеры==
**Настраиваемые методы (generic methods):**  
	Объявляют свой типовой параметр перед возвращаемым типом. Могут быть в обычном или настраиваемом классе.
```
public static \<T> T getFirst(List\<T> list) {
    return list.get(0);
}
```
Вызов: `String s = getFirst(stringList);` — тип T выводится автоматически.

**Настраиваемые конструкторы:**  
	Конструктор может использовать типовые параметры класса, но не может объявлять свои типовые параметры (синтаксически невозможно). Он просто принимает параметры типа T:
```
public class Box\<T> {
    public Box(T value) { this.value = value; }
}
```

**Настраиваемые интерфейсы:**  
	Определяют контракт с параметризованными типами. Широко используются в стандартной библиотеке.
```
public interface Comparable\<T> {
    int compareTo(T o);
}
public interface List\<T> extends Collection\<T> {
    T get(int index);
}
public interface Map<K, V> {
    V get(K key);
    V put(K key, V value);
}
```
#### 4.16 Перечислимые типы. Метаданные.
Перечислимые типы (enum) — специальный вид класса для задания фиксированного набора констант. Объявляются через `enum`. Могут иметь поля, методы, конструкторы (только private). 

Неявно наследуют `java.lang.Enum`. Поддерживают switch, итерацию (values()), сравнение через "\==". Метаданные — информация о программе, представленная аннотациями. Аннотации (`@interface`) позволяют добавлять метаданные к классам, методам, полям. 

Могут быть с параметрами. Доступны во время выполнения через рефлексию (RetentionPolicy.RUNTIME). Стандартные аннотации: @Override, @Deprecated, @SuppressWarnings.
##### 4.16-1 ==Перечислимые типы==
**Enum (перечисление)** — это специальный тип класса, экземпляры которого представляют фиксированный набор констант. Появился в Java 5.

**Объявление:**
```
public enum Season {
    WINTER, SPRING, SUMMER, AUTUMN
}
```
**Особенности enum:**
- Все enum неявно наследуют класс `java.lang.Enum`.
- Нельзя явно наследовать другой класс (уже наследуют Enum).
- Могут реализовать интерфейсы.
- Конструктор enum всегда **private** (неявно, указывать не нужно).
- Константы enum — это public static final экземпляры самого enum-класса.
- Можно объявлять поля и методы.

**Полезные методы enum:**

- `values()` — возвращает массив всех констант в порядке объявления.
- `valueOf(String name)` — возвращает константу по имени строки.
- `name()` — строковое имя константы.
- `ordinal()` — порядковый номер (начиная с 0). Использовать с осторожностью.
##### 4.16-2 ==Метаданные==
**Метаданные** в Java — это данные, описывающие саму программу: ее структуру, свойства, поведение. Основной механизм метаданных — **аннотации (annotations)**, появившиеся в Java 5.

**Аннотация** — специальная форма синтаксических метаданных, которая может быть добавлена к объявлениям классов, методов, полей, параметров, пакетов. Аннотации не влияют напрямую на выполнение кода, но могут обрабатываться компилятором, инструментами или во время выполнения через рефлексию.

**Стандартные аннотации Java:**
- **@Override** — указывает, что метод переопределяет метод суперкласса. Компилятор проверяет соответствие сигнатуры.
- **@Deprecated** — помечает устаревший элемент. Компилятор выдает предупреждение при использовании.
- **@SuppressWarnings** — подавляет предупреждения компилятора (например, "unchecked").
- **@FunctionalInterface** — помечает интерфейс как функциональный (ровно один абстрактный метод).
- **@SafeVarargs** — подавляет предупреждения для varargs с generics.

Аннотации широко используются в современных Java-фреймворках (Spring, Hibernate, JUnit) для конфигурации, валидации, внедрения зависимостей и тестирования.
#### 4.17 Интерфейс Set. Интерфейс List. Интерфейс Iterator. Интерфейс Map. Классы коллекций.
Collection — корневой интерфейс иерархии коллекций. 
Set — множество уникальных элементов (HashSet — неупорядоченный, LinkedHashSet — порядок добавления, TreeSet — отсортированный). 
List — упорядоченный список с доступом по индексу (ArrayList — массив, LinkedList — двусвязный список). 
Iterator — объект для последовательного обхода коллекции (методы hasNext(), next(), remove()). 
Map — отображение ключ-значение (HashMap — без порядка, LinkedHashMap — порядок добавления, TreeMap — сортировка по ключам). Основные реализации находятся в java.util. 
Collections — утилитарный класс с методами сортировки, поиска, синхронизации.
##### 4.17-1 ==Интерфейс Set==
**Set** — интерфейс коллекции, которая **не может содержать дубликаты**. Моделирует математическое множество. Равенство элементов определяется методами equals() и hashCode().

**Основные реализации:**
- **HashSet** — основан на хэш-таблице. Порядок элементов не гарантируется. Операции add, remove, contains — O(1) в среднем. Допускает null. Лучший выбор, если порядок не важен.
- **LinkedHashSet** — расширяет HashSet, сохраняя двусвязный список элементов. Порядок итерации равен порядку добавления. Чуть медленнее HashSet.
- **TreeSet** — основан на красно-черном дереве. Элементы отсортированы (natural ordering или Comparator). Операции — O(log n). Не допускает null (при использовании compareTo). Требует Comparable или Comparator.

Set не добавляет новых методов по сравнению с Collection. Основные операции: add(), remove(), contains(), size(), isEmpty(), clear().
##### 4.17-2 ==Интерфейс List==
**List** — упорядоченная коллекция (сохраняет порядок добавления). Допускает дубликаты и null. Обеспечивает доступ к элементам по индексу.

**Основные методы List:**
- `get(int index)` — получение элемента по индексу.
- `set(int index, E element)` — замена элемента.
- `add(int index, E element)` — вставка в указанную позицию.
- `remove(int index)` — удаление по индексу.
- `indexOf(Object o)`, `lastIndexOf(Object o)` — поиск позиции.

**Основные реализации:**
- **ArrayList** — массив переменной длины. Быстрый доступ по индексу O(1). Вставка/удаление в середине — O(n). Используется по умолчанию для списков.
- **LinkedList** — двусвязный список. Медленный доступ по индексу O(n). Быстрая вставка/удаление в начале и конце O(1). Реализует также интерфейсы Deque и Queue.
- **Vector** — устаревший аналог ArrayList, все методы синхронизированы. Не рекомендуется.
- **Stack** — устаревший наследник Vector, LIFO-стек. Заменен на ArrayDeque.
##### 4.17-3 ==Интерфейс Iterator==
**Iterator** — объект для последовательного обхода элементов коллекции. Позволяет удалять элементы во время обхода без ConcurrentModificationException.

**Основные методы:**
- `boolean hasNext()` — есть ли следующий элемент.
- `E next()` — возвращает следующий элемент и переводит итератор дальше.
- `void remove()` — удаляет последний элемент, возвращенный next(). Вызывается не более одного раза за вызов next().

**Использование**:
Iterator\<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    if (s.startsWith("A")) {
        it.remove(); // безопасное удаление при обходе
    }
}
##### 4.17-4 ==Интерфейс Map==
**Map** — интерфейс для хранения пар **ключ-значение**. Не является наследником Collection. Ключи уникальны (определяются через equals() и hashCode()), значения могут дублироваться. Каждый ключ отображается на одно значение

**Основные методы:**

- `V put(K key, V value)` — добавить пару (или заменить значение).
- `V get(Object key)` — получить значение по ключу.
- `V remove(Object key)` — удалить пару по ключу.
- `Set<K> keySet()` — множество всех ключей.
- `Collection<V> values()` — коллекция всех значений.
- `Set<Map.Entry<K,V>> entrySet()` — множество пар ключ-значение.

**Основные реализации:**
- **HashMap** — хэш-таблица. Ключи и значения могут быть null. Порядок не гарантирован. Операции O(1). Используется по умолчанию.
- **LinkedHashMap** — сохраняет порядок добавления (или порядок доступа при access-order). Чуть медленнее HashMap.
- **Hashtable** — устаревший аналог HashMap, все методы синхронизированы, не допускает null. Заменен на ConcurrentHashMap.

**Интерфейс Map.Entry:** внутренний интерфейс, представляющий пару ключ-значение. Методы: getKey(), getValue(), setValue(V).
##### 4.17-5 ==Классы коллекций==
**java.util.Collections** — утилитарный класс со статическими методами для работы с коллекциями.

**Основные методы:**
- **Сортировка:** sort(List), sort(List, Comparator).
- **Поиск:** binarySearch(List, key), binarySearch(List, key, Comparator).
- **Максимум/минимум:** max(Collection), min(Collection).
- **Перемешивание:** shuffle(List).
- **Заполнение:** fill(List, element).
- **Копирование:** copy(dest, src).
- **Реверсирование:** reverse(List).
- **Частота:** frequency(Collection, element).
- **Синхронизированные обертки:** synchronizedList(), synchronizedSet(), synchronizedMap() — оборачивают коллекцию в потокобезопасную.
- **Неизменяемые обертки:** unmodifiableList(), unmodifiableSet(), unmodifiableMap() — создают read-only представление.
- **Пустые коллекции:** emptyList(), emptySet(), emptyMap().
- **Singleton коллекции:** singleton(), singletonList(), singletonMap().
#### 4.18 Проблемы однопоточного подхода. Особенности многопоточности. Использование класса Thread. Использование интерфейса Runnable. Приоритеты потоков.
Однопоточное приложение выполняет задачи последовательно, блокируя интерфейс и не используя ресурсы многоядерных процессоров. Многопоточность позволяет параллельное выполнение задач, повышая отзывчивость и производительность. Поток — независимая единица выполнения внутри процесса. Создание потока: наследование класса Thread (переопределить run()) или реализация интерфейса Runnable (передать в конструктор Thread). Запуск — метод start() (не run()). Приоритеты потоков (1–10, по умолчанию 5) — рекомендация планировщику, не гарантируют порядок выполнения.
##### 4.18-1 ==Проблемы однопоточного подхода==
Однопоточное приложение имеет только один поток выполнения (main). Все задачи выполняются последовательно.

**Основные проблемы:**
- **Блокировка интерфейса:** длительная операция (загрузка файла, запрос к БД) замораживает весь интерфейс, приложение не отвечает.
- **Неэффективное использование ресурсов:** современные процессоры многоядерные, но один поток использует только одно ядро.
- **Низкая пропускная способность:** задачи не могут выполняться параллельно, даже если они независимы.
- **Простой в ожидании:** если поток ждет ввода-вывода, процессор простаивает, хотя мог бы выполнять другие задачи.

Пример: сервер, обрабатывающий запросы клиентов. При однопоточном подходе все клиенты встают в очередь, сервер обслуживает строго по одному.
##### 4.18-2 ==Особенности многопоточности==
Многопоточность — одновременное выполнение нескольких потоков в рамках одного процесса. Потоки разделяют общую память (кучу), но имеют собственный стек вызовов.

**Преимущества многопоточности:**
- **Повышение отзывчивости:** основной поток может оставаться отзывчивым, пока фоновые потоки выполняют тяжелые задачи.
- **Утилизация многоядерных процессоров:** потоки распределяются по ядрам, выполняясь действительно параллельно.
- **Увеличение пропускной способности:** несколько задач обрабатываются одновременно.
- **Естественное моделирование параллельных процессов:** сервер обслуживает нескольких клиентов одновременно.

**Сложности многопоточности:**

- **Состояние гонки (race condition):** результат зависит от порядка выполнения потоков.
- **Взаимные блокировки (deadlock):** потоки бесконечно ждут друг друга.
- **Голодание (starvation):** поток не получает доступ к ресурсу.
- **Сложность отладки и тестирования:** ошибки трудновоспроизводимы, зависят от планировщика.
- **Накладные расходы:** создание и переключение потоков требует ресурсов.
##### 4.18-3 ==Использование класса Thread==
Класс `Thread` — это базовый инструмент для создания и управления независимыми потоками выполнения в многопоточных приложениях. Он позволяет параллельно запускать несколько блоков кода, повышая общую производительность программы. Основные операции включают создание потока (через конструктор), запуск (метод `start()`) и приостановку (например, `join()`).
```
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Поток " + getName() + " работает");
    }
} 
```
MyThread t = new MyThread();
t.start(); // создает новый поток и вызывает run() в нем

**Основные методы класса Thread:**
- `start()` — запускает поток (однократно).
- `run()` — код, выполняемый в потоке (переопределяется).
- `sleep(long millis)` — приостанавливает поток на указанное время (static). Поток переходит в состояние TIMED_WAITING.
- `join()` — ожидание завершения другого потока.
- `interrupt()` — прерывание потока (устанавливает флаг).
- `isAlive()` — жив ли поток.
- `getName()`, `setName()` — имя потока.
- `currentThread()` — получение ссылки на текущий поток (static).
- `yield()` — предложение планировщику уступить процессор (static, редко используется).
##### 4.18-4 ==Использование интерфейса Runnable==
Интерфейс `Runnable` в Java используется ==для создания задач, выполняемых параллельно==. Чтобы использовать его, реализуйте метод `run()`, содержащий нужный код, и передайте экземпляр класса в объект `Thread` для запуска Интерфейс Runnable и класс Thread в Java - ExamClouds.

**Вот базовый пример реализации:**
```
public class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Задача выполняется в потоке: " + Thread.currentThread().getName());
    }
    public static void main(String[] args) {
        MyTask task = new MyTask();
        Thread thread = new Thread(task);
        thread.start(); // Запуск потока
    }
}
```
**Основные способы использования:**
- **Лямбда-выражения:** Так как `Runnable` — это функциональный интерфейс (содержит только один метод `run`), его можно записать компактно без создания отдельного класса
- **Использование с пулами потоков (ExecutorService):** Это более современный и предпочтительный подход для управления множеством потоков, так как он оптимизирует их переиспользование Runnable и Callable в Java: когда использовать каждый интерфейс.

**Преимущества Runnable перед Thread:**
- Класс может наследовать другой класс, не занимая единственный слот extends.
- Разделение задачи (Runnable) и механизма выполнения (Thread).
- Можно передать один Runnable нескольким потокам.
- Лучше соответствует принципу единственной ответственности.
##### 4.18-5 ==Приоритеты потоков==
Каждый поток имеет приоритет — целое число от 1 до 10. 
Константы: `Thread.MIN_PRIORITY` (1), `Thread.NORM_PRIORITY` (5), `Thread.MAX_PRIORITY` (10). 
По умолчанию поток получает приоритет родительского потока.

**Методы:**
- `setPriority(int priority)` — установка приоритета.
- `getPriority()` — получение приоритета.

**Важные замечания:**

- Приоритет — это лишь **рекомендация** планировщику потоков. JVM не обязана строго соблюдать приоритеты.
- Поведение приоритетов **зависит от платформы** (Windows и Linux обрабатывают их по-разному).
- **Не полагайтесь на приоритеты** для корректности программы. Логика не должна зависеть от них.
- Потоки с более высоким приоритетом получают больше процессорного времени, но не гарантированно.
- Низкоприоритетные потоки рискуют столкнуться с голоданием.

Лучшая практика: использовать приоритеты только для тонкой настройки производительности, а не для управления порядком выполнения.
#### 4.19 Управление потоками. Нерекомендуемые действия над потоками. Прерывание потока. Группы потоков. Операции в группе потоков. Демон-потоки. Пример. Демон-группы потоков.
Управление потоками включает запуск (start), приостановку (sleep), ожидание (join), прерывание (interrupt). 
Нерекомендуемые (deprecated) методы: stop(), suspend(), resume(), destroy() — небезопасны, вызывают deadlock и повреждение данных. 
Прерывание потока — кооперативный механизм: вызов interrupt() устанавливает флаг, поток должен сам проверять isInterrupted() и завершаться. 
Группы потоков (ThreadGroup) объединяют потоки для массовых операций. 
Демон-потоки — служебные потоки, JVM завершает их принудительно, когда все обычные потоки завершены. Группа демон-потоков автоматически уничтожается при опустении.
##### 4.19-1 ==Управление потоками==
Управление жизненным циклом потока осуществляется через методы класса Thread.

**Основные легальные методы управления:**
- `start()` — запуск потока. Переводит из NEW в RUNNABLE. Вызывается однократно.
- `sleep(long ms)` — приостанавливает текущий поток на указанное время. Поток переходит в TIMED_WAITING. Не освобождает мониторы (lock).
- `join()` — текущий поток ожидает завершения указанного потока. Может быть с таймаутом: `join(long ms)`.
- `interrupt()` — устанавливает флаг прерывания. Если поток в sleep/wait/join — выбрасывается InterruptedException.
- `yield()` — предложение планировщику передать процессор другому потоку. Не гарантирует переключения.

**Состояния потока (Thread.State):**
- **NEW** — создан, но start() еще не вызван.
- **RUNNABLE** — выполняется или готов к выполнению.
- **BLOCKED** — заблокирован в ожидании монитора (synchronized).
- **WAITING** — ожидает сигнала от другого потока (wait, join без таймаута).
- **TIMED_WAITING** — ожидание с таймаутом (sleep, wait(timeout), join(timeout)).
- **TERMINATED** — завершил выполнение.
##### 4.19-2 ==Нерекомендуемые действия над потоками==
Ряд методов класса Thread помечены как **deprecated** (не рекомендуются к использованию) из-за небезопасности.

**Методы и причины отказа от них:**
- **stop()** — принудительная остановка потока. Снимает все мониторы и освобождает блокировки, оставляя объекты в потенциально поврежденном состоянии. Поток убивается мгновенно без возможности освободить ресурсы. Замена: использовать флаг и interrupt().
- **suspend() / resume()** — приостановка/возобновление потока. Поток приостанавливается, но не освобождает мониторы. Возникает риск deadlock, если другой поток ждет этот монитор. Замена: использовать wait/notify или Lock из concurrent.
- **destroy()** — уничтожение потока без очистки. Еще опаснее stop(). Никогда не был реализован полностью, всегда бросал NoSuchMethodError.

**Правильный подход:** вместо принудительного управления поток должен сам кооперироваться — проверять флаг прерывания и корректно завершать свою работу.
##### 4.19-3 ==Прерывание потока==
В Java потоки нельзя останавливать принудительно (например, устаревшим методом `stop()`). Вместо этого используется механизм «вежливой» остановки: метод `interrupt()` ставит потоку флаг прерывания, а сам поток внутри цикла проверяет этот флаг методом `isInterrupted()` и корректно завершает работу.

Для реализации безопасного прерывания потока используются два основных подхода в зависимости от того, что делает ваш поток.
1. Если поток выполняет вычисления (активный цикл)

В цикле необходимо периодически проверять состояние флага прерывания с помощью метода `isInterrupted()`.

```
public class MyTask implements Runnable {
    @Override
    public void run() {
        // Проверяем флаг прерывания на каждой итерации
        while (!Thread.currentThread().isInterrupted()) {
            // Выполняем полезную работу
            System.out.println("Поток работает...");
        }
        System.out.println("Поток корректно завершен.");
    }
}
```
##### 4.19-4 ==Группы потоков. Операции в группе потоков==
**ThreadGroup** — класс, позволяющий объединять потоки в группы для массового управления.

**Создание группы:**
```
ThreadGroup group = new ThreadGroup("MyGroup");
Thread t = new Thread(group, () -> { ... }, "ThreadName");
```
Каждый поток принадлежит какой-то группе (по умолчанию — группе родительского потока). Группы могут быть вложенными.

**Операции над группой:**
- `activeCount()` — примерное количество активных потоков в группе и подгруппах.
- `enumerate(Thread[] list)` — копирует ссылки на потоки в массив.
- `getMaxPriority()` / `setMaxPriority()` — максимальный приоритет для новых потоков.
- `interrupt()` — прерывает все потоки в группе.
- `destroy()` — уничтожает группу (должна быть пустой).
- `setDaemon()` — устанавливает группу как демон-группу.
##### 4.19-5 ==Демон-потоки. Пример. Демон-группы потоков==
**Демон-поток (daemon thread)** — служебный поток, который не препятствует завершению JVM. JVM завершает работу, когда все **не-демон** потоки завершены. Демон-потоки принудительно останавливаются.

**Установка демон-статуса**
```
Thread daemon = new Thread(() -> {
    while (true) {
        // бесконечная фоновая работа
    }
});
daemon.setDaemon(true); // установить до start()!
daemon.start();
```
**Свойства демон-потоков:**
- При завершении последнего обычного потока JVM останавливает все демон-потоки без выполнения finally-блоков.
- Демон-потоки не должны выполнять критические операции с данными (риск повреждения).
- Потоки, порожденные демон-потоком, также становятся демонами.
- Головной поток main — не демон.
- Garbage Collector — пример демон-потока.
#### 4.20 Совместное использование ресурсов. Характерные ошибки. volatile. Специальные методы класса Object. Особенности использования методов класса Object.
При совместном доступе потоков к общим данным возникают состояния гонки (race condition) — результат зависит от порядка выполнения. 
Синхронизация через synchronized блокирует доступ к критической секции. volatile гарантирует видимость изменений переменной между потоками, но не атомарность compound-операций (инкремент). 
Методы Object: wait() — освобождает монитор и ждет notify/notifyAll; notify() — будит один случайный ожидающий поток; notifyAll() — будит все ожидающие потоки. Эти методы должны вызываться только внутри synchronized-блока на том же объекте, иначе IllegalMonitorStateException.
##### 4.20-1 ==Совместное использование ресурсов==
В многопоточной среде потоки разделяют общую память процесса (кучу). Когда несколько потоков одновременно читают и изменяют одни и те же данные, возникает необходимость координации.

**Критическая секция** — участок кода, работающий с общим ресурсом. В один момент времени только один поток должен находиться в критической секции.

**Механизмы синхронизации в Java:**
- **synchronized-блок** — захватывает монитор объекта. Пока один поток держит монитор, другие ждут.
- **synchronized-метод** — то же самое, монитором выступает объект this (для экземплярного) или класс (для static).
- **Явные блокировки (Lock)** из java.util.concurrent.locks — более гибкий механизм (ReentrantLock с tryLock, ReadWriteLock).
- **Атомарные классы (AtomicInteger, AtomicReference)** — обеспечивают атомарность операций без явной блокировки (CAS-алгоритмы).

**Пример synchronized-блока:**
```
synchronized (lockObject) {
    // критическая секция
}
```
##### 4.20-2 ==Характерные ошибки==
**Состояние гонки (race condition):** результат зависит от того, какой поток первым выполнит операцию. Проявляется нестабильно, трудно воспроизводится.

Пример: два потока делают `counter++` (не атомарно: чтение, увеличение, запись). Итоговое значение непредсказуемо — часть инкрементов теряется.

**Взаимная блокировка (deadlock):** два и более потоков бесконечно ждут друг друга. Поток A захватил ресурс X и ждет Y, поток B захватил Y и ждет X.

Условия deadlock (все четыре должны выполняться одновременно):
- Взаимное исключение (ресурсы нельзя разделять).
- Удержание и ожидание (поток держит ресурс и ждет другой).
- Отсутствие принудительного освобождения.
- Циклическое ожидание (A→B→...→A).

**Голодание (starvation):** поток не может получить доступ к ресурсу, потому что его постоянно опережают другие потоки. Часто возникает при неправильном использовании приоритетов.

**Состояние livelock:** потоки активны, но не могут завершить работу, постоянно реагируя на действия друг друга (как два человека в коридоре, пытающиеся уступить дорогу).
##### 4.20-3 ==volatile==
**volatile** — модификатор переменной, гарантирующий её видимость между потоками.

**Что гарантирует:**
- Чтение volatile-переменной всегда возвращает последнее записанное значение (не из кэша потока).
- Запись в volatile немедленно становится видимой всем потокам.
- Операции чтения/записи volatile **атомарны** (кроме long/double на 32-битных JVM).

**Что НЕ гарантирует:**
- Атомарность составных операций. `count++` с volatile — всё равно не атомарно (чтение+изменение+запись).
- Если два потока делают `volatile++`, результат непредсказуем.

**Когда использовать volatile:**
- Флаг завершения потока (`while (!stopped)`), где stopped объявлен как volatile.
- Одно чтение/одна запись без зависимости от предыдущего значения.
- DCL (Double-Checked Locking) для синглтона (совместно с synchronized).
##### 4.20-4 ==Специальные методы класса Object==
Три метода для межпоточного взаимодействия, унаследованные всеми классами:

**wait():**
- Переводит текущий поток в состояние WAITING.
- **Освобождает монитор** объекта, чтобы другие потоки могли его захватить.
- Ждет, пока другой поток вызовет notify()/notifyAll() на том же объекте.
- Может быть с таймаутом: `wait(long timeout)`.
- После пробуждения поток снова пытается захватить монитор.

**notify():**
- Будит **один случайный** поток, ожидающий на этом объекте (wait()).
- Пробужденный поток переходит из WAITING в BLOCKED (ждет освобождения монитора).
- Если никто не ждет — вызов игнорируется.

**notifyAll():**
- Будит **все** потоки, ожидающие на этом объекте.
- Они конкурируют за монитор, выполнение продолжит только один.

**Правило вызова:** все три метода должны вызываться **только внутри synchronized-блока или метода** на том же объекте. Иначе — **IllegalMonitorStateException**.
##### 4.20-5 ==Особенности использования методов класса Object==
**wait() нужно вызывать в цикле, а не в if:**  
Ложные пробуждения (spurious wakeups) — поток может проснуться без вызова notify. Поэтому всегда проверять условие в цикле while:

```
synchronized (lock) {
    while (conditionNotMet) {
        lock.wait();
    }
    // действие после выполнения условия
}
```

**Монитор — это объект:**  
synchronized-блок привязан к объекту. Нельзя использовать wait/notify на строковых литералах (из-за пула строк) и на примитивных типах.

**Современная альтернатива — пакет java.util.concurrent:**  
Вместо wait/notify рекомендуется использовать высокоуровневые средства:

- Lock и Condition (аналог wait/notify с раздельными условиями).
- CountDownLatch, CyclicBarrier, Semaphore.
- BlockingQueue (потокобезопасная очередь с ожиданием).
#### 3.21 Пакет java.util.concurrent. ReentrantLock, ReadWriteLock. Интерфейсы Callable и Future. Интерфейсы Executor, ExecutorService, ScheduledExecutorService. Пул потоков. Executors.
Пакет java.util.concurrent (Java 5) предоставляет высокоуровневые средства для многопоточности. 
ReentrantLock — явная блокировка с возможностью tryLock, прерывания, честной очереди. ReadWriteLock разделяет чтение (много потоков) и запись (один поток). 
Callable — аналог Runnable, но возвращает результат и бросает исключения. Future — результат асинхронной задачи, позволяет отменить задачу и дождаться результата. 
ExecutorService управляет пулом потоков, принимает Callable/Runnable. 
ScheduledExecutorService — выполнение по расписанию. Executors — фабрика для создания пулов (fixed, cached, single, scheduled).
##### 4.21-1 ==Пакет java.util.concurrent==
Пакет **java.util.concurrent** (сокращённо JUC) появился в Java 5 и содержит высокоуровневые средства для многопоточного программирования. Решает типовые задачи, снижая риск ошибок по сравнению с низкоуровневыми wait/notify/synchronized.

**Основные группы классов:**
- **Блокировки:** Lock, ReentrantLock, ReadWriteLock, StampedLock.
- **Атомарные типы:** AtomicInteger, AtomicLong, AtomicReference, LongAdder.
- **Исполнители (Executors):** Executor, ExecutorService, ThreadPoolExecutor.
- **Коллекции:** ConcurrentHashMap, CopyOnWriteArrayList, BlockingQueue.
- **Синхронизаторы:** CountDownLatch, CyclicBarrier, Semaphore, Phaser.
- **Future/Callable:** асинхронные задачи с результатом.

Пакет полностью вытесняет устаревшие подходы: Thread, wait/notify, Hashtable/Vector.
##### 4.21-2 ==ReentrantLock, ReadWriteLock==
**ReentrantLock** — реализация интерфейса Lock. Предоставляет более гибкий механизм блокировки, чем synchronized.

**Преимущества перед synchronized:**
- **tryLock()** — попытка захвата без блокировки. Можно задать таймаут: `tryLock(1, TimeUnit.SECONDS)`.
- **lockInterruptibly()** — возможность прервать ожидание блокировки.
- **Справедливость:** `new ReentrantLock(true)` — очередь FIFO, поток с самым долгим ожиданием получает блокировку.
- **Разделение условий (Condition):** `lock.newCondition()` создаёт несколько очередей ожидания для разных условий. Аналог wait/notify, но с выбором, какой поток будить.

**Обязательный паттерн**
```
Lock lock = new ReentrantLock();
lock.lock();
try {
    // критическая секция
} finally {
    lock.unlock(); // всегда в finally
}
```

**ReadWriteLock** — интерфейс с двумя блокировками:
- **readLock()** — разделяемая, могут захватить много потоков одновременно (если нет записи).
- **writeLock()** — эксклюзивная, только один поток, читатели блокируются.
##### 4.21-3 ==Интерфейсы Callable и Future==
**Callable\<V>** — аналог Runnable, но:
- Возвращает результат типа V методом `call()`.
- Может выбрасывать checked-исключения.
- Передаётся в ExecutorService.
```
Callable<Integer> task = () -> {
    Thread.sleep(1000);
    return 42;
};
```

**Future\<V>** — представляет результат асинхронной задачи. Возвращается при отправке Callable в ExecutorService.

**Методы Future:**
- `V get()` — блокирует поток и ждёт результат. Может бросить InterruptedException, ExecutionException.
- `V get(long timeout, TimeUnit unit)` — с таймаутом.
- `boolean isDone()` — задача завершена (успешно, ошибкой или отменой).
- `boolean cancel(boolean mayInterruptIfRunning)` — отмена задачи. Если ещё не запущена — не запустится. Если выполняется — может быть прервана.
- `boolean isCancelled()` — была ли задача отменена.
##### 4.21-4 ==Интерфейсы Executor, ExecutorService, ScheduledExecutorService==
**Иерархия интерфейсов для управления потоками:**

**Executor** — базовый интерфейс с одним методом:
```
void execute(Runnable command);
```

Отделяет отправку задачи от способа выполнения (в новом потоке, в пуле, синхронно).

**ExecutorService** расширяет Executor:
- `Future<?> submit(Runnable task)` — отправка Runnable, возвращает Future.
- `<T> Future<T> submit(Callable<T> task)` — отправка Callable.
- `void shutdown()` — мягкое завершение: новые задачи не принимаются, старые дорабатывают.
- `List<Runnable> shutdownNow()` — жёсткое завершение: попытка остановить все задачи.
- `boolean awaitTermination(long timeout, TimeUnit unit)` — ожидание завершения.
- `invokeAll()`, `invokeAny()` — массовое выполнение задач.

**ScheduledExecutorService** расширяет ExecutorService:
- `schedule(Callable, delay, TimeUnit)` — отложенный запуск.
- `scheduleAtFixedRate(Runnable, initialDelay, period, TimeUnit)` — периодический запуск с фиксированной частотой.
- `scheduleWithFixedDelay(Runnable, initialDelay, delay, TimeUnit)` — периодический запуск с фиксированной задержкой между концом предыдущего и началом следующего.
##### 4.21-5 ==Пул потоков. Executors==
Создание потоков — дорогая операция. Пул потоков содержит заранее созданные потоки и переиспользует их для задач.

**Executors** — фабричный класс для создания пулов:
- **newFixedThreadPool(int n)** — пул с фиксированным числом потоков. Если все заняты, задачи в очереди.
- **newCachedThreadPool()** — динамический пул: создаёт потоки по необходимости, удаляет простаивающие через 60 секунд.
- **newSingleThreadExecutor()** — пул с одним потоком. Задачи выполняются последовательно.
- **newScheduledThreadPool(int n)** — пул для задач по расписанию.
- **newWorkStealingPool()** (Java 8) — пул с ForkJoinPool, потоки «воруют» задачи друг у друга.
#### 4.22 Модель OSI. Модель «Клиент-сервер». Понятие порта. Абстракция сокета. Пакет java.net. Класс Socket. Порядок работы с сокетом клиента. Класс ServerSocket. Сервер параллельной обработки запросов. Дейтаграммы. Uniform Resource Locator.
##### 4.22-1 ==Модель OSI==
**OSI (Open Systems Interconnection)** — эталонная семиуровневая модель сетевого взаимодействия. Разработана ISO для стандартизации сетевых протоколов.

**Уровни (сверху вниз):**

| Уровень | Название      | Назначение                     | Примеры                      |
| ------- | ------------- | ------------------------------ | ---------------------------- |
| 7       | Прикладной    | Интерфейс приложений           | HTTP, FTP, SMTP, DNS         |
| 6       | Представления | Формат данных, шифрование      | SSL/TLS, кодировки           |
| 5       | Сеансовый     | Управление сеансами связи      | NetBIOS, RPC                 |
| 4       | Транспортный  | Надёжная доставка, сегментация | TCP, UDP                     |
| 3       | Сетевой       | Маршрутизация, адресация       | IP, ICMP                     |
| 2       | Канальный     | Кадры, MAC-адреса              | Ethernet, Wi-Fi              |
| 1       | Физический    | Биты, сигналы, кабели          | Ethernet-кабель, оптоволокно |
##### 4.22-2 ==Модель «Клиент-сервер»==
**Клиент-сервер** — архитектурная модель, в которой задачи распределены между поставщиком ресурсов (сервером) и потребителем (клиентом).

**Сервер:**
- Пассивно ожидает запросы на определённом порту.
- Обслуживает множество клиентов (параллельно или последовательно).
- Предоставляет доступ к ресурсам (файлы, БД, вычисления).

**Клиент:**
- Активно инициирует соединение с сервером.
- Отправляет запрос и получает ответ.
- Обычно взаимодействует с одним сервером.

**Этапы взаимодействия:**
1. Сервер запускается и начинает прослушивание порта.
2. Клиент устанавливает соединение с сервером (IP + порт).
3. Клиент отправляет запрос.
4. Сервер обрабатывает запрос.
5. Сервер отправляет ответ.
6. Соединение закрывается (или сохраняется для следующих запросов).
##### 4.22-3 ==Понятие порта. Абстракция сокета==
**Порт** — целое число от 0 до 65535 (16 бит), идентифицирующее конкретное приложение или службу на хосте. IP-адрес идентифицирует хост, порт — процесс на хосте.

**Диапазоны портов:**
- **0–1023** — системные (well-known): HTTP — 80, HTTPS — 443, FTP — 21, SSH — 22.
- **1024–49151** — зарегистрированные (пользовательские).
- **49152–65535** — динамические (частные, эфемерные).

**Сокет** — абстрактное понятие, представляющее конечную точку сетевого соединения. Определяется парой: **IP-адрес + порт**. Для TCP-соединения сокет характеризуется четвёркой: (IP клиента, порт клиента, IP сервера, порт сервера).

Сокет предоставляет потоки ввода-вывода для передачи данных. В Java представлен классами Socket (клиент) и ServerSocket (сервер).
##### 4.22-4 ==Пакет java.net. Класс Socket. Порядок работы с сокетом клиента==
**Пакет java.net** содержит классы для сетевого взаимодействия:
- **Socket** — клиентский TCP-сокет.
- **ServerSocket** — серверный TCP-сокет.
- **DatagramSocket** — сокет для UDP.
- **DatagramPacket** — пакет данных UDP.
- **URL** — представление URL.
- **InetAddress** — представление IP-адреса (IPv4/IPv6).

**Класс Socket (клиент):**  
Создаётся с указанием IP (или хоста) и порта сервера:
```
Socket socket = new Socket("localhost", 8080);
```
**Порядок работы клиента:**

1. Создать сокет: `Socket socket = new Socket(host, port);`
2. Получить потоки:
    - `InputStream in = socket.getInputStream();` — чтение от сервера.
    - `OutputStream out = socket.getOutputStream();` — запись серверу.
3. Обмен данными (через обёртки: BufferedReader/PrintWriter для текста, DataInputStream/DataOutputStream для бинарных данных).
4. Закрыть потоки.
5. Закрыть сокет: `socket.close();`

**Пример:**
```
try (Socket socket = new Socket("example.com", 80)) {
    PrintWriter out = new PrintWriter(socket.getOutputStream(), true);
    BufferedReader in = new BufferedReader(new InputStreamReader(socket.getInputStream()));
    out.println("GET / HTTP/1.1");
    out.println("Host: example.com");
    out.println();
    in.lines().forEach(System.out::println);
}
```
##### 4.22-5 ==Класс ServerSocket. Сервер параллельной обработки запросов==
**ServerSocket** — класс для создания сервера, прослушивающего входящие TCP-подключения.

**Порядок работы сервера:**
1. Создать ServerSocket на порту: `ServerSocket server = new ServerSocket(port);`
2. Ожидать подключения: `Socket client = server.accept();` — блокирующий вызов.
3. Получить потоки клиента и обработать запрос.
4. Закрыть клиентский сокет.
5. Вернуться к accept() для следующего клиента.
6. Закрыть ServerSocket при завершении.

**Параллельный сервер:** каждый клиент обслуживается в отдельном потоке:
```
ServerSocket server = new ServerSocket(8080);
while (true) {
    Socket client = server.accept();
    new Thread(() -> {
        // обработка клиента
        client.close();
    }).start();
}
```
**Недостатки модели «поток на клиента»:** накладные расходы при большом числе клиентов. Альтернатива — пул потоков (ExecutorService) или неблокирующий ввод-вывод (NIO, java.nio.channels).
##### 4.22-6 ==Дейтаграммы==
**Дейтаграмма (UDP)** — способ передачи данных без установки соединения. Не гарантирует доставку, порядок и отсутствие дубликатов. Преимущества: низкая задержка, меньше накладных расходов.

**Классы Java для UDP:**
- **DatagramSocket** — сокет для отправки и приёма дейтаграмм.
- **DatagramPacket** — пакет данных, содержит буфер, адрес, пор

**Отличия UDP от TCP:**
- Нет соединения (не нужен accept/connect).
- Нет гарантии доставки и порядка.
- Быстрее, меньше накладных расходов.
- Подходит для стриминга (видео/аудио), DNS, VoIP, онлайн-игр.
##### 4.22-7 ==Uniform Resource Locator==
**URL (Uniform Resource Locator)** — унифицированный указатель ресурса. Строка, однозначно определяющая расположение ресурса в сети.

**Структура URL:**
```
протокол://хост:порт/путь?параметры#якорь
```

Пример: `https://example.com:443/path?key=value#section`

**Класс URL в Java:**
```
URL url = new URL("https://example.com/page");
String host = url.getHost();
int port = url.getPort(); // -1 если не указан явно
String path = url.getPath();
String protocol = url.getProtocol();

// чтение содержимого
try (BufferedReader in = new BufferedReader(new InputStreamReader(url.openStream()))) {
    String line;
    while ((line = in.readLine()) != null) {
        System.out.println(line);
    }
}
```
#### 4.23 Статические вложенные классы. Вложенные интерфейсы. Нестатические вложенные классы. Локальные классы. Анонимные классы.
Вложенные классы делятся на статические (static nested class) и нестатические (внутренние, inner classes). 
Статический вложенный класс не имеет ссылки на внешний объект, может содержать static-члены. 
Нестатический (внутренний) класс связан с экземпляром внешнего класса, имеет неявную ссылку на него. 

Локальный класс объявляется внутри метода/блока, видит локальные переменные (если они final/effectively final). Анонимный класс — безымянный, создается на лету как реализация интерфейса или наследник класса. Интерфейсы могут быть вложенными, неявно static.
##### 4.23-1 ==Статические вложенные классы==
**Статический вложенный класс (static nested class)** объявляется внутри другого класса с модификатором `static`.

**Особенности:**
- Не имеет доступа к нестатическим членам внешнего класса напрямую (только через экземпляр).
- Ведет себя как обычный класс, просто находится внутри другого для логической группировки.
- Может содержать как static, так и нестатические члены.
- Внешний класс имеет доступ к private-членам статического вложенного класса и наоборот.

**Применение:** вспомогательные классы, тесно связанные с внешним классом, но не требующие ссылки на него (например, Builder в паттерне Builder, Node в структурах данных).
##### 4.23-2 ==Вложенные интерфейсы==
**Вложенный интерфейс** — интерфейс, объявленный внутри класса или другого интерфейса. Неявно всегда **static**, даже если модификатор static не указан.

**Особенности:**
- Может быть public, private, protected, package-private.
- Реализуется обычным образом: `class Impl implements Outer.InnerInterface`.
- Используется для логической группировки с внешним классом или интерфейсом.

**Практическое применение:**
Часто вложенные интерфейсы применяются для создания вспомогательных контрактов (например, интерфейс `Entry` внутри `Map` в стандартной библиотеке Java) или для определения действий с определенным компонентом, чтобы не загрязнять глобальное пространство имен.
##### 4.23-3 ==Нестатические вложенные классы==
**Нестатический вложенный класс (inner class)** объявляется без модификатора `static` внутри другого класса.

**Особенности:**
- Имеет **неявную ссылку** на экземпляр внешнего класса (через `Outer.this`).
- Может обращаться ко всем членам внешнего класса (включая private) напрямую.
- **Не может содержать static-члены** (поля, методы), кроме констант времени компиляции (`static final`).
- Создается только через экземпляр внешнего класса: `outer.new Inner()`.

**Применение:** когда внутренний класс тесно связан с внешним и должен иметь доступ к его состоянию (например, итераторы — `ArrayList.Itr`).

**Недостаток:** неявная ссылка на внешний объект мешает сборщику мусора (утечки памяти), если внутренний объект живет дольше внешнего.
##### 4.23-4 ==Локальные классы==
**Локальный класс (local class)** — класс, объявленный внутри блока: метода, конструктора, блока инициализации.

**Особенности:**
- Виден только внутри блока, в котором объявлен.
- Имеет доступ к полям и методам внешнего класса.
- Может обращаться к локальным переменным и параметрам метода, только если они **final** или **effectively final** (не изменяются после инициализации).
- Не может содержать static-члены (как и inner class).
- Может реализовать интерфейсы и наследовать классы.

**Применение:** когда класс нужен только внутри одного метода и переиспользование не требуется. Исторически использовался для обработчиков событий, сейчас чаще заменяется анонимными классами или лямбдами.
##### 4.23-5 ==Анонимные классы==
**Анонимный класс (anonymous class)** — локальный класс без имени. Создается одновременно с объявлением через оператор `new`.

**Особенности:**
- Не имеет имени и конструктора (используется конструктор суперкласса).
- Создается на лету как наследник класса или реализация интерфейса.
- Может обращаться к полям/методам внешнего класса и к final/effectively final локальным переменным.
- Не может содержать явный конструктор и static-члены.
- Компилируется в отдельный .class файл с именем `ВнешнийКласс$N.class`, где N — порядковый номер.

**Синтаксис:**
```
Runnable task = new Runnable() {
    @Override
    public void run() {
        System.out.println("Анонимный класс");
    }
};
```
**Ограничения:**

- Не может реализовать несколько интерфейсов.
- Может либо наследовать класс, либо реализовать один интерфейс.
- Код громоздкий, особенно при множестве методов.

Анонимные классы по-прежнему нужны, когда интерфейс имеет несколько методов или нужно наследовать класс с несколькими методами.
